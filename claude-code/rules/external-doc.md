# 외부-가시 문서 작성 공통 원칙

> **한 줄 요약**: 외부 제3자가 **빠르게(가독성) 정확하게(정직성) 이해하도록** 돕기 위한 원칙 묶음.

> **적용 대상**: GitHub issue/PR, ADR, PRD/SRS, CS 응답, 외부 전달 초안(draft), 회의록(외부 참석자와 후임자 참조 전제), PoC 결과 전달 등 **작업 컨텍스트가 없는 제3자**(동료 개발자, 의사결정자, 이해관계자, 고객)가 읽을 가능성이 있는 문서를 생성하는 스킬 전반.

> **적용 대상 외**: 개인 기록과 내부 분석(daily, memo, learn, coaching, leadership, analyze-request, todo, briefing, brainstorm 등).

## 대상 독자

**작업 컨텍스트가 없는 동료 개발자 + 의사결정자 + 이해관계자(+고객)** 를 가정한다. 팀 내 구두로 오가던 배경도 문서에서 처음 보는 정보. 작성자 본인 기준으로 "설명 안 해도 알 것"을 판단하지 말 것.

## 왜 저맥락이 중요한가

저맥락화는 **독자 배려**일 뿐만 아니라 **작성자 자신의 사고 품질 유지** 수단이기도 하다.

| 측면 | 고맥락의 위험 | 저맥락화의 효과 |
|------|-------------|---------------|
| **작성자 사고** | 디테일에 매몰되어 **토끼굴에 빠지고** 문제 본질에서 멀어짐 | 첫 문단으로 요약하려는 과정이 본질 재점검을 강제 |
| **외부 도움** | 맥락 재구성 비용이 커서 **누가 봐도 쉽게 조언하기 어려움** | 상황이 전달되면 조언과 질문이 쉽게 들어옴 |
| **리뷰 품질** | 리뷰어가 "알아서 잘했겠지" 넘어가거나 **세부에만 집중해 구조 놓침** | 구조와 가정이 드러나 리뷰어가 근거부터 되짚기 가능 |

저맥락 문서는 작성자가 **혼자 오래 굴리지 않게** 만드는 구조적 장치다. 이 원리는 코드에도 동일 적용된다. `extract method` 과중첩과 문서의 고맥락화는 동형이며, 각각 `Inline`과 저맥락 요약으로 풀어야 본질이 드러난다. 자세한 배경은 `GROUND_CLAUDE.md`의 "맥락 밀도" 섹션 참조.

## 1. 외부 공개 톤

내부에서만 쓰는 의심이나 판단성 라벨, 담당자 목록 나열, 작성자 주관 추정은 피한다. 사실 기반 간결 표현만.

## 2. 받는 쪽이 못 여는 참조 금지 (접근 가능성 기준)

기준은 **받는 사람이 그 대상을 열어볼 수 있는가**다.

- `.claude/*`, `.local.claude/*`, `CLAUDE.md` 등 **로컬에만 존재하는** 파일 경로는 **어떤 형태로도 포함하지 않는다** (참조, 링크, "근거 SRS" 등 일체). 외부 사용자에게는 보이지 않는 파일이라 죽은 링크가 되고, 내부 정보가 새어 나간다. 해당 문서의 정보가 필요하면 **내용을 문서에 직접 풀어 쓴다**.
- 반대로 받는 쪽이 접근 가능한 것은 참조해도 된다: 공개 URL(공식 문서, RFC 등), 같은 리포지토리의 코드 경로와 줄번호, PR과 이슈 번호. 원칙 10이 근거 출처로 요구하는 표기와 같은 방향이다.
- 외부 반출 여부가 분명하지 않으면 사용자에게 확인한다.

## 3. 저맥락 (용어 풀어쓰기)

- 제품 고유 용어, 아키텍처 패턴 용어, 내부 프로토콜, 이벤트, 메시지명은 **괄호로 한 줄 부연**하거나, 뺄 수 있으면 빼서 **일상 표현**으로 바꾼다
- 다이어그램 노드와 레이블도 동일. 내부 이벤트명 대신 "요청 / 완료" 식의 일상 표현

## 4. Outside-in 서술

디테일을 빼라는 게 아니라 **배치 순서**의 문제. 바깥(사용자 체감, 비즈니스 동기, 팀 관점 한 줄 요약)에서 안쪽(구현 전략, 기술 세부) 순으로 내려간다.

- **첫 단락**은 팀 외부 독자도 2~3줄 안에 "무슨 일이 일어나는지" 이해할 수 있어야 함. 첫 단락에 제품 고유 용어, 라이브러리명, 프로토콜명이 나오면 잘못된 것
- 깊은 기술 세부(프로토콜 메시지, 내부 구조 등)는 `<details>` 접기나 본문 후반으로 내린다
- 의사결정자는 **왜**를 보고 승인하고, 동료 개발자는 **결과**를 보고 영향 범위를 가늠하며, 리뷰어는 **세부**까지 필요할 때만 펼친다

## 5. 스캔 가능성 우선 (스캔 > 정독)

독자는 문서를 처음부터 끝까지 읽지 않고 **훑어서(scan)** 핵심을 집어낸다.

- 산문 단락은 최소화, **표 > 라벨 불릿 > 산문** 순으로 우선
- 산문이 필요해도 문단은 2~3줄 이하
- 불릿은 **첫 단어에 결론/핵심 키워드를 굵게** 세워 눈을 흘리면서도 요점이 잡히게 한다
  - 예: "**기존 경로** 그대로 유지" / "**비상 탈출구**: 문제 시 기존 버튼 사용 가능"
- 3개 이상의 속성이 나열되면 표로 정리하는 것이 거의 항상 낫다

## 6. What > How (본질 변화 > 구현 수단)

제목, 첫 단락, 효과, 목표 서술의 **주어**에 구현 수단(특정 서버, 프로토콜, 엔드포인트, 라이브러리명 등)을 두지 않는다.

- 수단은 프로젝트 진행 중 얼마든 바뀔 수 있다(서버에서 함수로, 다시 에이전트로, 라이브러리 A에서 B로 등)
- 바뀌지 않는 것은 **본질 변화**(경로 통합, 데이터 소유권 이전, 책임 분리, 신뢰 경계 재설정 등)이며, 팀 외부 독자도 그 본질로 기억한다
- 수단은 본질을 달성하는 **방법**으로 부연하거나 기술 세부(`<details>`)에 배치

**리트머스 테스트**: "이 문장에서 수단 단어(서버명, 프로토콜명, 엔드포인트)를 지우면 의미가 남는가?" 답이 "남는다"여야 함.

**제목 예시 (나쁨에서 좋음 순)**

| 버전 | 제목 | 판정 |
|------|------|------|
| A | `[개선] 알림 발송 신규 큐 도입 (1단계)` | 수단 "큐"가 주어 (✗) |
| B | `[개선] 알림 발송 전용 워커 분리 (1단계)` | 수단 "워커"가 여전히 주어 (△) |
| C | `[개선] 알림 발송 책임 분리 (1단계)` | 본질 "책임 분리"가 주어 (✓) |

## 7. 단계별 효과는 그 시점 기준

단계를 나눈 작업(1/N단계, 단계 PR 등)의 기대 효과를 쓸 때, **최종 상태 전체의 효과**를 그 단계만의 효과처럼 쓰지 않는다.

- 예: 1단계에 병렬 경로만 추가하는 경우, "코드 중복 해소"는 1단계 완료로는 실현되지 않으므로(오히려 코드가 늘어남) 빼거나 **"최종 목표"**로 분리
- 의사결정자와 리뷰어가 **그 단계의 투자 대비 효과**를 정확히 판단할 수 있어야 한다

## 8. 내부 상수와 플래그명 노출 금지

"범위 외 / 후속" 섹션에서도 `MAX_RETRY_COUNT`, `isEnabled=Y` 같은 내부 상수와 플래그 이름을 그대로 옮기지 않는다. TODO/SRS의 표현을 복붙하면 새어 나오기 쉬우니, **사용자 관점 언어**로 바꾸거나 의미가 외부에 전달되지 않으면 삭제. 단계 번호(2단계/3단계)로 그룹핑하는 방식이 자연스럽다.

## 9. 정량과 절대 표기

모호한 양, 시간, 상태 표현은 **절대 표기**로 바꾼다. 외부 독자는 "최근", "곧", "대략", "몇 개" 같은 상대적이고 모호한 표현을 작성자와 다르게 해석하고, 문서가 시간이 지나면 그 표현은 의미를 완전히 잃는다.

| 구분 | 모호 (X) | 절대 (O) |
|------|---------|---------|
| **시간** | 최근, 조만간, 다음 주 | `2026-04-21`, `2026-04 말`, `2026-W17` |
| **수량** | 많다, 여럿, 대부분 | `28건`, `5개 중 4개`, `90%` |
| **버전** | 최신, 이전 버전 | `v2.3`, `v2.2` |
| **상태와 일정** | 곧 나옴, 작업 중 | `2026-04-25 예정`, `PR #297 진행 중` |

모를 때는 `TBD (2026-04-25 확정 예정)` 식으로 **모르는 상태 자체를 구체적으로** 표기. 상대 표현을 허용하면 작성 당시엔 편하지만 **문서 수명이 짧아진다**.

## 10. 추측과 단정 구분 (근거 출처 명시)

아직 검증 안 된 추정을 확정 사실처럼 쓰지 않는다. 외부 독자는 "이게 확정인가 추정인가"를 구분할 수 있어야 안심하고 의사결정에 사용한다.

- **확정 사실**은 근거 출처와 함께 (PR 번호, `파일:줄`, 공식 문서 인용, 측정값, 직접 인용 중 하나 이상)
- **추정과 가정**은 표시와 함께 (`추정`, `미확정`, `TBD (2026-04-25 확정 예정)` 등)
- **분기형 문장** 허용("A 조건이면 X, B 조건이면 Y"). 가정이 바뀌는 단계엔 단정보다 정정 비용이 낮음
- 가정이 3개 이상 얽힌 분석은 **"내가 가정한 것 N가지"**를 명시하고 독자가 검증할 기회를 줌

**리트머스 테스트**: 문서의 각 주장에 근거를 물었을 때 구체 출처(PR, `파일:줄`, 측정값, 직접 인용)로 답할 수 있는가? 없으면 `추정` 표시 필요.
