# Skill Contract

이 저장소의 모든 스킬이 따라야 할 원칙. 신규 스킬 추가 시 이 계약을 먼저 읽고 작성.
계약 위반 자동 검출은 `/skill-check` 메타-스킬이 담당.

## 1. Frontmatter

### 필수 필드
```yaml
---
name: kebab-case-name           # 디렉터리명과 동일
description: 1-2 문장 — 트리거 조건과 핵심 가치
disable-model-invocation: true  # 명시 호출만 허용 (자동 호출 방지)
allowed-tools: ...              # 사용 도구 명시 (와일드카드 Bash 지양 권장)
---
```

### 선택 필드
```yaml
effort: max | high | (default)            # 비싼 사고 비용 명시
context: fork                              # 산출물 격리 필요 시
argument-hint: "[--mode X] [--scope Y]"    # CLI 힌트
usage-scenario: solo | team | b2b | oss   # 사용 시나리오 (활성화 인프라 도입 후)
```

### 위반 사례
- `disable-model-invocation` 누락 → 모호한 자연어 요청에 자동 발화 위험
- description 이 "도와줍니다" 같은 모호한 표현 → 트리거 조건 불명확
- allowed-tools 가 단순 `Bash` → 위험 명령 (rm -rf 등) 거부 안 됨 (편의성 트레이드오프 인지)

## 2. 본문 구조

다음 섹션 순서를 따름 (해당 항목 없으면 통째 생략 — 빈 placeholder 금지).

**H1 금지** — frontmatter `name:` 이 스킬 제목의 SSOT(single source of truth)이며, 본문 H1(`# {스킬명}`)은 정보 중복이자 이름 변경 시 불일치 위험원. 본문은 `## 역할` 부터 시작한다.

```markdown
## 역할          # 무엇을 하는가, 톤(키워드만)
## 핵심 원칙     # 다른 스킬과의 경계, 검증·편향 방지
## 외부 데이터 의존  # 필수 — §4 참조
## 프로젝트 컨텍스트  # 호출 전 read 권장 파일
## 모드 판단     # (모드 있는 스킬만)
## 프로세스      # Phase 별 단계
## 출력 구조 / 산출물  # frontmatter + 마크다운 골격
## 저장 경로    # 산출 파일 위치
## 분량 임계   # 분할 정책 — §5 참조
## 다음 스킬 연결
## 제약조건     # do / don't 명확
## 검증 시나리오  # 필수 — §6 참조
```

## 3. 도메인 가정 외부화 원칙 (DRY)

**본문에 도메인 가정을 박지 않는다.** 외부 데이터로 위임.

### 위반 예시
```markdown
- 멀티테넌트 격리 — 모든 쿼리에 테넌트 키 조건 (review 본문 박힘)
- 결재 완료 후 수정 차단 (review 본문 박힘)
- {특정 도메인 테이블명}(헤더), {도메인 폴더 테이블} (특정 프로젝트 코드 예시 박힘)
- "고객사 담당", "회사 PR 템플릿" (직책·조직 가정)
```

### 올바른 패턴
```markdown
- 도메인 점검: `.local.claude/biz-rules.md` 의 도메인 카테고리 자동 활용 (Tier 2)
- 팀원 매핑: `.local.claude/team.md` 참조
- 고객 정보: `.local.claude/customers/{name}.md` 참조 (있을 시)
- 코딩 컨벤션: `CLAUDE.md` 참조
- 모듈 상세: `.local.claude/modules/{name}.md` 참조
- 인프라: `.local.claude/INFRASTRUCTURE.md` 참조 (있을 시)
```

### 구분 기준
| 본문에 박혀도 OK | 본문에서 빼야 함 (외부 위임) |
|---------------|----------------------|
| 일반 SW 패턴 (페이징·N+1·SQL injection) | 특정 도메인 (멀티테넌트·승인·도메인 약어) |
| 일반 워크플로우 (PR·이슈·리뷰) | 특정 조직 구조 (CEO·팀장·고객사) |
| 코드 검증 가능한 일반 규칙 | 코드만으로 알 수 없는 비즈니스 규칙 |

## 4. 외부 데이터 의존 (필수 표)

각 스킬 본문에 다음 표 포함:

```markdown
## 외부 데이터 의존

| 데이터 | 경로 | 필수/선택 | 부재 시 동작 |
|--------|------|:------:|------------|
| 도메인 점검 카테고리 | `.local.claude/biz-rules.md` 의 도메인 카테고리 섹션 | 선택 | "도메인 점검 생략" 안내 |
| 팀원 매핑 | `.local.claude/team.md` | 선택 | "git author 그대로 표기" |
| 모듈 상세 | `.local.claude/modules/{name}.md` | 선택 | 코드 직접 Read 로 fallback |
| ... | | | |
```

**부재 시 동작은 명시 필수**. "있어야 동작" 가정 금지 — 빈 프로젝트에서도 안내 메시지로 graceful degrade.

## 5. 분량 임계 (자동 분리)

| 산출물 유형 | 임계 1 (알림) | 임계 2 (강제 분리) |
|----------|:----------:|:--------------:|
| CLAUDE.md 자동 로드 | 200줄 | optimize-claude-md 위임 |
| biz-rules.md (자동 로드 후보) | 200 | 400 → biz-rules-detail.md |
| 일반 산출물 (SETUP.md 등) | 300 | 500 → 영역별 분리 |
| 진단 보고서 (*-diagnosis) | 500 | PART 별 분리 |
| 누적 데이터 (customers, modules) | 300 | 500 → 디렉터리 패턴 |

**핵심 원칙**:
- CLAUDE.md `@` 자동 로드는 핵심 파일만 — detail/ref 절대 `@` 금지
- 각 스킬은 갱신 시 자체 임계 검사
- garden 이 정원 관리 시 전수 검사 (이중 안전망)

## 6. 검증 시나리오 (필수)

### 6-1. 공통 3블록 (모든 스킬 공통 기본값)

다음 3블록은 **모든 스킬에 동일 적용**된다. 각 스킬 본문은 이 공통 기본값을 재진술하지 말고 "→ CONTRACT §6-1 참조"로 링크한 뒤, **6-2 고유 실패 시나리오만** 3~5줄로 작성한다.

```markdown
### 빈 프로젝트 (의존 데이터 0)
- 동작: §4 외부 데이터 의존 표의 모든 "필수" 항목이 없으면 즉시 중단 + 안내. 모든 항목이 "선택"이면 graceful degrade + `[데이터 부재]` 태그.
- 출력 예: "데이터 부재 — `/on-boarding` 또는 관련 데이터 작성 권장"

### 부분 데이터 (일부만 있음)
- 동작: 있는 데이터로 가능한 부분만 수행. 부재 영역은 `[데이터 부재]` 태그로 명시. 신뢰도(★1~5) 하향 표기.

### 풀 데이터
- 동작: 정상 동작 — 본문 프로세스 그대로 수행. Adversarial Review/자기 검증 포함.
```

### 6-2. 스킬 고유 실패 시나리오 (필수 — 스킬별 차별화)

각 스킬 본문의 "## 검증 시나리오" 섹션에는 **6-1 공통 블록 참조 한 줄 + 해당 스킬만의 고유 실패 시나리오 3~5줄**을 적는다. 다음 5개 카테고리 중 **2개 이상**에 해당하는 것을 골라 구체 예시로:

| 카테고리 | 예시 신호 | 권장 대응 |
|---------|---------|---------|
| **환경·규모** | git 저장소 2GB+, 파일 수 10만+, 네트워크 타임아웃 | AskUserQuestion 으로 범위 축소 옵션 제시 |
| **데이터 결함** | 30일+ 경과 문서, 모순 감지, 스키마 불일치 | `[데이터 충돌]` 태그 + 양쪽 기술, 사용자 확인 |
| **도메인 특수성** | standalone vs domain-native, 다중 테넌트 없음 | Tier 1만 수행 (도메인 무관 일반 점검) |
| **의존성 부재** | MCP 미연결, 파서 도구 미설치, API 제한 | 설치/연결 가이드 제공 후 중단 or 부분 수행 |
| **사용자 개입 필요** | 로그인 만료, 인증 실패, 권한 부족 | AskUserQuestion + 재시도 옵션 |

### 6-3. 작성 양식 (각 스킬 본문)

```markdown
## 검증 시나리오

공통 3블록(빈/부분/풀 데이터)은 CONTRACT §6-1 참조.

### 이 스킬의 고유 실패 시나리오

**[{카테고리}]** {구체 신호}
- 신호: {어떻게 감지하는가}
- 대응: {어떻게 처리하는가}

**[{다른 카테고리}]** {구체 신호}
- ...
```

**왜 필요한가**: 사용자가 받아서 자기 프로젝트에 적용 시 어디서 깨지는지 즉시 파악 가능. 공통 블록은 중복을 제거하고, 스킬 고유성은 차별화 정보로 남긴다.

## 7. 산출물 형식 (인덱싱 가능 구조)

### 7-1. 기본 형식

**필수**
- frontmatter (type, date, mode 등)
- anchor 가능한 헤더 (`## 1. ...`)
- 코드 레퍼런스 완전 경로 (`...` 축약 금지)

**권장**
- 표 (메트릭·비교)
- 정량 지표 명시 (`p50`, `4주 추세` 등)
- TL;DR 3줄 (긴 보고서)

**제외 (선택)**
- 회고·발산·대화형 산출물 (daily, memo, brainstorm, draft, pr, issue, coaching, leadership) 은 유연성 우선 — 위 형식 강제 안 함

### 7-2. 산출물 frontmatter 스키마 (공통)

각 스킬이 생성하는 산출물 파일은 다음 frontmatter를 포함한다. 후속 스킬(`/garden`, `/absorb`, `/learn`)이 읽어 인덱싱·신선도·보존 정책·지식 추출을 수행한다.

```yaml
---
created: YYYY-MM-DD         # 필수. /garden 이 신선도·retention 판단
category: {카테고리명}       # 필수. 보통 스킬명과 동일 (cs, daily, review, coaching 등)
retention: {기간}           # 필수. 30d | 90d | project-end | permanent
harvest_targets:            # 선택. /absorb·/learn 이 지식 추출 대상 glob
  - {glob 패턴}
---
```

**카테고리별 retention 기본값**:

| 스킬 유형 | retention |
|---|---|
| 회고·메모 (daily, daily-todos, memo) | 30d |
| 코칭·리더십 (coaching, leadership) | 30d |
| 진단 보고서 (*-diagnosis) | 90d |
| 프로젝트 문서 (prd, srs, rfc, adr, review) | project-end |
| PR·이슈·CS (pr, issue, cs) | 30d |
| 영구 참조 (biz-rules, customer-profile, people, team) | permanent |

**본문에서의 참조 형식**: 각 SKILL.md 본문은 전체 yaml 블록을 반복하지 말고 한 줄로 참조한다.

```markdown
Frontmatter (CONTRACT §7-2 표준): `category: cs, retention: 30d, harvest_targets: [customers/*.md]`
```

- `harvest_targets` 가 비면 해당 키 생략
- retention 이 위 표의 기본값과 같으면 `CONTRACT §7-2 기본값` 로 축약 가능

## 8. 다른 스킬과의 경계

각 스킬은 다음을 명시:

```markdown
## 다른 스킬과의 경계

| 질문 | 담당 | 이 스킬에서 |
|------|------|-----------|
| ... | /other-skill | ❌ 다루지 않음 |
| ... | **이 스킬** | ✅ 핵심 |
```

영역 침범 금지. 인접 스킬과 중복되는 영역은 위임 명시.

## 9. 다음 스킬 연결

각 스킬 하단에 다음 명시:

```markdown
## 다음 스킬 연결

- {조건} → `/{스킬}`
- 데이터 부재 → `/on-boarding` 또는 `/setup-guide`
- 산출물 분리 권장 → `/garden` 또는 자체 스킬
```

스킬 체이닝 = 사용자 발견성. 명령 하나로 시작해도 다음 단계 자동 안내.

## 10. 제약조건 (do/don't)

각 스킬은 자기 제약을 명시. 일반 원칙:

- **본문 도메인 가정 금지** (§3 참조)
- **부재 데이터 graceful degrade** (§4 참조)
- **민감 정보 마스킹** (시크릿·PII)
- **자동 갱신 전 사용자 확인** (덮어쓰기 위험)
- **사람 비난 금지** (시스템·패턴 분석)
- **추정값 표기** (`(추정, ★★)`)
- **권고 우선순위 명확** (P1/P2/P3)
- **외부-가시 스킬은 `rules/external-doc.md` 준수** — 외부 제3자(동료 개발자·의사결정자·이해관계자·고객)가 읽을 가능성이 있는 산출물을 생성하는 스킬(issue, pr, adr, prd, srs, cs, draft, poc, meeting-notes, rfc 등)은 10원칙(가독성 5 + 정직성 5) 공통 준수. 스킬 본문에는 참조 라인만 박고 원칙 재작성 금지

## 13. 마커·레이블 규약 (이모지 금지)

### 원칙

스킬 본문과 산출물에서 이모지를 쓰지 않는다.

**이유**: 이모지가 있어야 읽히는 문서는 텍스트 구조 자체가 부실하다는 신호다. 컬럼 헤더·섹션 제목·명시적 레이블로 해결해야 할 정보를 이모지로 보강하는 것은 구조적 부채이며, 이모지가 필수처럼 느껴지는 위치는 곧 리팩토링 대상이다.

### 대체 텍스트 마커 (표준)

| 용도 | 금지 이모지 | 표준 텍스트 마커 |
|---|---|---|
| 심각도 (리뷰·진단) | 🔴 🟠 🟡 🔵 | `[치명적]` `[주의]` `[참고]` `[보안]` |
| 상태 (체크·결과) | ✅ ❌ ⚠️ 🟢 ⚪ 🚨 | `[OK]` `[FAIL]` `[WARN]` `[정상]` `[미정]` `[URGENT]` |
| 공개 범위 (메모·산출물) | 🔓 🔒 | `[PUBLIC]` `[PRIVATE]` |

### 구조 보강 원칙

이모지를 제거한 뒤 "허전하다"고 느껴지면, 그 위치의 텍스트 구조가 약한 것이다. 다음 중 하나로 보강한다:
- 표에 `[심각도]`, `[상태]` 등 컬럼 헤더 추가
- 섹션 제목을 구체화 (예: `## 시작하기`, `## 핵심 요점`)
- 체크리스트를 하위 섹션으로 분리 (`### 완료 / ### 미해결`)
- 본문 문단 첫머리에 `**{레이블}**:` 형태로 명시

### 예외

- **사용자 제공 데이터 원본** (고객 발언 인용·실 대화 로그 등): 원본 그대로 보존
- **외부 시스템 API 요구사항** (GitHub 체크박스 `- [ ]` 등 특정 기호 강제): 해당 섹션만 명시적 예외
- **산출물 포맷이 외부 규약에 의해 강제되는 경우**: 해당 외부 규약 우선
- **유니코드 기호 (이모지 아님)** — `→ ← ↔ ★ ✓ ✗` 등 텍스트 방향·강도·체크 기호는 본 규약 대상이 아니며 그대로 사용 가능

예외에 해당하지 않는 모든 이모지 사용은 §12 위반 자동 검출 대상.
