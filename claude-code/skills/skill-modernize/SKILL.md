## 역할

새 Claude 모델(예: Opus 5), 새 도구(MCP, Skill tool, 신규 하네스 기능), 새 프롬프트 작성 규범(예: "긍정문 우선", "리터럴 지시")이 공개되면, 이 저장소의 모든 스킬이 새 능력을 **능동적으로 활용**하고 구세대 가정을 **현재 세대 표현으로 교체**하도록 유지한다.

## 핵심 원칙

### 이 스킬이 "선언적"인 이유

이 스킬은 **실행 방법을 본문에 박지 않는다**. 이유는 CONTRACT §3(도메인 가정 박힘 금지)의 메타 적용이다. "3 Explore 병렬", "Wave 6개 분할", "1M 동시 Read" 같은 구체 실행 방법은 **현재 모델 세대의 능력 가정**이며, 다음 세대에서 더 좋은 방식이 나오면 구식 관용구가 된다.

따라서 이 스킬은 **불변 목표 + 불변 점검 카테고리**만 박고, 실행 방법은 **호출 시점의 모델이 자기 능력을 스스로 파악하여 설계**하도록 위임한다.

### 불변 원칙 (세대가 바뀌어도 유지)

1. **선언적 지시 우선** — 실행 절차를 단계별로 박기보다 목표·제약·기준만 선언하고 실행은 현 세대 모델에 위임
2. **긍정문 우선** — "X를 하지 마라"보다 "Y를 하라"로 작성 (최근 프롬프트 가이드가 부정문보다 긍정문을 더 잘 따름을 보인 반영)
3. **리터럴 지시** — 추상 표현(예: "적절히", "필요시")보다 구체 경로·키워드·임계값 명시
4. **점검 카테고리만 고정** — "무엇을 보는가"는 불변, "어떻게 보는가"는 가변
5. **이전 실행 기록은 참조용** — 강제 템플릿 아님. 방법을 재사용하지 말고 "구식이 된 부분"을 역점검
6. **자기 부트스트랩 원칙** — 새 세대 호출 시 이 스킬을 **가장 먼저 점검**한다. 카테고리 5-b(현 세대 특화 예시), 카테고리 3(새 기능 리스트), 카테고리 5 수집원 URL 등 세대 의존 부분을 현 세대 기준으로 갱신한 뒤에야 다른 스킬 점검을 진행. 이 순서를 어기면 구세대 기준으로 스킬들을 "최신화"하는 모순이 발생

## 불변 점검 카테고리 (What)

다음 8개 카테고리는 세대가 바뀌어도 유지된다. 각 카테고리의 **세부 항목은 실행 시점에 현 세대 기준으로 재정의**한다.

### 카테고리 1. Frontmatter 최신성
- 필수 필드(name, description, disable-model-invocation, allowed-tools) 존재
- 현 세대에서 추가된 필드 반영 (예: 2026년 현재 기준 `model`, `effort`, `context`, `argument-hint`, `usage-scenario`)
- deprecated 도구·필드 제거
- `allowed-tools` 에 최신 도구명 반영 (MCP 도구, 신규 내장 도구 포함)

### 카테고리 2. CONTRACT 전 항목(§1~§13) 준수
- 핵심 점검: §1 Frontmatter / §2 본문 구조(H1 금지) / §3 도메인 가정 외부화 / §4 외부 데이터 의존 표 / §5 분량 임계 / §6 검증 시나리오 / §7-2 산출물 frontmatter 스키마 / §8 다른 스킬 경계 / §9 다음 스킬 연결 / §10 제약조건 / §13 이모지 금지·텍스트 마커
- CONTRACT.md 자체가 세대별로 개정될 수 있으므로 "현재 유효한 CONTRACT 버전"을 먼저 확인

### 카테고리 3. 새 세대 특화 기능 활용도
- 실행 시 WebSearch + 사용자 입력으로 "이번 세대의 새 기능 리스트"를 동적 수집
- 예시 수집원: Anthropic 릴리스 노트, `docs.claude.com`, 모델 시스템 카드, 하네스 changelog
- 수집한 기능별로 "이 스킬이 활용하는가?"를 점검
- **반영 방법을 본문에 박지 않는다** — 현 세대 모델이 적절히 판단

### 카테고리 4. 구세대 잔재 제거
- 이전 실행 기록의 "당시 반영했던 패턴 리스트" 역점검
- 예: 2026-06 Opus 4.8 전환 시 `[Opus 4.7 / 1M 활용]`·`[Opus 4.7 / 메타인지]` 류 모델명 스탬프를 `[1M 활용]`·`[메타인지]` 로 일반화함 — 차세대에서도 모델명 직접 박힘은 같은 방식으로 일반화
- 이전 모델명·이전 context 크기·이전 도구명 직접 언급 탐지 및 일반화

### 카테고리 5. 프롬프트 작성 규범 변화

규범이 세대마다 바뀌므로 실행 시점에 최신 가이드를 수집한 뒤 점검. 수집원: Anthropic 공식 Prompt Engineering 가이드, `docs.claude.com`, 모델 시스템 카드의 "prompting tips", Release notes.

각 규범은 현 세대 기준으로 유효성을 재판정하여 적용. 다음 세대에 반대 방향으로 바뀔 가능성을 항상 열어둔다.

#### 5-a. 세대 불변에 가까운 하위 규범 (장수 유효 추정)
- **리터럴 지시** — 추상 표현("적절히", "필요시") 대신 구체 경로·키워드·임계값 명시
- **Few-shot 예시 우선** — 추상 설명 3줄보다 구체 before/after 예시 1개 배치
- **"Don't simulate" 원칙** — "~인 척 해라", "~인 것처럼 답하라" 표현을 실제 수행 지시로 교체
- **선언형 균형** — 단계별 명령보다 목표·제약·기준 선언 비중 (스킬 본문 한정)

#### 5-b. 현 세대 특화 예시 (Opus 4.8 / 2026-06 기준 — 세대 전환 시 이 리스트 자체가 역점검 대상)

> 아래 항목들은 `/skill-modernize` 실행 시점의 현 세대 경향이다. 다음 세대 호출 시 **이 스킬부터 먼저 점검**하여 이 리스트를 갱신한 후 다른 스킬 점검을 진행한다 (자기 부트스트랩 원칙).

- **긍정문 우선** — "X를 하지 마라" 대신 "Y를 하라". 다음 세대에서 부정문 처리 개선 시 약화 가능
- **구조화 태그 활용** — XML 태그(`<context>`, `<example>`, `<instruction>`) 또는 명시적 구분자. 차세대에서 JSON-native·markdown-native 등으로 선호 변화 가능
- **Chain-of-thought / 사고 유도** — "먼저 계획을 풀어라" 트리거 또는 adaptive thinking(4.8 기본). 4.8은 사고 깊이를 자동 조절하므로 과도한 "think"·"단계별로 생각" 유도는 정리. `budget_tokens`·extended thinking 표현은 deprecated
- **Persona 과잉 지양** — "당신은 20년 경력 시니어입니다" 류 역할 부여 과다 탐지. 이전 세대에선 persona가 도움됐으므로 세대 간 방향이 반대로 바뀐 사례
- **Prompt caching 배치 원칙** — 정적 컨텍스트는 앞, 동적 입력은 뒤. API 진화에 따라 세부 지침 변동
- **출력 형식 강제 방식** — JSON/YAML 강제 시 직접 지시 vs 스키마 제공 vs prefill 중 현 세대에서 가장 잘 따르는 방식. 세대마다 최적 방식 교체
- **본문 H1 금지 + frontmatter name SSOT** (CONTRACT §2) — SKILL.md는 frontmatter `name:` 이 스킬 제목의 단일 진실 원천. 본문 `# {스킬명}` H1은 정보 중복이자 이름 변경 시 불일치 위험원. 차세대에서 H1이 다른 용도로 재정의되거나 프론트매터 외 제목 규약이 표준화되면 방향 전환 재점검
- **이모지 대신 텍스트 마커** (CONTRACT §13) — 심각도·상태 표시는 `[치명적]`/`[OK]`/`[WARN]` 같은 텍스트 마커를 우선. 현 세대는 이모지 의존이 구조 약점(컬럼 헤더·섹션 제목 부재)의 덮개로 해석됨. 차세대에서 이모지 의미 파싱·멀티모달 처리가 향상되거나 이모지 렌더 환경이 UX 필수가 되면 허용 범위 확대 가능
- **Frontmatter 프롤로그 본문 반복 제거** (CONTRACT §7-2) — 산출물 yaml 스펙을 본문에 ```yaml 블록으로 반복하지 말고 CONTRACT §7-2 참조 한 줄로 압축. 차세대에서 메타데이터 스키마 정의·배포 표준이 바뀌면 참조 형식 재점검
- **과잉 narration 정리** (Opus 4.8) — 4.8은 도구 호출 사이·종료 요약을 기본적으로 더 길게 함. "N단계마다 진행 요약" 류 강제 스캐폴딩은 제거하고, 코딩·절차형 스킬엔 "불필요한 중간 narration 자제" 기본값. 차세대에서 narration 기본값이 또 바뀌면 재점검
- **자율성/ask-rate 가이드** (Opus 4.8) — 4.8은 사소한 결정도 더 자주 되물음. "사소한 선택(이름·기본값·동등 대안)은 정하고 기록, scope 변경·파괴적 작업만 확인" 명시. 차세대에서 ask 성향이 바뀌면 재조정
- **능력 트리거 명시** (Opus 4.8) — 4.8은 서브에이전트·메모리·검색·커스텀 도구를 기본적으로 덜 씀. "언제 위임/검색/메모리 사용" 트리거를 본문과 도구 설명(prescriptive description)에 명시. 차세대에서 기본 사용 성향이 오르면 완화
- **severity 필터 위치** (Opus 4.8, 리뷰·진단) — 4.8은 "high-severity만"·"보수적" 지시를 곧이곧대로 따라 recall 하락. 발견 단계는 전부 보고 + `[심각도]`·`[확신도]` 태깅, 필터는 별도 단계로 분리. 차세대에서 필터 정밀도가 바뀌면 재점검

### 카테고리 6. 분량·상호참조 건전성
- 본문 600줄 초과 스킬 분할 검토
- 깨진 `/skill-name` 참조 탐지
- 고립 스킬(Inbound=0) 연결 강화
- 순환 참조 탈출 조건 명시

### 카테고리 7. 안전·거절 규범 변화

모델마다 안전 거절 카테고리가 추가·완화된다. 세대 전환 시 다음을 점검:

- 보안 관련 스킬(`/security-diagnosis`, `/review` 의 보안 축 등)에 현 세대가 거절할 수 있는 지시가 포함되어 있는지 시뮬레이션
- 듀얼 유즈(정당한 보안 테스트 vs 악용) 맥락에서 스킬이 "정당한 용도임을 충분히 선언"하는지 점검
- 새로 추가된 안전 카테고리에 해당하는 스킬 본문 표현 교체
- 과거 세대에서 필요했던 "안전 우회 표현"이 현 세대에서 불필요해졌는지 점검

### 카테고리 8. 비용·캐시·성능 최적화

- 스킬이 호출하는 도구·파일 수의 상한 적정성 (token 예산)
- Prompt caching hit 가능성 — 정적 참조(CONTRACT.md, rules/) 배치 방식
- Streaming·병렬 실행 활용도 (현 세대가 지원하는 방식 기준)
- 불필요한 컨텍스트 재로드 패턴 탐지 (같은 파일을 여러 번 Read 지시)
- 산출물 크기 상한 (사용자가 한 번에 읽을 수 있는 분량)

## 가변 실행 방법 (How) — 호출 시점 모델이 자율 설계

### 실행 방법을 자유롭게 정하라

현 세션에서 사용 가능한 능력을 먼저 확인하고, 그에 최적인 방법을 선택한다. 다음 중 어느 조합이든 현 세대에 맞으면 사용:

- 단일 메시지 병렬 도구 호출
- Explore/general-purpose 서브에이전트 위임 (병렬 또는 순차)
- 메인 컨텍스트에서 다중 파일 동시 Read 후 교차 분석
- TaskCreate 로 단계 추적
- AskUserQuestion 으로 사용자 확정
- 위에 없는 **현 세대의 새 능력**

각 카테고리 점검 시에도 "어떻게 점검하는가"는 현 세대 기준으로 선택. 이전 실행 기록의 방법은 참고만 하고 강제 재사용은 피한다.

### 모델·도구 정보 수집은 반드시 동적으로

호출 시점에 다음을 확인한 뒤 실행 계획을 세운다:

1. 현 세션 모델의 식별자·능력 (system prompt 또는 사용자 입력)
2. 현 세션에서 사용 가능한 도구 목록 (deferred tools 포함 가능성)
3. Anthropic 공식 릴리스 노트 (WebSearch)
4. 사용자에게 "이번 세대에서 가장 큰 변화가 무엇인가" 질문 (`AskUserQuestion`)

## 프로세스 (목적만 고정, 순서·방법 자유)

4가지 목적을 모두 달성한다. 순서는 의존성이 있을 때만 지키고 나머지는 자유.

### 목적 A. 감사 (현황 파악)

**A-0. 자기 점검 선행 (필수 첫 단계)**
- `skills/skill-modernize/SKILL.md` 자신을 **가장 먼저** Read
- 카테고리 5-b(현 세대 특화 예시) 리스트가 현 세대 기준인지 확인
- 카테고리 3의 수집원 URL·Anthropic 공식 참조가 유효한지 확인
- 구식화된 예시가 있으면 이 스킬을 먼저 수정한 뒤 다른 스킬 감사 진행
- 선행 없이 다른 스킬을 감사하면 구세대 기준으로 최신화하는 모순 발생

**A-1. 다른 스킬 감사**
- 8개 카테고리 각각에 대해 현 시점 스킬 수(`ls -d skills/*/ | wc -l`로 동적 산출)의 현황 조사
- 산출: 카테고리별 위반·개선 후보 목록

### 목적 B. 계획 (시급도 분류 + 사용자 승인)
- 감사 결과를 P1(즉시)/P2(구조 결손)/P3(개선) 로 분류
- 사용자에게 `AskUserQuestion` 으로 승인 요청
- 거대 수정은 옵션 제시 (예: 전체 vs P1만 vs 감사만)

### 목적 C. 실행 (수정 집행)
- 승인된 수정을 적용
- 긍정문·리터럴·선언형 원칙 준수
- 수정 규모가 크면 현 세대 능력에 맞는 병렬화·분할 전략 선택

### 목적 D. 검증 (커버리지·회귀 확인)
- 각 카테고리별 Before/After 커버리지 측정
- `/skill-check` 재호출로 CONTRACT 위반 재확인
- 수정 중 새로 생긴 문제(예: 제목 중복 잔재) 탐지 및 수정

## 이전 실행 기록 참조 방식

이전 기록은 **강제 재사용이 아닌 역점검 재료**로 사용한다.

다음 순서로 활용:
1. 최신 1~3개 기록 Read
2. 기록 속 "당시 반영한 패턴" 리스트 추출 (예: "2026-06: `[Opus 4.7]` 스탬프 → `[1M 활용]` 일반화")
3. 현 세대 기준으로 각 패턴의 유효성 재판정 — 아직 유효하면 보존, 구식이 됐으면 카테고리 4 대상에 추가
4. "이전에 어떤 방식으로 실행했는가"는 현 세대 실행 계획의 힌트로만 사용 (강제 아님)
