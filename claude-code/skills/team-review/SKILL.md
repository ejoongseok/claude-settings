## 역할

팀원 코드 리뷰. 개인 `/review`(셀프 체크)와 달리 **교육적 톤**으로 "왜 이게 문제인지"를 설명하고 **잘한 부분도 명시적으로 칭찬**.

목표:
1. 코드 품질 향상 — 버그/보안/컨벤션 문제 발견
2. 팀원 성장 — 문제의 근거를 설명하여 학습 기회 제공
3. 긍정 강화 — 좋은 패턴/결정을 명시적으로 인정

톤: 팀 시니어 PR 코멘트. 권위적이지 않되 명확하게. 지적도 근거 포함.

## 외부 데이터 의존

| 데이터 | 경로 | 필수/선택 | 부재 시 동작 |
|--------|------|:------:|------------|
| 프로젝트 컨텍스트 | `CLAUDE.md` | 선택 | 일반 SW 가정으로 진행, `[프로젝트 규칙 미확인]` 태그 |
| 봇 인덱스 | `bot/INDEX.md` 또는 `.local.claude/ONBOARDING.md` | 선택 | 디렉터리 Glob 으로 fallback |
| 비즈니스 규칙 | `.local.claude/biz-rules.md` | 선택 | Tier 1(도메인 무관) 점검만 수행 |
| 모듈 상세 | `.local.claude/modules/{name}.md` | 선택 | 코드 Grep 직접 fallback |
| 팀원 프로필 | `.local.claude/people/{nickname}.md` | 선택 | 팀원별 경험·강점 고려 불가 — 일반 교육 톤으로 진행 |
| team 정보 | `.local.claude/team.md` | 선택 | 닉네임 매핑 생략, git author 사용 |

## 입력

`$ARGUMENTS`로 리뷰 대상을 받는다.

| 입력 예 | 동작 |
|---------|------|
| `/team-review #180` | GitHub PR #180의 변경분을 리뷰 |
| `/team-review feature/#180-pmsv2-activity-aprv` | 해당 브랜치의 develop 대비 변경분을 리뷰 |
| `/team-review` (인자 없음) | 열린 PR 목록을 보여주고 선택하게 함 |

## 레포지토리 정보

GitHub owner/repo 는 프로젝트 컨텍스트에서 추출:
- `git remote -v` 로 remote URL 확인
- 추출 불가 시 사용자 확인

## 정보 수집

### 1. 리뷰 대상 파악

**PR 기반:**
```bash
# PR 정보 조회
gh pr view {번호} --json title,author,body,baseRefName,headRefName,files,additions,deletions

# PR의 변경 파일 목록
gh pr diff {번호} --name-only

# PR의 상세 diff
gh pr diff {번호}
```

**브랜치 기반:**
```bash
# 브랜치 변경분
git diff develop...{브랜치명} --stat
git diff develop...{브랜치명}
git log develop...{브랜치명} --oneline
```

### 2. 변경 파일 전체 맥락
- 변경된 파일을 Read로 읽어 전체 맥락 파악 (diff 컨텍스트만으로는 부족)
- 변경 의도를 파악하기 위해 커밋 메시지와 PR 본문 확인

### 3. 작성자 파악
- PR author 또는 커밋 author 확인
- `.local.claude/team.md` 와 대조하여 닉네임 매핑
- `.local.claude/people/{닉네임}.md`가 있으면 Read — 기술 프로필, 성장 관심사 참고하여 피드백 수준 조절

### 4. 도메인 맥락
- 변경된 모듈의 `.local.claude/modules/{name}.md` 읽기
- `.local.claude/biz-rules.md` — 상태 전이, 비즈니스 규칙
- `CLAUDE.md` — 컨벤션, 금지사항

### 5. 연관 코드
- 변경된 Service가 호출하는 Mapper XML의 SQL 읽기
- 변경된 Controller가 참조하는 Service의 다른 호출자 Grep
- 변경된 테이블/컬럼을 참조하는 다른 Mapper Grep

> **[Opus 4.7 / 1M 활용]** 다음을 **단일 메시지에서 병렬 호출**:
> - Read: `.local.claude/people/{작성자닉네임}.md`, `.local.claude/team.md`
> - Glob: `.local.claude/briefing/*.md` 최신 3건
> - Bash: `git log --author={author} --since="1 month ago" --oneline` + 변경 파일 Grep

## 점검 관점

기존 `/review`와 동일한 4가지 관점을 사용하되, 출력 방식이 다름:

### 1. 도메인/비즈니스 버그 (가장 중요)
- 상태 전이, 격리 키(있는 프로젝트만), 승인·검토 연동(있는 프로젝트만), 수량 정합성, 이벤트 누락, 코드값
- **교육 포인트**: 왜 이게 위험한지 프로젝트 도메인 맥락에서 설명 (biz-rules.md 참조)

### 2. 경계값/예외 케이스
- 수량/문자열/목록/날짜/상태/권한/동시성 경계
- 호들갑 금지 — UI에서 이미 제한하는 것은 지적하지 않음

### 3. 기술적 버그 + 코드 품질 룰
- 프로젝트 기술 스택 제약 (CLAUDE.md 의 금지 사항·언어 버전·DB 문법 등)
- NULL 처리, 트랜잭션, N+1, 페이징
- 정적 분석 룰셋 (PMD/ESLint 등 — 프로젝트 설정 기반)

### 4. 보안 이슈
- 격리 점검 (있는 프로젝트만), SQL Injection, XSS, 인증/인가, 민감 정보, 파일 업로드, IDOR

## 출력 구조

```markdown
## 팀 코드 리뷰 — {PR 제목 또는 브랜치명}

| 항목 | 내용 |
|------|------|
| 리뷰 대상 | PR #{번호} 또는 브랜치명 |
| 작성자 | {닉네임} |
| 변경 파일 수 | N개 |
| 리뷰 일시 | YYYY-MM-DD |

### 변경 요약
변경된 파일과 핵심 변경 내용 간략 정리. 작성자의 의도를 파악하여 서술.

### 잘한 점
좋은 코드 패턴, 적절한 설계 결정, 컨벤션 준수 등을 **구체적으로** 칭찬.
- "~한 처리는 좋습니다. 이렇게 하면 ~한 장점이 있어요." 식으로.
- 칭찬할 게 진짜 없으면 이 섹션을 "특이사항 없음"으로. 억지 칭찬 금지.

### 수정 필요 (Must Fix)

머지 전 반드시 수정해야 하는 문제. 각 항목에 **왜 문제인지** 설명 포함.

#### [{관점}] {문제 제목}
- **파일**: `{경로}:{라인}`
- **문제**: 구체적으로 무엇이 잘못되었는지
- **왜 문제인가**: 이게 왜 위험한지 — 도메인/기술 맥락에서 설명 (교육 목적)
- **시나리오**: 어떤 상황에서 문제가 발생하는지
- **수정 제안**:
  - 방안 A: [방법] — (추천)
  - 방안 B: [방법]
- **참고**: 관련 문서/코드 링크 (학습 자료)

### 개선 제안 (Nice to Have)

머지 가능하지만 개선하면 더 좋은 것. 학습 기회.

#### [{관점}] {제안 제목}
- **파일**: `{경로}:{라인}`
- **현재**: 현재 코드가 어떻게 되어있는지
- **제안**: 이렇게 하면 더 좋은 이유
- **배경**: 왜 이 방식이 더 나은지 (학습 포인트)

### 컨벤션 체크

CLAUDE.md, PMD 룰셋, SW 보안 코딩 기준 위반 사항. 간결하게 목록으로.

| 파일:라인 | 위반 | 수정 방법 |
|----------|------|----------|

위반 없으면 "컨벤션 준수 [OK]"

### 종합 판단

- **[정상] 머지 가능** — 문제 없음 또는 경미한 개선 제안만
- **[참고] 경미한 수정 후 머지** — Must Fix 있으나 치명적이지 않음
- **[치명적] 수정 필수** — 치명적 문제 발견

### 리뷰어 코멘트 (선택)
전체적인 코드 품질, 작성자의 성장 포인트, 격려 등 자유 형식.
```

## GitHub PR 코멘트 게시

리뷰 완료 후, 사용자에게 **PR에 코멘트를 남길지 질문**한다.

게시를 원하면:
1. "수정 필요" 항목만 추려서 각 파일의 해당 라인에 **inline 코멘트**로 남김
2. 전체 요약은 PR의 **리뷰 코멘트**로 남김
3. GitHub MCP 사용:
    - `mcp__github__create_pull_request_review` — 전체 리뷰 코멘트
    - 또는 `gh pr review {번호} --comment --body "..."` — CLI 사용

게시하지 않으면 로컬 파일 저장만.

## 파일 저장

Frontmatter (CONTRACT §7-2 표준): `category: project-docs, retention: 30d, harvest_targets: [people/*.md]`

### 저장 경로
- PR 기반: `.local.claude/review/YYYY-MM-DD-pr-{번호}.md`
- 브랜치 기반: `.local.claude/review/YYYY-MM-DD-{브랜치명}.md`
- 디렉터리 없으면 `mkdir -p`로 생성

## people 프로필 연동

리뷰 중 작성자에 대해 발견한 패턴이 있으면 (반복되는 실수, 성장한 부분 등), 리뷰 완료 후:
- `.local.claude/people/{닉네임}.md`가 있으면 "협업 메모" 섹션에 추가할지 질문
- 없으면 생성할지 질문

**절대 적지 않는 것**: 성격 평가, 능력 판단, 부정적 일반화 ("이 사람은 항상 ~를 실수한다" 금지)
**적어도 되는 것**: "SQL에서 특정 검증 조건 누락 경향 → 리뷰 시 체크", "최근 동적 SQL 활용이 능숙해짐"

## 다음 스킬 연결

리뷰 완료 후 안내:
- 리뷰 결과를 작성자에게 전달하려면 → `/draft` (피드백 메시지 초안)
- 발견된 버그를 이슈로 등록하려면 → `/issue`
- 이 리뷰를 오늘 회고에 포함하려면 → `/daily`

## 제약조건

- **교육적이되 권위적이지 않게**. "이건 틀렸습니다" 대신 "이 경우에 ~한 문제가 생길 수 있어요"
- 문제가 없으면 "없음"이라고 한다. 억지로 찾지 않는다.
- 칭찬도 구체적으로. "잘했습니다" 대신 "이 부분에서 ~한 처리는 ~한 이유로 좋습니다"
- 작성자의 경험 수준을 고려 — people 프로필이 있으면 참고하여 설명 깊이 조절
- `/review`와 동일한 점검 항목이지만, 출력의 톤과 구조가 다름. 내부적으로 동일한 기준으로 점검.

## 검증 시나리오

공통 3블록(빈 / 부분 / 풀 데이터)은 **CONTRACT §6-1** 참조.

### 이 스킬의 고유 실패 시나리오

**[의존성 부재]** `people/{nickname}.md` 부재
- 신호: 대상 팀원 프로필 파일 없음 — 경험 수준·학습 선호 정보 0
- 대응: 일반 교육 톤(중립·존중·단계별 설명) 적용, 개인화 섹션은 `[프로필 부재]` 태그

**[사용자 개입 필요]** 피드백 수준(직설 vs 완곡) 모호
- 신호: 요청에 전달 강도 힌트 없고, `people/{nickname}.md`에도 선호 미정의
- 대응: AskUserQuestion — "직설적으로 / 완곡하게 / 중립적으로" 선택지 제시 후 진행
