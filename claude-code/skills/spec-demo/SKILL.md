# spec-demo

문서로 "설명하는" 명세가 아니라, 직접 만져보며 합의하는 **동작하는 명세**를 만든다. 목적은 정확한 보고가 아니라 **이해관계자 정렬** — 추상 규칙을 실행 가능하게 만들어 "말로는 안 맞춰지는 오해"를 빨리 드러내는 것. 성공 기준은 예쁨이 아니라 "이견이 드러나고 같은 그림에 도달했는가".

## 입력

- 정렬이 필요한 **케이스 하나**(문서 전체가 아님): 상태 전파·이벤트 연쇄·조건부 변경·유스케이스 등 "오해가 생기는 동작".
- 파일 경로(SRS/PRD/회의록) 또는 구두 설명.

## 절차

1. **실제 규칙 추출 — 추측 금지.** 프로젝트의 코드·명세·도메인 규칙에서 트리거·규칙·결과를 확인한다(Grep/Read). 이벤트 기반이면 발행/구독 지점을, 상태 기반이면 상태값·전이를 추적한다. **불확실한 규칙은 지어내지 말고** `note.warn`으로 **"합의 포인트"** 표기 — 이게 정렬 도구의 정수다.
2. **케이스 → 표현 선택 — 단일 패턴 강요 금지.** `assets/components.md`의 "케이스 → 표현 선택" 표에서 고른다(전파→집계/조건부, 시간순→시퀀스, 상태 변화→상태도, 흐름·분기→플로우차트, 행×열→매트릭스 등). 해당 표현의 `assets/playbook.md` 학습을 먼저 read.
3. **생성.** `assets/components.md`의 스킨 CSS를 `<style>`에 인라인 + 표현 컴포넌트로 self-contained 인터랙티브 HTML. **구조가 복잡한 표현은 `assets/examples/`의 검증된 데모를 복사**해 도메인 데이터만 케이스에 맞게 교체한다. 보는 사람이 값을 바꿔보며 결과를 확인하게 한다(슬라이더·트리거·폼·세그먼트). 케이스가 추상적이면(집계 등) hero를 질문형으로.
4. **저장.** 작업 폴더의 `rendered/{YYYY-MM-DD}-{slug}.html`(폴더 없으면 생성). 저장 후 경로 안내. 열기는 OS에 맞게 — macOS `open <경로>` / Linux `xdg-open <경로>` / Windows `start "" <경로>`. 원본 미변경 확인.
5. **footer = 최소 또는 생략.** 데이터 출처·가정이 본문 `note`에 있으면 footer 없이도 된다. 메타·자랑 금지. 코드 출처는 화면이 아니라 HTML 주석에 보존.
6. **피드백 시** `assets/playbook.md` 해당 표현에 1줄 append(날짜·간결).

## 표현 (상세·케이스 매칭 assets/components.md)

- **스킨으로 직접**: 집계 · 조건부 전파 · 이벤트 연쇄 · 유스케이스 · 비교 · 매트릭스(행×열) · 타임라인
- **examples/ 복사 (구조 복잡)**: 상태도 · 시퀀스 · 플로우차트(2D 분기) · ER 전파 · 아키텍처 토폴로지 · 다단계 트리(소요량) · 유스케이스 다이어그램(역할·권한) · 화면 흐름(CRUD 프로토타입)

표현은 닫힌 목록이 아니라 케이스에 맞춰 고르고, 새 표현은 playbook에 누적한다.

## 비협상 제약

- self-contained — Pretendard CDN 1개만, 그 외 CSS·JS 인라인. 외부 다이어그램 라이브러리(mermaid 등) 금지.
- 실제 규칙 기반 — 가짜 규칙 금지. 가정은 "합의 포인트"로 명시.
- 원본 코드·문서 불변 — 항상 별도 `.html`. Readable Light 스킨 고정. 이모지 금지.
- **affordance 정직** — 동작은 button/input/세그먼트. 정적 표시(상태·거쳐온 경로)는 버튼처럼 보이는 알약 칩 금지 → 경로 텍스트·단계로.
- **표현 기법명(상태도·시퀀스·2D 분기 등)을 화면 eyebrow/라벨로 노출 금지** — eyebrow는 케이스 상황·맥락으로.

## 모드

- **기본**: 규칙 가정 명시. 코드 출처는 화면이 아니라 HTML 주석에 보존.
- **공유**(`--external`): 내부 경로·코드·자랑 제거, 비즈니스 언어, "합의용 예시"만.

## 워크플로우 위치

`prd`/`srs`/`todo` 스킬이 문서를 만들면, 그 안에서 오해가 생길 핵심 케이스를 spec-demo가 동작하는 화면/명세로 띄워 합의한다.

- **PRD 단계**("이렇게 생각한 거 맞지?") — 화면 흐름 · 유스케이스 다이어그램 · 비교로 의도·화면 합의.
- **SRS/TODO 단계**("그래서 진짜 이렇게 동작/구현한다") — 전파 · 이벤트 연쇄 · 상태도 · 시퀀스 · ER로 규칙 합의.

## UI 화면 목업이 필요하면

이 스킬은 동작·규칙 정렬용이다. 픽셀 단위 화면 목업이 필요하면 별도 UI 디자인 스킬에 위임하고(출력은 self-contained HTML로), 그 위에 합의 포인트를 얹는다.
