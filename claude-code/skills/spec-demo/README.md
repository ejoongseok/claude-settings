# /spec-demo

도메인 동작과 규칙을 **직접 만져보며 합의하는 "동작하는 명세"**로 만듭니다. 문서나 그림으로 설명하는 대신, 트리거를 누르고 값을 바꿔보면 규칙대로 결과가 변하는 self-contained 인터랙티브 HTML입니다.

## 왜 쓰나

"이렇게 동작합니다"를 말, 문서, 그림으로 합의하면, 다들 끄덕이지만 각자 다른 그림을 떠올립니다. 그 오해는 구현이 끝난 뒤에야 터집니다. spec-demo는 추상 규칙을 **실행 가능하게** 만들어, 합의 자리에서 이견이 즉시 드러나게 합니다.

## 쓰면 얻는 것

- **말로 안 맞춰지던 오해가 드러남**: 두 해석(예: 단순평균 vs 가중평균)을 나란히 실행해 차이를 눈앞에 보여줍니다.
- **불확실한 규칙은 곧 합의 포인트**: 코드로 단정 못 하는 부분을 지어내지 않고 "여기 맞춰야 함"으로 표시합니다.
- **예외 흐름까지 합의**: 정상 흐름만이 아니라 검증 실패나 조건 분기 같은 "터지는 지점"을 미리 맞춥니다.
- **파일 하나로 공유**: 단일 `.html`, 외부 의존 거의 없음(폰트 1개). 회의에서 그대로 열고 만집니다.
- **원본은 안전**: 코드와 문서를 수정하지 않습니다.

## 무엇을 얻으려는 건가 (목적)

정확한 보고서가 아니라 **이해관계자 정렬**입니다. 성공 기준은 "예쁘다"가 아니라 **"이견이 드러나고 같은 그림에 도달했는가"**입니다. Figma/PPT가 "보여주기"라면, 이건 "맞춰보기"입니다.

## 이럴 때

상태 전파, 이벤트 연쇄, 조건부 변경, 유스케이스 검증처럼 **"코드만 봐선 안 보이고, 말로는 안 맞춰지는 동작"**을 고객, 기획, 개발이 같이 맞춰야 할 때.

## 표현 (케이스에 맞춰 선택)

- **간단히 생성**: 집계, 조건부 전파, 이벤트 연쇄, 유스케이스, 비교, 매트릭스(권한이나 상태×액션), 타임라인
- **검증된 예시 활용**: 상태도, 시퀀스, 플로우차트, ER 전파, 아키텍처 토폴로지, 다단계 트리(소요량), 유스케이스 다이어그램, 화면 흐름(CRUD)

PRD 단계("이 화면과 흐름 맞지?")엔 화면 흐름, 유스케이스, 비교를, SRS/TODO 단계("진짜 이렇게 동작")엔 전파, 이벤트, 상태도, 시퀀스, ER을 씁니다. `prd`/`srs`/`todo` 다음 단계로 핵심 케이스를 동작으로 합의.

## 설치

이 폴더를 `~/.claude/skills/` 또는 프로젝트 `.claude/skills/` 아래에 그대로 두면 됩니다. 외부 파일 의존이 없습니다(웹폰트 CDN 1개 제외).

## 사용법

- `/spec-demo "주문이 취소되면 어떤 항목이 환불 대기로 돌아가나"`: 구두 케이스
- `/spec-demo specs/order-flow.md`: 문서에서 케이스 추출
- `/spec-demo ... --external`: 외부(고객) 공유 모드(내부 경로와 코드 제거)

## 언제 쓰나 / 언제 아닌가

말로는 오해가 생기는 동작(전파, 이벤트, 상태 전이, 엣지 검증)을 합의할 때 가치가 큽니다. 단순 CRUD엔 과잉이고, 집계처럼 추상적인 케이스보다 조건부 전파, 이벤트 연쇄, 유스케이스 검증이 더 효과적입니다.

## 품질 기준

디자인과 문서 품질은 확립된 공개 원칙에 근거합니다: Nielsen, Norman, Rams, Gestalt, Laws of UX, WCAG, Diátaxis 등 (`assets/design-principles.md`).
