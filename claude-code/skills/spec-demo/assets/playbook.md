# spec-demo 케이스 학습 로그

> 표현별 "효과적이었던 것 / 수정과 회피 이력"을 누적한다. 생성 전 해당 표현 read, 피드백 시 1줄 append(날짜, 간결). 원칙(고정)은 `design-principles.md`, 여긴 학습(누적).
>
> 아래는 일반 시드. 쓰면서 자신의 케이스로 채워가세요.

## 공통

- footer는 생략 가능. 데이터 출처와 가정이 본문 `note`에 있으면 footer 없이도 된다. 메타와 자랑 금지.
- affordance: 정적 상태 표시를 알약 칩으로 그리면 "클릭되는 줄" 오해. 현재 상태는 큰 텍스트, 거쳐온 상태는 경로(모노 + 화살표). 칩 막대 금지.
- 입력 선택지는 밋밋한 기본 `<select>` 대신 세그먼트 버튼.
- 불확실한 규칙은 지어내지 말고 `note.warn` "합의 포인트"로. 데이터와 규칙은 실제 기준, 가정만 명시.
- 케이스 강도 차이: 집계가 가장 추상적이라 약하다. 조건부 전파, 이벤트 연쇄, 유스케이스 검증이 "코드로 안 보이는 것"을 보여줘 더 강하다. 추상 케이스는 hero를 질문형으로.
- 다이어그램은 한 종류가 아니다. 케이스 성격에 맞는 표현을 고른다. 시간순과 연쇄는 시퀀스, 상태 변화는 상태도, 흐름과 분기는 플로우차트. 단일 패턴 강요 금지.
- ER 순수 구조도(관계선)는 정적이라 약하다. 예시 레코드 + FK 색 매칭 + 클릭 강조는 유효. 순수 구조 설명은 정적 문서로.
- 예시 데이터는 컬럼-값 매핑을 명확히. 값만 나열하면 어느 컬럼인지 모른다. 컬럼 헤더 미니표 + FK는 같은 색 매칭.
- 토글 해제는 별도 함수로(`clearFocus`). 토글에 null 인자를 또 넘기면 해제가 꼬인다.
- 표현 기법명을 화면 라벨로 노출 금지. eyebrow에 "상태도, 시퀀스, 2D 분기" 같은 기법명 금지. eyebrow는 케이스 상황과 맥락으로 쓰거나 생략.
- 단계 재생은 양방향 + 첫 버튼 "시작". 거쳐온 화살표/경로도 되돌릴 때 같이 해제. 시퀀스와 플로우차트 공통.
- 상태도와 플로우차트는 특정 노드 직접 클릭으로 점프 + 정/역 전이 버튼은 규칙 표현. 플로우차트는 한 줄 나열 금지, 분기를 2D로.
- 이벤트는 self loop 금지, 별도 참여자(리스너) 화살표로. 색과 범례로 "자동 처리(이벤트)" 설명.
- 영향도와 우선순위 라벨엔 기준을 명시. 판단 자체가 합의 대상임을 밝힌다.

## 표현별

- **집계**: 규칙 후보를 나란히 계산(예: 단순평균 vs 가중평균)해 이견 노출. 질문형 hero + 극적인 프리셋 버튼. 초기값이 모두 같으면 차이가 0이니, 슬라이더 조작 시 갈림.
- **조건부 전파**: A 세그먼트 조작 시 B에서 조건 맞는 행만 `.changed`(노랑)/안 맞으면 `.kept`(흐림). 필터 조건이 합의 포인트. 트리거는 실제 규칙, 필터는 가정.
- **이벤트 연쇄**: 트리거, 이벤트 노드, 리스너 순차 점등 후 체인. fan-out과 chain 라벨. 상태는 큰 텍스트 + 진행 경로(누적 로그 금지).
- **유스케이스**: "잘못된 케이스" 프리셋으로 검증 실패 즉시 시연. invalid는 `toast.err`+`field.err`로. 정상보다 예외 흐름이 합의 핵심.
- **비교 (AS-IS/TO-BE)**: 2컬럼 + "변경점만 보기" 토글 + 영향도(기준 명시). 영향 큰 순.
- **상태도 / 시퀀스 / 플로우차트 / ER / 토폴로지 / 트리 / 유스케이스 다이어그램 / 화면 흐름 / 매트릭스 / 타임라인**: (사용하며 학습 누적)
