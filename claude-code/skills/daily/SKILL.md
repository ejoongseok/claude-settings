## 역할

입력은 하나, 출력은 여러 개: 사용자가 자유롭게 쓴 내용에서 업무 기록, 기술 성장, 인사이트, 팀 공유 메시지, 개인 고민을 각각 뽑아 정리.

톤: 동료 개발자 1:1, 솔직, 구체적.

## 외부 데이터 의존

| 데이터 | 경로 | 필수/선택 | 부재 시 동작 |
|--------|------|:------:|------------|
| 오늘의 할 일 | `.local.claude/daily-todos/{today}.md` | 선택 | "계획 vs 실제" 섹션 생략 |
| 지나가는 메모 | `.local.claude/memo/*.md` (최근 30일) | 선택 | 메모 기반 패턴 감지 생략 |
| 이전 회고 | `.local.claude/daily/*.md` (최근 30일/주간/분기) | 선택 | 심층 분석 생략, 첫날 데일리만 |
| 모듈 상세 | `.local.claude/modules/{name}.md` | 선택 | 도메인 인사이트 생략, 코드 직접 Read fallback |
| 비즈니스 규칙 | `.local.claude/biz-rules.md` | 선택 | 도메인 점검 생략 |
| 팀원 매핑 | `.local.claude/team.md` | 선택 | git author 그대로 표기 |

## 모드 판단

`/daily`는 기본적으로 **데일리 회고 + 심층 분석을 항상 같이** 수행한다.
축적된 데일리 파일이 있으면 심층 분석 포함, 없으면(첫날) 데일리만.
`$ARGUMENTS`가 있으면 사용자 추가 맥락으로 활용 (예: `/daily 오전에 {고객사} 미팅도 다녀왔어`).

특정 모드를 명시적으로 요청하면 해당 모드만 실행:

| 입력 | 모드 |
|------|------|
| `/daily` (기본) | **데일리 회고 + 심층 분석** |
| "주간 정리", "이번 주 요약" | **주간 종합** |
| "분기 목표", "분기 회고", "OKR" | **분기 회고** |
| "커리어", "성장", "역량 정리" | **성장 스냅샷** |

## 정보 수집

> **[1M 활용]** 아래 7개 스텝은 **단일 메시지에서 병렬 호출**로 처리:
> - Bash: `git log --since="today 00:00" --oneline --all`, `git branch --show-current`, `git status`, `git diff --stat`
> - Glob/Bash: `find .local.claude/ -name "*.md" -mtime -1`, `ls .local.claude/daily/ | tail -5`, `ls .local.claude/memo/ | tail -7`
> - Read: `.local.claude/daily-todos/{today}.md`, `.local.claude/memo/{today}.md`, 이전 데일리 최근 5개
>
> 순차 호출 대신 병렬로 git·산출물·계획·메모·이전 회고를 한 번에 수집.

### 1. 대화 맥락 (현재 세션)
- 이 세션에서 나눈 대화를 시간순 정리
- 어떤 작업을 했고, 어떤 결정을 내렸고, 어떤 문제를 만났는지
- 사용한 스킬 (`/prd`, `/srs`, `/todo` 등)

### 2. git 활동 (오늘/이번 주)
```bash
git log --since="today 00:00" --oneline --all
git branch --show-current
git status
git diff --stat
```

### 3. 생성/수정된 산출물
```bash
find .local.claude/ -name "*.md" -mtime -1 2>/dev/null
```

### 4. 오늘의 할 일 (daily-todos)
```bash
cat .local.claude/daily-todos/$(date +%Y-%m-%d).md 2>/dev/null
```
오늘의 `daily-todos` 파일이 있으면 Read — 계획 대비 실제 진행 비교에 활용.

### 5. 이전 회고 (연속성)
```bash
ls .local.claude/daily/ 2>/dev/null | tail -5
```
이전 회고의 "내일 할 일"이 오늘 실제로 진행되었는지 확인.

### 6. 오늘/이번 주 메모 (memo)
```bash
cat .local.claude/memo/$(date +%Y-%m-%d).md 2>/dev/null
ls .local.claude/memo/ 2>/dev/null | tail -7
```
`/memo`로 축적된 지나가는 메모. 심층분석에서 패턴 감지에 활용.

### 7. 사용자 추가 입력
`$ARGUMENTS`가 있으면 추가 맥락으로 활용.

## 민감도 분류

| [공개] 분류 | 기준 |
|------|------|
| [PUBLIC] 팀 공유 가능 | 기술 학습, 업무 진행, 코드 발견, 기술 개선 제안 |
| [PRIVATE] 개인용 | 조직/문화 불만, 특정 동료 평가, 연봉/이직, 커리어 고민, 번아웃/스트레스 |

모호하면 [PRIVATE]로 보수적 분류.

## 데일리 회고 모드

```markdown
## YYYY-MM-DD (X요일)

### 계획 vs 실제
daily-todos 파일이 있으면 아침 계획 대비 실제 진행을 비교.
- [OK] 계획대로 완료한 것
- [추가] 계획에 없었지만 추가된 것
- [미완료] 못 끝낸 것 + 이유
daily-todos가 없으면 이 섹션 생략.

### 오늘 한 일
업무 내용 간결 정리. 기술적 결정이 있었으면 결정 사항과 근거 함께 기록.

### 배운 것 / 깊어진 것
새로 배운 기술이나 도메인 지식.
이미 알던 것이 깊어진 경우도 포함.
4단계: 인지(안다) → 적용(쓸 수 있다) → 설계(구조를 잡을 수 있다) → 전파(가르칠 수 있다)

### 코칭 피드백

오늘의 작업에 대해 3가지 관점에서 솔루션, 인사이트, 개선 피드백을 제공.
해당 없는 영역은 생략. 억지로 채우지 않음.

#### 업무/기술 코칭
오늘 한 작업의 접근 방식에 대한 피드백.
- **[OK] 좋았던 점**: "~한 접근은 좋았다. 왜냐하면..."
- **[제안] 대안 제안**: 2개 이상의 접근법과 트레이드오프를 제시. 비기술적 대안(프로세스 변경, 역할 분담, 일정 조정 등)도 포함.
  예: "방법 A: ~하면 빠르지만 ~의 단점. 방법 B: ~하면 견고하지만 시간이 더 필요"
- **[WARN] 보완 포인트**: "~는 괜찮지만, ~를 추가하면 더 견고해진다"
- **[인사이트] 도메인 인사이트**: 프로젝트 도메인/고객 맥락에서의 관찰 (modules/*.md 참조 기반)

#### Claude Code 활용 코칭
Claude Code, 스킬, MCP, 자동화 활용에 대한 피드백.
- **[제안]** "이 작업은 `/스킬명`으로 하면 더 빠르고 기록도 남는다"
- **[제안]** "이런 반복 작업은 커스텀 스킬로 만들어두면 효과적"
- **[제안]** "GitHub MCP로 ~하면 수동 작업을 줄일 수 있다"
- **[WARN]** "이건 Claude에 맡기기보다 직접 하는 게 낫다. 왜냐하면..."
- **[OK]** "~한 활용법은 효과적이었다. 이 패턴을 다른 곳에도 적용해볼 만하다"

#### 프로세스 코칭
작업 순서, 시간 배분, 의사결정 프로세스에 대한 피드백.
- **[제안]** "이 순서보다 ~를 먼저 했으면 피드백을 빨리 받을 수 있었다"
- **[WARN]** "이건 지금 단계에서 굳이 안 해도 됐을 것. ~단계에서 하는 게 적절"
- **[OK]** "Outside-In으로 진행한 것은 좋은 판단이었다"
- **[제안]** "이 결정은 팀/기획에 먼저 확인하고 진행하는 게 리스크가 낮다"

### 막힌 것 / 미결
해결하지 못한 문제, 확인 필요한 것, 보류한 결정.
각 미결 항목에 대해 가능한 해결 방향을 복수안으로 제시:
- 방향 A: ... / 방향 B: ... (기술적·비기술적 모두 포함)

### 내일 / 다음 액션
오늘의 맥락에서 이어지는 작업 2~3개. 우선순위순.

### [PRIVATE] 개인 노트
민감 내용이 있을 때만. 없으면 생략.

### 스탠드업 한 줄 [PUBLIC] (팀 공유용)
30초 안에 말할 수 있는 1~2문장.

### 생성된 산출물
| 산출물 | 경로 | 상태 |

### git 활동 요약
| 항목 | 내용 |

---

### 심층 분석 (축적 데이터 기반)

축적된 데일리가 있을 때만. 없으면 생략.

#### 미해결 고민 (해결 방향 포함)
| 고민 | 처음 등장 | 횟수 | 해결 방향 제안 |

#### 반복 패턴
반복되는 문제/병목/감정 패턴과 근본 원인 접근 제안.

#### 도메인/비즈니스 인사이트
고객사 패턴, 모듈 간 연관, 제품 방향 피드백, 도메인 지식 맵.

#### Claude Code 활용 제안
더 효과적인 스킬 활용법, 새로운 자동화 아이디어.

#### 이번 주 집중 제안
위 분석을 바탕으로 가장 임팩트가 큰 것 2~3개.
```

## 주간 종합 모드

사용자가 "주간 정리", "이번 주 요약" 등을 요청하면 작동.
`.local.claude/daily/`에서 이번 주 데일리 파일들을 Read로 읽어 종합.

```markdown
## 이번 주 종합 (MM/DD ~ MM/DD)

### 핵심 성과 [PUBLIC] (3~5개)
구체적 행동과 결과 중심.

### 기술 깊이 변화
이번 주에 깊어지거나 새로 습득한 기술/도메인 지식.
4단계 척도: 인지 → 적용 → 설계 → 전파

### 코칭 피드백 종합
이번 주 데일리에서 나온 코칭 피드백을 영역별로 종합.
- **업무/기술**: 이번 주 반복된 패턴, 개선된 점, 아직 보완 필요한 점
- **Claude Code 활용**: 효과적이었던 활용법, 새로 시도해볼 만한 것
- **프로세스**: 이번 주 작업 흐름에서 개선할 점

### 다음 주 포커스 (2~3개)

### [PRIVATE] 이번 주 개인 회고
에너지 흐름, 만족/불만족, 커리어 관점. 없으면 생략.
```

## 분기 회고 모드

사용자가 "분기 회고", "OKR 회고" 등을 요청하면 작동.
`.local.claude/daily/`에서 해당 분기 파일들을 종합.

```markdown
## QN 2026 분기 회고 (MM/DD ~ MM/DD)

### 주요 성과 [PUBLIC]
실제로 완수한 업무, 기여한 프로젝트, 해결한 문제. 정량적 표현 포함.

### 기술 성장 맵
영역별 깊이 변화: "영역: 분기 초 → 현재 (근거)"

### 코칭 피드백 분기 종합 [PUBLIC]
분기 동안의 코칭 피드백 패턴 분석.
- 반복적으로 나온 개선 포인트와 실제 개선 여부
- Claude Code 활용 성숙도 변화
- 프로세스 효율 변화

### [PRIVATE] 분기 개인 회고
만족도, 성장 체감, 커리어 방향성. 다음 분기에 바꾸고 싶은 것.

### 다음 분기 목표 (2~3개)
"구체적 행동 + 측정 가능한 결과" 형태.
```

## 성장 스냅샷 모드

사용자가 "커리어 정리", "역량 정리", "성과 정리", "연봉 협상" 등을 요청하면 작동.
`.local.claude/daily/`의 전체 데일리 + 주간 + 분기 파일을 종합.

```markdown
## 성장 스냅샷 (YYYY-MM-DD 기준)

### 핵심 역량 현황
가장 강한 기술 영역 3개와 근거. 성장 중인 영역 2개와 다음 단계.

### 커리어 자산
시장 가치 있는 경험/기술 조합.
예: "{도메인} 전문성 + {아키텍처 패턴} + {특화 기술} 조합"

### [PRIVATE] 성과 포트폴리오 (연봉 협상/평가용)

축적된 데일리/주간/분기 회고에서 **정량적 성과**를 자동 추출하여 정리.

#### 주요 기여
| 기간 | 프로젝트/작업 | 기여 내용 | 임팩트 |
데일리의 "오늘 한 일"에서 주요 작업을 추출하고, 가능하면 정량화.
- 기능 구현 건수, 해결한 CS/버그 건수
- 고객사별 기여 (어떤 고객사의 어떤 요청을 해결했는지)
- 아키텍처/구조 개선 기여

#### 도메인 전문성 확장
입사 시점 대비 현재까지 축적한 프로젝트 도메인 지식.
- 어떤 모듈을 얼마나 깊이 이해하게 되었는지
- 고객사 업무 흐름에 대한 이해도
- 이 지식이 팀에 어떤 가치를 주는지

#### 기술적 기여
- 도입/개선한 기술, 도구, 프로세스 (예: Claude Code 스킬 파이프라인 구축)
- 코드 품질 기여 (리뷰, 리팩토링, 컨벤션 정립)
- 팀 생산성 향상에 기여한 것

#### 숫자로 보는 성과
git log 기반으로 자동 수집 가능한 정량 지표:
- 커밋 수, PR 수, 이슈 해결 수 (기간별)
- 참여한 모듈/영역 범위
- 생성한 산출물 (PRD, SRS, 분석 문서 등)

#### [PRIVATE] 협상 포인트 정리
위 성과를 바탕으로 연봉 협상 시 강조할 포인트 2~3개.
"내가 없었으면 어떻게 됐을까?"에 답할 수 있는 구체적 사례 중심.

### [PRIVATE] 커리어 방향성 메모
커리어 관련 고민/방향성 종합. 선택지와 장단점을 구조적으로.
```

## 심층 분석 (데일리에 항상 포함)

> **[1M 활용]** 서브 분할 없이 메인에서 직접 **다중 파일 동시 로드** 후 교차:
> - 로드: `.local.claude/daily/*.md` (최근 30일), `daily/*W*-주간.md` (최근 주간), `daily/*Q*-분기.md` (최근 분기), `.local.claude/memo/*.md` (최근 30일)
> - 대조: 미해결 고민의 반복 등장 탐지, 성장 정체 패턴(같은 수준 작업 반복) 감지, 고객사 요청 패턴 교차, 주간/분기 목표 대비 실제 진행 괴리

데일리 회고 생성 시 `.local.claude/daily/`의 **축적된 데일리 파일들을 Read**하여 개별 회고에서는 보이지 않는 패턴과 인사이트를 도출한다. 축적된 파일이 없으면(첫날) 이 섹션은 생략.

### 분석 관점

#### 1. 미해결 고민 추적
데일리의 "막힌 것/미결" 섹션과 "[PRIVATE] 개인 노트"에서 반복 등장하는 고민을 추적.
- 여러 날에 걸쳐 같은 고민이 나오는지?
- 해결된 고민과 아직 미해결인 고민 분리
- 미해결 고민에 대해 **축적된 맥락을 기반으로 해결 방향 제안**
  - 코드베이스 탐색으로 기술적 해결책 제시
  - 유사한 문제를 다른 모듈에서 어떻게 해결했는지 찾기
  - "이 고민은 ~에게 물어보면 풀릴 수 있다" 같은 행동 제안

#### 2. 반복 패턴 감지
- **같은 유형의 문제**: "특정 이슈가 N주째 반복된다" → 근본 원인 다르게 접근 제안
- **같은 병목**: "특정 단계의 대기/확인이 매주 반복된다" → 프로세스 개선 제안
- **성장 정체**: "같은 수준의 작업이 반복된다" → 도전 영역 제안
- **에너지 패턴**: "월요일에 항상 부담을 느낀다" → 워크로드 분배 제안

#### 3. 도메인/비즈니스 인사이트
축적된 회고에서 도메인·비즈니스 인사이트를 도출.
- **고객사 패턴**: "{고객사} 요청이 이번 달에 3건 — 공통 주제 식별 → 핵심 Pain Point 추론"
- **모듈 간 연관**: "{모듈A} 작업하면서 {모듈B}도 자주 건드린다 — 통합 필요성의 증거"
- **제품 방향 인사이트**: "커스터마이징 요청 패턴을 보면 ~기능을 표준화하면 여러 고객사에 적용 가능"
- **도메인 지식 축적**: "{도메인 영역 A} 관련 작업을 많이 해서 흐름의 전반부는 이해가 깊어졌다. 후반부 ({도메인 영역 B}) 는 아직 약하다"

#### 4. Claude Code 활용 성숙도
- 어떤 스킬을 자주 쓰고 어떤 걸 안 쓰는지
- 효과적이었던 활용 패턴과 비효율적이었던 패턴
- "이런 상황에서는 ~스킬을 쓰면 더 효과적이었을 것" 제안
- 새로운 스킬이나 자동화 아이디어

### 출력 구조

```markdown
## 심층 분석 — YYYY-MM-DD 기준 (N일간 데이터)

### 미해결 고민 (해결 방향 포함)
| 고민 | 처음 등장 | 횟수 | 현재 상태 | 해결 방향 제안 |

각 미해결 고민에 대해:
- 축적된 맥락 요약
- 코드베이스/도메인 지식 기반 해결 방향
- 구체적 다음 행동 (누구에게 물어볼지, 뭘 시도해볼지)

### 반복 패턴
| 패턴 | 빈도 | 영향 | 제안 |

### 도메인/비즈니스 인사이트
- 고객사 패턴에서 보이는 것
- 제품 방향에 피드백할 만한 관찰
- 도메인 지식 축적 현황과 약한 영역

### Claude Code 활용 분석
| 스킬 | 사용 빈도 | 효과 | 개선 제안 |

### 종합 제안
위 분석을 바탕으로 "이번 주/이번 달에 집중하면 가장 임팩트가 큰 것" 2~3개.
```

## 후속 질문 프로토콜

회고 정리 후, 더 깊이 파고들 가치가 있으면 후속 질문.

**질문하는 조건** (하나 이상 해당):
- **숨은 깊이**: "복잡했다", "어려웠다" 뒤에 구체적 맥락이 빠짐
- **결정 근거 누락**: 설계/기술 선택은 있는데 왜 그 방식인지 없음
- **감정과 사실의 모순**: "잘 끝냈다" + "찝찝하다" 같은 모순
- **반복 패턴**: 비슷한 문제/병목/감정이 여러 날에 걸쳐 반복

**질문하지 않는 조건**:
- 입력이 짧고 단순한 날
- 이미 충분히 구체적인 경우
- 같은 주에 이미 2회 이상 후속 질문한 경우 (피로 방지)

**형식**: 회고 하단에 분리하여 배치. 최대 2개. 각 질문에 "답하면 ~가 더 명확해집니다" 한 줄.

```
**더 깊이 파볼까요?**

1. [질문] → 답하면 [기대 효과]
```

사용자가 답하면 회고 업데이트. 무시하면 그대로 유지.

## 파일 저장

Frontmatter (CONTRACT §7-2 표준): `category: daily, retention: 14d, harvest_targets: [biz-rules.md, modules/*.md]`

### 저장 경로
| 모드 | 경로 |
|------|------|
| 데일리 | `.local.claude/daily/YYYY-MM-DD.md` |
| 주간 | `.local.claude/daily/YYYY-WNN-주간.md` (예: `2026-W14-주간.md`) |
| 분기 | `.local.claude/daily/YYYY-QN-분기.md` (예: `2026-Q2-분기.md`) |
| 성장 스냅샷 | `.local.claude/daily/YYYY-MM-DD-성장스냅샷.md` |

- 같은 날짜 데일리 파일이 이미 있으면 내용을 합쳐서 업데이트 (덮어쓰기 아님)
- 디렉터리 없으면 `mkdir -p`로 생성
- 저장 후 파일 경로 안내

## 제약조건

- **사실만 기록한다.** 추측이나 과장 없음.
- [PUBLIC]/[PRIVATE] 분류는 매 항목마다 적용.
- 스탠드업 한 줄은 [PUBLIC] 내용만으로 구성.
- 기술 깊이 추적에서 과대평가하지 않음. "직접 구현했다"고 말한 것만 수준 상향.
- 커리어 고민에 특정 방향을 밀지 않음. 선택지와 판단 기준을 구조적으로 제시.
- 데일리 분량은 A4 반 페이지, 주간은 A4 1페이지 이내.
- 대화가 여러 세션에 걸쳤을 수 있으므로 **git 활동과 산출물 파일로 보완**.
- 이전 회고의 "내일 할 일"이 실제 진행되었는지 확인하여 연속성 유지.
- 대화에서 나온 내용 중 **메모리에 저장할 만한 것**이 있으면 별도 안내.
- 코칭 피드백에서 개선 방향을 제시할 때는 가능하면 복수안으로 제시하고 트레이드오프를 명시 (CLAUDE.md '문제 해결 방안 제시 원칙' 참조).

### 이 스킬의 고유 실패 시나리오

**[환경·규모]** auto memory 가 500줄 이상
- 신호: 당일 대화/산출물 누적이 500줄 초과 → 단일 회고에서 심층 분석 시 컨텍스트 포화
- 대응: 심층 분석을 병렬 Agent 로 위임(예: "코칭 피드백 agent", "메모리 후보 추출 agent") + 최종 회고는 요약만 합성

**[사용자 개입 필요]** 민감도 분류 모호 (공개 [PUBLIC] vs 비공개 [PRIVATE])
- 신호: 기록 내용에 인사·고객·보안 관련 내용이 섞여 있어 자동 분류 신뢰도 낮음
- 대응: 해당 항목을 나열하고 AskUserQuestion 으로 [PUBLIC]/[PRIVATE] 확인 후 저장 위치 결정 (공유용 vs 개인 로컬)
