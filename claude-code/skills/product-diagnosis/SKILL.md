## 역할

**고객을 본다.** 축적된 고객 접점 데이터를 가로질러 패턴을 읽고, "고객 문제를 잘 풀고 있는가?"에 답한다.

역할:
1. 고객사별 Pain Point와 요청 패턴에서 **제품 방향 신호**를 포착
2. 커스터마이징으로 대응 중인 것 중 **제품화해야 할 것**을 식별
3. 핵심 사용자 여정 (제품의 메인 플로우) 에서 **고객이 겪는 병목**을 찾아 제품 개선 방향 도출
4. 경쟁 대비 **차별화 포인트와 취약점**을 파악
5. 위 모든 것을 종합하여 **RICE 기반 기능 우선순위**를 도출

톤: 제품 전략 보고서. 고객 언어, 근거는 데이터로.

## 핵심 원칙

### 기술 스킬과의 차이

| | 기술 | 제품 |
|--|-----|-----|
| 메인 데이터 | 코드, git, 인프라 | 고객 프로필, CS, 요청, 회의록, PRD |
| 핵심 질문 | "기술적으로 건강한가?" | "고객 문제를 잘 풀고 있는가?" |
| 메인 루프 | bash 측정 → 정량 분석 | **문서 횡단 읽기 → 패턴 감지** |
| 의사결정 | 기술 투자 우선순위 | 기능/제품 투자 우선순위 |

**bash로 Controller 수를 세는 건 /tech-diagnosis 가 이미 한다. 제품 관점은 세지 않고 읽는다.**

### 검증과 편향 방지

1. **"고객이 원한다"와 "고객에게 필요하다"를 구분.** 요청(How)이면의 근본 문제(What)를 파악. "A 기능 만들어주세요"는 솔루션이지 문제가 아닐 수 있다
2. **고객 편향 방지.** 대형 고객 1곳의 요청이 로드맵을 지배하지 않도록 Reach(영향 고객 수) 기반 평가
3. **복수 출처 교차.** 고객 판단은 customers/*.md + CS + 이슈 + 회의록 + auto memory 교차. 단일 출처면 `[단일 출처]` 태그
4. **경쟁사 정보 한계.** WebSearch 기반이므로 항상 `[WebSearch]` 태그. 마케팅 정보 ≠ 실제 기능
5. **코드 검증은 필요 시만.** "이 기능이 실제로 있는지" 확인할 때만 코드를 본다. 메인 활동이 아님

## 영역 1: 고객 신호 (제품 의사결정의 핵심)

> "고객이 보내는 신호를 읽는다"

### 1-1. 고객사별 Pain Point 맵

customers/*.md + CS 이력 + auto memory를 횡단하여:

| 고객사 | 핵심 Pain Point | 반복 요청 | 사용 모듈 | 미사용 모듈 | 이탈 신호 | 확장 기회 |
|--------|---------------|---------|---------|-----------|---------|---------|

각 고객사에 대해:
- **반복 요청**: 같은 고객이 2회+ 요청한 것 = 진짜 아픈 곳
- **미사용 모듈**: 제품에 있는데 이 고객이 안 쓰는 것 = 적합성 갭 or 온보딩 부족
- **이탈 신호**: CS 빈도 증가, 불만 톤 변화, 경쟁사 언급
- **확장 기회**: "이런 것도 되나요?" 류의 질문

### 1-2. 요청 패턴 분석 (횡단)

CS + 이슈 + PRD + 회의록을 **가로질러** 반복 패턴을 추출:

| 요청 | 등장 횟수 | 요청 고객사 | 현재 대응 | 요청 이면의 문제 | 제품화 가치 |
|------|:-------:|----------|---------|--------------|:---------:|

**핵심 판단 기준:**
- 같은 요청 × 2개+ 고객사 → **제품화 강력 후보**
- 같은 요청 × 1개 고객사 반복 → **커스터마이징 유지, 단 근본 문제 확인**
- 요청의 How vs What 분리: "엑셀 다운로드 버튼 추가해주세요" → What은 "데이터를 빠르게 공유하고 싶다"

### 1-3. 고객 여정 병목

biz-rules.md의 상태 전이를 고객 여정으로 읽어서, 어디서 막히는지:

| 여정 단계 | 상태 전이 | 병목 증상 | 근거 (CS/요청) | 제품 개선 방향 |
|----------|---------|---------|-------------|-------------|

예: "{단계A} → {단계B} 사이에 {승인 단계}가 오래 걸린다" → 승인 흐름 간소화
예: "{대기 상태} → {진행 상태} 전환이 수동" → 자동 전환 트리거

### 1-4. 커스터마이징 vs 제품화

| 기능 | 현재 방식 | 요청 고객 수 | 제품화 영향 | 구현 난이도 | 권고 |
|------|---------|:---------:|----------|:--------:|------|

- **제품화**: 2개+ 고객사 요청 + 구현 난이도 중 이하
- **커스터마이징 유지**: 1개 고객사 전용 + 다른 고객에 불필요
- **검토 필요**: 1개 고객사지만 시장 니즈가 있을 가능성

**→ 의사결정 포인트:** 가장 제품화 가치 높은 항목 1줄

## 영역 2: 제품 적합성

> "제품이 시장에 맞는가"

### 2-1. 비즈니스 흐름 단계별 제품 깊이

프로젝트의 비즈니스 흐름 (ONBOARDING.md 또는 bot/INDEX.md 의 단계 매핑) 기준으로, 각 단계에서 고객이 겪는 현실:

| 비즈니스 단계 | 프로젝트 모듈 | 고객 활용도 | 기능 공백 | 고객 피드백 |
|-----------|-------------|:--------:|---------|-----------|

"화면이 몇 개"가 아니라 "고객이 이 단계에서 제품을 신뢰하는가".
활용도 판단: 고객 프로필에서 해당 모듈 언급 빈도 + CS 빈도

### 2-2. 기능 공백

고객이 요청하는데 없는 것 (영역 1의 요청 패턴에서 도출):

| 공백 기능 | 요청 근거 | 요청 고객 수 | 경쟁사 보유 여부 | 우선순위 |
|----------|---------|:---------:|:------------:|:------:|

### 2-3. 차별화 포인트

프로젝트에 있고 경쟁사에 없는(또는 약한) 것:

| 차별화 기능 | 자사 수준 | 경쟁사 | 고객 인지도 |
|-----------|:----------:|--------|:--------:|

"고객이 이 차별화를 알고 있는가?"가 중요. 모르면 마케팅 문제.

### 2-4. 경쟁 포지셔닝

| 경쟁사 | 포지션 | 자사 대비 강점 | 자사 대비 약점 |
|--------|--------|----------------|----------------|

`[WebSearch]` 기반. 실제 기능 ≠ 마케팅 주장.

**→ 의사결정 포인트:** 경쟁에서 가장 취약한 영역 1줄

---

## 영역 3: 제품 로드맵

> "다음에 뭘 할 것인가"

### 3-1. 기능 우선순위 (RICE)

영역 1(고객 신호) + 영역 2(제품 갭)을 종합하여:

| 기능 | Reach | Impact | Confidence | Effort | RICE | 근거 |
|------|:-----:|:------:|:----------:|:------:|:----:|------|

- **Reach**: 영향받는 고객 수 (영역 1의 "요청 고객 수"에서 도출)
- **Impact**: 고객 가치 1~3 (Pain Point 심각도에서 도출)
- **Confidence**: 확신도 % (데이터 충분성에서 도출)
- **Effort**: 인월 범위 (/tech-diagnosis 보고서 참조 또는 `[추정]`)

### 3-2. 분기별 로드맵

```
[현재]                       [Q+1]                         [Q+2]
[제품화 후보 #1]         →   [기능 공백 #1]             →   [차별화 강화]
[고객 병목 해소]         →   [커스터마이징 제품화 #2]    →   ...
```

이전 보고서가 있으면 로드맵 진행 상황 추적.

### 3-3. Build vs Buy vs Partner

| 기능 | Build | Buy/통합 | Partner | 권고 | 근거 |
|------|-------|---------|---------|------|------|

**→ 의사결정 포인트:** 다음 분기에 반드시 해야 할 제품 투자 1줄

---

### Phase 3-B: 자기 검증 (Adversarial Review)

**Adversarial Review** — 핵심 권고/판단 Top 3 각각에 대해:
1. **근거 재점검**: 코드/측정 기반인가, 문서 기술인가? 단일 출처인가 교차 확인인가?
2. **전제 검증**: 권고가 유효하려면 어떤 전제가 필요한가? 그 전제가 코드/문서에서 확인되는가?
3. **반대 증거 탐색**: "왜 이미 하지 않았나?" / "현재가 충분하다면?" — 반박 1개 이상 탐색

반박 유효 시 본문 수정, 부분 반박 시 "단, {가능성}" 인라인 추가.

1. **고객 편향 점검**: 특정 고객사의 요청이 로드맵을 왜곡하고 있지 않은지. Reach 기반으로 재검증
2. **How vs What 점검**: 요청을 그대로 로드맵에 넣지 않았는지. 근본 문제로 변환했는지
3. **경쟁사 정보 한계**: WebSearch 기반 분석의 불확실성 명시
4. **데이터 부족 영역**: 판단 근거가 약한 곳에 `[데이터 부족]` 태그 누락 없는지
5. **/tech·/business-diagnosis 와의 정합성**: 기술 실현 가능성(/tech-diagnosis), 사업 우선순위(/business-diagnosis)와 모순 없는지

## 제약조건

- **메인 활동은 문서 읽기.** bash 측정은 규모 파악용 최소한만. Controller를 세지 않는다
- **고객 편향 방지.** Reach(영향 고객 수) 기반 평가. 대형 고객 1곳이 로드맵을 지배하면 안 됨
- **How vs What 분리.** 고객 요청을 그대로 로드맵에 넣지 않는다. 근본 문제로 변환
- **경쟁사 정보 한계.** WebSearch 기반 = `[WebSearch]` 태그 필수
- **하드코딩 금지.** 고객사, 기능, KPI 모두 데이터에서 도출
- **데이터 부족 시 솔직하게.** "모릅니다"가 "추측합니다"보다 낫다. `[데이터 부족]` 태그
- **자기 검증 필수.** Phase 3-B Adversarial Review 실행
- **다른 보고서와의 관계**: 제품 관점은 "고객에게 무엇을", 기술 관점은 "기술적으로 어떻게", 사업 관점은 "사업적으로 어떻게". 상호 참조 권장
