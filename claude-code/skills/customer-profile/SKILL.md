## 역할

각 고객사의 특성·설정 차이·Pain Point·CS 히스토리·커스터마이징 내역을 고객사별 파일로 관리.

이 정보는 다른 스킬들이 참조:
- `/cs` → 고객사별 빈출 문제, 설정 차이 확인
- `/prd` → 고객 요청의 배경과 맥락 파악
- `/poc` → 데모 대상 고객의 관심사/기대 수준 파악
- `/review` → 고객사별 프로파일 설정이 영향 주는지 확인
- `/daily` → 도메인/비즈니스 인사이트에서 고객 패턴 분석

## 외부 데이터 의존

| 데이터 | 경로 | 필수/선택 | 부재 시 동작 |
|--------|------|:------:|------------|
| 프로젝트 컨텍스트 | `CLAUDE.md` | 선택 | 일반 SW 가정으로 진행, `[프로젝트 규칙 미확인]` 태그 |
| 봇 인덱스 | `bot/INDEX.md` 또는 `.local.claude/ONBOARDING.md` | 선택 | 디렉터리 Glob 으로 fallback |
| 비즈니스 규칙 | `.local.claude/biz-rules.md` | 선택 | Tier 1(도메인 무관) 점검만 수행 |
| 모듈 상세 | `.local.claude/modules/{name}.md` | 선택 | 코드 Grep 직접 fallback |
| 환경 설정 파일 | `application-*_{customer}.yml` 등 | 선택 | 환경 분기 키 자동 추출 불가 — 사용자에게 질문 |
| CS/미팅 이력 | `.local.claude/cs/`, `.local.claude/meetings/` | 선택 | 자동 연결 생략, 수동 링크 권장 |

## 파일 구조

기본: 고객사당 하나 `.local.claude/customers/{고객사명}.md`.

**분량 임계 — 자동 분리** (대형 고객사):

| 임계 | 동작 |
|------|------|
| ≤300줄 | 단일 파일 유지 |
| 301~500줄 | "곧 분리 권장" 알림 + 사용자 결정 |
| >500줄 | **분리 강제** — 디렉터리 패턴으로 전환 `customers/{고객사명}/` |

**분리 시 디렉터리 구조**:
```
customers/{고객사명}/
├── README.md              # 핵심 (기본 정보, Pain Point, 현재 상태) ≤200줄
├── cs-history.md          # CS/장애 히스토리 (가장 빠르게 누적)
├── customization.md       # 커스터마이징 내역 상세
└── meetings.md            # 미팅/소통 히스토리
```

기존 단일 파일 → README.md 로 이전 + 섹션별 분리. 다른 스킬의 참조 경로 (`customers/{고객사명}.md`) 는 README.md 자동 fallback (Glob 으로 둘 다 매치).

## 동작

### 1. 고객사 파일 확인
```bash
ls .local.claude/customers/ 2>/dev/null
```

### 2. `$ARGUMENTS` 분석

입력 예시:
```
/customer-profile {고객사} — 특정 알림 기능 추가 요청함
/customer-profile {고객사} — 오늘 CS: 프로젝트 목록 안 보임, 격리·테넌트 이슈
/customer-profile {새 고객사} — 새 고객사, {업종}, {모듈} 도입 예정
/customer-profile {고객사} — {신규 사업부}도 도입 검토 중이라는 시그널
```

### 3. 파일이 없으면 → 새 프로필 생성

> **[Opus 4.7 / 1M 활용]** 다음을 **단일 메시지에서 병렬 호출**:
> - Glob: `**/application-*.yml`, `**/application-*.properties`, `**/config/*_{customer}.yml`
> - Grep (병렬): `{고객사키워드}`를 각 설정 파일 패턴에서 동시 검색
> - Read: `CLAUDE.md` 의 "고객사 분기" 섹션 + `.local.claude/biz-rules.md` 의 테넌트/격리 키 정의 섹션

```bash
# 프로젝트 환경 설정 파일에서 고객사 키워드 검색
grep -rl "{고객사키워드}" {프로젝트 환경 설정 디렉터리} 2>/dev/null
```

가능한 정보를 코드베이스에서 수집하여 초기 프로필 생성.
모르는 항목은 비워두고, 사용자가 점진적으로 채움.

### 4. 파일이 있으면 → 업데이트

입력 내용을 분석하여 해당 섹션에 추가:
- CS 관련 → "CS/장애 히스토리" 섹션
- 미팅 관련 → "미팅/소통 히스토리" 섹션
- 요청 관련 → "요청 사항 추적" 섹션
- 특성 관련 → "Pain Point & 관심사" 또는 "비즈니스 특성" 섹션
- 커스터마이징 → "커스터마이징 내역" 섹션
- 변경 이력에 날짜 + 내용 기록

### 5. 다른 스킬 결과 자동 연결

다른 스킬에서 고객사 관련 파일이 생성되면, 이 프로필에 링크를 추가하라고 안내:
- `/cs`로 CS 분석 → "이 결과를 `/customer-profile {고객사}`로 히스토리에 추가하세요"
- `/meeting-notes`로 고객 미팅 기록 → "이 회의록을 `/customer-profile {고객사}`로 연결하세요"

## 파일 저장

Frontmatter (CONTRACT §7-2 표준): `category: customers, retention: permanent`

### 저장 경로
- `.local.claude/customers/{고객사명}.md`
- 디렉터리 없으면 `mkdir -p`로 생성

## 제약조건

- **사실만 기록.** 고객에 대한 주관적 평가는 최소화. "특정 영역에 민감"은 OK, "까다로운 고객"은 X.
- 민감 정보(계약 금액, 내부 갈등 등)는 기록하지 않음.
- 변경 이력은 삭제하지 않음.
- 환경 분기 키 (Spring Profile 등) 가 확인되면 정확히 기록 — 다른 스킬에서 고객사별 분기 확인에 사용.

## 검증 시나리오

공통 3블록(빈 / 부분 / 풀 데이터)은 **CONTRACT §6-1** 참조.

### 이 스킬의 고유 실패 시나리오

**[의존성 부재]** 환경 분기 키 파일 부재
- 신호: 고객 식별에 필요한 설정 파일(tenant config, account ID 매핑 등) 없음
- 대응: 사용자에게 수동 입력 요청 — "고객사 이름/환경/주요 특성을 직접 알려주세요"

**[데이터 결함]** 기존 프로필 작성일 90일 이상 경과
- 신호: `customer/{name}.md` frontmatter `updated`가 오늘(2026-04-21) 기준 90일+ 경과
- 대응: 프로필 상단에 `[stale]` 태그 추가 + 갱신 권장 메시지 — "90일+ 경과, 계약·담당자·규모 변동 확인 권장"
