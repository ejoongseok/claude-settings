> **외부-가시 문서 공통 원칙 준수** — `rules/external-doc.md` (가독성 5 + 정직성 5). 이 스킬은 팀 외부(경영진·파트너·신규 합류자) 참조가 전제.

## 역할

아직 결정 전 단계에서 **2~3가지 이상의 선택지를 펼쳐놓고 이해관계자(팀장·PM·경영진·동료)와 논의하기 위한 결정 요청 문서** 생성.

**존재 이유**:

1. **"뭐가 좋을지 같이 좀 봐줘"가 필요한 순간.** 기술이든 비기술이든 몇 가지 길이 있고, 각각 장단점·소요·되돌림 가능성이 다를 때. 혼자 결정하지 않고 **대안을 펼쳐 논의를 시작**하는 문서.
2. **"정석 vs 덕테이프"를 동등하게 저울질.** "오래 걸리지만 정석인 A", "덕테이프지만 바로 가능한 B" 같은 접근 성격 대비를 **명시적으로** 드러내 의사결정 맥락을 선명하게. 덕테이프를 열등한 것으로 표기하지 않음 — 시급성·영향도에 따라 덕테이프가 정답일 수 있음.
3. **결정까지 한 파일에서 완결.** 대안 제시 → 피드백 리비전 누적 → 결정 기록이 같은 파일에 쌓인다. ADR로 별도 승격하지 않고, 이 문서 자체가 결정 배경 참조용으로 보존.

**톤**: 선언형 + 피드백 열기. "{A}안 추천드립니다. 다른 시각 있으면 알려주세요." 질문형 금지.

## 입력 처리

`$ARGUMENTS`를 판별:

- **기존 RFC 파일** (파일명이 `RFC-*.md`) → **업데이트 모드** 진입
- **PRD/SRS/brainstorm 등 복수 대안 포함 문서** → 대안 추출 후 RFC 구조로 변환
- **일반 파일 경로(.md)** → Read, 단일 대안만 있으면 축을 비튼 대립 안 추가 제시
- **디렉터리** → `ls`로 기존 RFC 여부 확인 + PRD/SRS/brainstorm 통합
- **자유 텍스트** → 프로젝트 매칭 + **스킬이 대안 능동 제시**
- **빈 인자** → 최근 프로젝트·최근 brainstorm 목록 제시 후 선택 요청

## 신규 vs 업데이트

이 스킬도 **한 번 만들고 끝나는 문서가 아니다**. 피드백이 돌아오면 리비전을 누적하고, 결정이 나면 "결정 기록" 섹션을 채워 문서를 닫는다.

### 업데이트 모드
- 기존 파일명 유지, 파일 덮어쓰기
- Frontmatter `updated` 갱신, `revision` +1
- 본문 상단 "이번 업데이트 요약" 블록 갱신 (직전 블록 제거 후 새 내용)
- "변경 이력" 표에 새 리비전 행 추가
- 피드백으로 새 안이 추가되거나 추천안이 바뀌면 본문에 반영

### 결정 닫기 (업데이트 모드의 특수 단계)
입력에 "A안으로 결정", "B로 가자" 같은 신호가 있으면:
- Frontmatter `status: decided` + `decided_at`·`decided_by` 기록
- "결정 기록" 섹션 채움: 선택된 안 / 결정 근거 / 결정자 / 논의 맥락
- "변경 이력"에 "결정: {안}으로 확정" 행 추가
- 파일 **삭제하지 않음** — 결정 배경 참조 문서로 보존

### 신규 생성 모드
- 새 파일명으로 저장
- `revision: 1`, `status: open`

## 프로세스

### Step 0: 멈추고 생각하기

> **[1M 활용]** 다음을 **단일 메시지에서 병렬 호출**:
> - Read: `$ARGUMENTS` 입력, `CLAUDE.md`, 관련 brainstorm `.local.claude/brainstorm/*.md`, PRD `.local.claude/prd/*.md`, SRS `.local.claude/srs/*.md`, 기존 ADR `.local.claude/adr/*.md`
> - Glob: `.local.claude/modules/*.md`, `.local.claude/customers/*.md`, `.local.claude/INFRASTRUCTURE.md`
> - Bash: `ls .local.claude/rfc/` 및 프로젝트 RFC (업데이트 모드 판별)

1. **정말 RFC가 맞나?**
   - 단일 방향이 확실 → `/briefing` 또는 `/srs`
   - 이미 결정 난 사안 → `/adr` 또는 그냥 결정 기록
   - "몇 가지 안이 있는데 같이 봐줘" → RFC
2. **결정 유형**: 기술 / 비기술 / 혼합. 비기술이어도 OK.
3. **수신자 계층**: 팀장·PM / 경영진 / 동료 (복수 가능). 동료 포함 가로 논의가 주 용도 중 하나.
4. **결정 시한**: 언제까지? 결정 지연 시 영향?
5. **대안 수**: 입력에 몇 개? 2개 미만이면 스킬이 능동 추가 제시 필요.

### Step 0.5: 입력 충분도 판정 & 구체화 질문

**RFC 본문을 채우는 데 필요한 구체 정보**가 충분한지 판정한다. 불충분하면 **작성 전에 사용자에게 질문**으로 먼저 해소.

**충분도 체크** (4개 중 **최소 3개** 파악 가능해야 1단계 진행):

1. 결정해야 할 질문이 한 문장으로 특정되는가? ("X를 어떻게 할 것인가")
2. 대안이 1개 이상 수집 가능한가? (입력에 있거나 스킬이 대립 안 생성 가능)
3. 배경/맥락이 파악 가능한가? (왜 이 결정이 필요한가, 왜 지금인가)
4. 의사결정 시한이 추정 가능한가? (D-N 또는 분기/월 단위)

**부족 시 구체화 질문 (최대 5개, 우선순위 순)** — 각 질문에 "왜 필요한지" 병기:

1. 결정해야 할 질문을 한 문장으로 다시 써주실 수 있나요? → "결정해야 할 질문" 섹션이 흐릿하면 대안 비교가 엉뚱한 축으로 감
2. 지금 떠오르는 대안 1~2개가 있는지? 없어도 OK (스킬이 축을 비틀어 대립 안 생성) → 대안 0개면 "정석 vs 덕테이프" 같은 축 선택도 불가
3. 이 결정이 **왜 지금** 필요한가? (외부 기한·팀 의존·사업 압박 중) → "배경" 섹션이 비면 수신자가 긴급도를 가늠 못함
4. 결정이 지연되면 구체적으로 뭐가 문제인가? → "의사결정 시한 — 지연 시 영향" 섹션 소재
5. 제약(예산·기술 스택·팀 리소스·고객 요구 등) 중 특히 중요한 것 1~2개는? → "전제 조건/제약" + 각 안의 "우리 상황에서의 평가" 소재

**질문 건너뛰는 조건** (바로 1단계로):

- 업데이트 모드이고 기존 RFC가 있으면 → 이번 리비전의 변경점만 확인
- 입력 문서(PRD/SRS/brainstorm)가 이미 결정 질문·대안·배경을 모두 담고 있으면 스킵
- 사용자가 "일단 초안만" 명시적 요청 시 초안 우선

### 1단계: 입력 분석 & 대안 추출

- 입력 Read, 대안 나열된 부분 수집
- 대안이 1개뿐이면 **축을 비튼 대립 안** 생성:
  - 정석 vs 덕테이프
  - 새로 만들기 vs 기존 재활용
  - 전체 교체 vs 점진적 마이그레이션
  - 기술 접근 vs 비기술(프로세스·교육·수기 처리)
  - 내부 개발 vs 외부 위탁/솔루션
  - 범위 확장 vs 범위 축소

### 2단계: 각 안 다듬기

각 안에 대해 아래를 채운다. 모르면 `[확인 필요]` 또는 `[추정: 근거]`.

- **한 줄 설명**
- **접근 성격**: 정석 / 덕테이프 / 점진적 / 완전 교체 / 프로세스·역할 변경 / 외부 위탁 / 기타
- **소요 (기대)**: 기대일 + 근거
- **소요 범위**: 최선 / 최악
- **주요 가정**: 2~3개. 가정이 깨지면 기간 흔들림
- **재산정 트리거**: 어떤 신호가 나타나면 일정 재계산 (예: "외부 API 스펙 변경 시", "신규 요구 추가 시")
- **2차 비용**: 선택 후 유지보수·운영·기술 부채 부담 — **덕테이프 안은 필수 기재**, 정석 안은 선택
- **확신도**: High / Medium / Low (+ 그 이유 한 줄)
- **장점**: 비즈니스 가치·개발 속도·유지보수성·팀 학습·리스크 감소 관점
- **단점**: 구체적으로
- **되돌림 가능성**: 쉬움 / 어려움 / 불가 + 한 줄 사유
- **성공 판단 기준**: 이 안을 선택했다고 보려면 무엇이 충족되어야?

### 3단계: "아예 다른 시각" C안 탐색 (선택)

기존 안들이 유사한 축에 있으면 **축이 다른 C안**을 고민:
- 기술 접근만 있으면 → 프로세스·교육·수기 처리 같은 비기술 경로
- 새로 만드는 안만 있으면 → 기존 모듈 재활용·외부 솔루션
- "다 한다" 안만 있으면 → 범위 축소 / Phase 분할 / 보류

**억지 C안은 만들지 말 것.** 축이 유의미하게 다를 때만 제시.

### 4단계: 프로젝트 상황 기반 평가

각 안을 **현재 프로젝트 상황**에서 평가 (아래 기준 중 해당되는 것 1~2개를 명시적으로 적용):

- **기술 스택 정합성** — `CLAUDE.md` 기술 스택 섹션 참조. 맞지 않으면 러닝 커브·도입 비용을 반영
- **팀 구성 정합성** — 인력 가용성·역량 분포. 모르면 생략
- **고객 영향** — `.local.claude/customers/*` 참조. 신규 기능 배포 영향·민감도
- **배포 인프라 정합성** — `.local.claude/INFRASTRUCTURE.md` 참조. 배포 창·다운타임 제약
- **비즈니스 가치** — 매출 기여 / 고객 체감 / 내부 효율
- **컨벤션 충돌 여부** — `CLAUDE.md` 금지 사항

각 안의 "우리 상황에서의 평가"를 **한 섹션에 명시적으로** 기록.

### 5단계: 추천안 선정

- 위 평가를 종합해 **1안 추천**
- 추천 이유 3~5줄. "비즈니스 가치 / 시급도 / 되돌림 가능성 / 팀 역량 적합" 같은 축 1~2개를 명시적 기준으로
- 추천이 유일 정답이 아님을 전제. 피드백 수용 열기

### 6단계: 선언형 톤 변환

- [FAIL] "A와 B 중 어느 쪽이 좋을까요?"
- [OK] "A안을 추천드립니다. 이유는 X. 다른 시각이나 반대 의견 있으면 알려주세요. 별다른 피드백이 없으면 A로 진행하겠습니다."

> **[메타인지]** 선언형 톤 변환 후 Adversarial review:
> 1. 근거 재점검 — 각 안의 소요·장단점·2차 비용·확신도에 근거가 있는가? 덕테이프 안의 2차 비용이 생략되지 않았는가?
> 2. 전제 검증 — 추천안이 유효하려면 어떤 전제(팀 역량, 기술 스택, 일정, 비용)가 성립해야 하는가? 전제가 깨지면 재고할 트리거는 명시되었는가?
> 3. 반대 증거 — "수신자가 '이 안이 왜 추천?'에 반박할 여지는?" 특히 **정석 편향**(덕테이프 저평가)·**현상 유지 편향**(기존 재활용 과평가) 반박 1개 이상 생성 후 반영

### 7단계: 결정 기록 (업데이트 모드, 결정 신호 감지 시)

결정이 났다는 신호가 입력에 있으면:
- frontmatter `status: decided` + `decided_at`·`decided_by`
- "결정 기록" 섹션 채움
- "변경 이력"에 결정 확정 행 추가
- 파일 보존 (결정 배경 참조용)

## 출력 구조

```markdown
# [주제] — RFC

> 작성: {작성자} | 최초 작성: YYYY-MM-DD | 최종 업데이트: YYYY-MM-DD | 리비전: v{N}

> 상태: {open / decided} | 결정 시한: YYYY-MM-DD (D-N)

> **이번 업데이트 요약 (v{N})** — 업데이트 모드에서만 표시, 다음 리비전 때 제거
> - {변경 핵심 1~3줄}

## 개요

**무엇을 결정해야 하는가** + **현재 추천안 한 줄**.

**톤 — 복수 수신자면 상위 계층 기준**:
- 경영진+팀장 → 경영진 톤 (업무 용어 중심)
- 팀장+동료 → 팀장 톤 (기술 원어/프레임워크명/아키텍처 패턴명 금지, 평문화)
- 동료 단독 → 원어 허용

## 배경 — 왜 이 결정이 필요한가
{비즈니스·기술·조직 맥락. 결정 지연 시 생기는 문제. TO-BE 먼저, AS-IS 다음.}
```

## 다이어그램

RFC 문서는 텍스트 비교가 중심. **시각화 최소주의** 원칙.

- **조건부 포함** (해당할 때만):

  | 조건 | 유형 | 배치 |
  |------|------|------|
  | 각 안의 업무 흐름 비교가 이해에 도움 | `flowchart` Before/After 2단 | 대안 비교 섹션 내부 |
  | 안별 일정·배포 순서가 크게 다름 | `gantt` | 추천 섹션 뒤 |
  | 상태 전이 개편이 본질인 결정 | `stateDiagram-v2` | 해당 안 내부 |

- **제외**: `sequenceDiagram`, `erDiagram` — 기술자용이라 결정 요청 문서에 부적합
- 노드 5개 이하

Frontmatter (CONTRACT §7-2 표준 + updated·revision·status·decided_at·decided_by·audience·decision_type·deadline·source_docs): `category: project-docs, retention: project-end, updated: YYYY-MM-DD, revision: N, status: open (decided 되면 변경), decided_at: YYYY-MM-DD (decided일 때만), decided_by: {역할/이름} (decided일 때만), audience: {경영진/팀장-PM/동료 중 택일 또는 복수}, decision_type: {technical / non-technical / mixed}, deadline: YYYY-MM-DD, source_docs: [{입력 파일 절대 경로 — 메타데이터용, 본문 노출 금지}]`

## 제약조건

- **외부-가시 문서 공통 원칙 준수** — `rules/external-doc.md` (가독성 5 + 정직성 5)
- **입력 부족 시 문서 작성 금지** — Step 0.5 충분도 체크 4항목 중 **2개 이상 파악 불가**면 구체화 질문으로 먼저 해소
- **대안 최소 2개** — 단일안이면 RFC가 아님. `/briefing` 또는 `/adr` 권장
- **내부 경로 링크 금지** (공유본 기준) — `.claude/`, `.local.claude/`, `CLAUDE.md`, `modules/`, `projects/` 같은 로컬 경로는 링크·경로 노출 모두 금지. 파일명만 허용 (`SRS-{기능명}.md` OK). frontmatter `source_docs`는 메타데이터라 절대 경로 유지 가능
- **선언형 + 피드백 열기** 톤 — "~맞을까요?", "~확인 부탁드립니다" 같은 질문형 금지. "~추천드립니다. 다르면 알려주세요" 사용
- **정석 vs 덕테이프 동등 저울질** — 덕테이프를 열등하게 표기 금지. 시급성·영향도에 따라 덕테이프가 정답
- **일정 점 추정 금지** — 각 안의 소요는 **기대 + 범위(최선/최악) + 가정 + 재산정 트리거** 필수
- **덕테이프 안의 2차 비용 생략 금지** — 빠른 구현의 그늘(유지보수·기술 부채)을 숨기지 않음
- **확신도(High/Medium/Low) 태그 필수** — 소요·추천안 모두. 태그에 근거 한 줄 병기
- **구현 세부(클래스명·SQL 등) 기본 섹션에 금지** — `<details>` 블록이나 생략
- **민감 정보 금지** — 매출·내부 협상·단가 등 외부 공개 부적합 내용 제외
- **결정 난 RFC는 삭제하지 말고 보존** — 결정 배경 참조 문서
