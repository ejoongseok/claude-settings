## 역할

함께 일하는 사람들(상사, 동료, 팀원)의 **업무 관련** 프로필을 관리한다.

이 정보는 다른 스킬들이 참조:
- `/team-review`: 작성자의 기술 프로필/성장 관심사에 맞춰 피드백 깊이 조절
- `/briefing`: 팀원별 담당 모듈/강점 참고
- `/draft`: 상대방의 커뮤니케이션 선호에 맞춰 메시지 톤 조절
- `/leadership`: 팀원/상사와의 관계에 맞는 조언 연결
- `/analyze-request`: 요청자의 역할/스타일 참고 (team.md 매핑 보완)

## 민감도 원칙

**적어도 되는 것 (업무 맥락):**
- 담당 모듈/고객사/역할
- 기술 강점/약점 영역 (FE/BE/DB/인프라)
- 커뮤니케이션/보고 선호 스타일
- 의사결정 스타일
- 코드 리뷰 시 중시하는 것
- 성장 관심사 (본인이 공유한 것만)
- 효과적이었던 협업 패턴
- 반복되는 코드 패턴 (리뷰에서 관찰한 것. 사실만, 판단 없이)

**절대 적지 않는 것:**
- 성격/태도 평가 ("까다롭다", "느리다", "실력이 부족하다")
- 개인사 (가족, 건강, 취미)
- 성과 등급/연봉/보상
- 갈등/불만 기록
- 이직/퇴사 관련
- 능력에 대한 일반화 ("항상 ~를 실수한다")

**경계선 (조심해서 적기):**
- **[OK]** "SQL 에서 특정 조건 누락 경향, 리뷰 시 체크" (관찰 사실 + 행동 가이드)
- **[FAIL]** "SQL을 못 짠다" (판단)
- **[OK]** "최근 동적 SQL 활용이 능숙해짐" (성장 관찰)
- **[FAIL]** "언어 X 실력이 부족하다" (부정적 일반화)

## 동작

### 1. 기존 프로필 확인
```bash
ls .local.claude/people/ 2>/dev/null
```

> **[1M 활용]** 다음을 **단일 메시지에서 병렬 호출**:
> - Read: `.local.claude/team.md`, `.local.claude/people/{닉네임}.md` (있을 시)
> - Glob: `.local.claude/people/*.md` 전체 목록 (관계 맵 파악)
> - Bash: 입력에 언급된 닉네임을 team.md 매핑과 대조

### 2. `$ARGUMENTS` 분석

입력 예시:
```
/people {팀장닉네임} - 간결한 보고를 선호, 판단 근거를 함께 달라고 함
/people {닉네임} - {특정 모듈} 쪽 강점, 최근 {기술} 학습 중
/people {경영진닉네임} - 데이터 기반 의사결정 선호, 요약 중심 보고
/people {디자인담당} - 기술 용어를 풀어서 설명해야 함
/people {닉네임} - 오늘 리뷰에서 동적 SQL 잘 활용함
```

인자 없이 `/people`만 실행하면 기존 프로필 목록 표시.

### 3. team.md 참조

새 프로필 생성 시 `.local.claude/team.md` 에서 기본 정보를 가져옴:
- 이름, 닉네임, 담당, 역할, 직군 카테고리, 대응 톤 가이드

### 4. 파일이 없으면 새 프로필 생성

team.md 에서 기본 정보를 채우고, 사용자 입력 내용을 해당 섹션에 배치.
모르는 항목은 비워두고 점진적으로 채움.

### 5. 파일이 있으면 업데이트

입력 내용을 분석하여 해당 섹션에 추가:
- 업무 스타일 관련: "업무 스타일" 섹션
- 기술 관련: "기술 프로필" 섹션
- 협업 관련: "협업 메모" 섹션
- 변경 이력에 날짜 + 내용 기록
- 마지막 업데이트 날짜 갱신

### 6. 다른 스킬 결과 자동 연결

다른 스킬에서 사람 관련 관찰이 나오면 이 프로필에 추가하라고 안내:
- `/team-review`로 리뷰 후: "리뷰에서 발견한 패턴을 `/people {닉네임}`으로 기록하세요"
- `/meeting-notes`에서 상대방 스타일 관찰 시: "이 관찰을 `/people {닉네임}`으로 기록하세요"

## 제약조건

- **사실과 관찰만 기록.** 판단/평가 금지. **[OK]** "특정 검증 조건 누락 경향", **[FAIL]** "실력이 부족"
- **본인이 공유한 것만** 성장 관심사에 기록. 추측 금지.
- 변경 이력은 삭제하지 않음 (축적).
- 민감도 원칙을 항상 준수. 위반 감지 시 경고하고 수정.
- team.md 와 중복되는 기본 정보 (닉네임-실명-부서-대응톤) 는 **team.md 가 정본 (source of truth)**. people 프로필에는 **team.md 에 없는 심층 정보** (업무 스타일, 기술 프로필, 협업 메모) 만 기록.

## 검증 시나리오

공통 3블록(빈 / 부분 / 풀 데이터)은 **CONTRACT 6-1절** 참조.

### 이 스킬의 고유 실패 시나리오

**[사용자 개입 필요]** 민감도 분류(공개/사적) 모호
- 신호: 기록할 내용이 성과 평가, 이직 고민, 건강 이슈 등 개인정보 경계선
- 대응: 저장 전 사용자 확인. "이 항목은 공개(팀 공유) / 사적(개인 노트)으로 저장할까요?"

**[데이터 결함]** `team.md`와 이름 불일치
- 신호: 요청에 언급된 이름이 `team.md`의 어느 항목과도 일치 않음(닉네임 vs 본명, 오타 가능성)
- 대응: 매핑 질문. "이 이름은 team.md의 X 또는 Y 중 누구인가요? / 신규 멤버라면 `/on-boarding` 권장"
