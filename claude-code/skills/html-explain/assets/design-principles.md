# 디자인·문서 원칙 (canon)

> 이 스킬 산출물의 품질 기준. 생성 전 훑고, 생성 후 §0 체크리스트로 자가 검토.
> 임시 감각이 아닌 확립된 canon에 근거한다. 누적 학습은 `playbook.md`(역할 분리: 여긴 고정 원칙, 거긴 케이스별 학습).

## 0. 자가 검토 체크리스트 (생성 전·후 필수)

산출물을 내보내기 전에 아래를 통과시킨다. 하나라도 어기면 고친다.

- [ ] **핵심 우선** — 맨 위에서 결론/요점이 잡히는가(역피라미드·BLUF). 스크롤 없이 "무엇을 알아야 하나"가 보이는가
- [ ] **위계** — 가장 중요한 것이 시각적으로 가장 먼저 눈에 드는가(크기·무게·색·여백). 모든 게 똑같이 강조되면 아무것도 강조 안 됨
- [ ] **스캔 가능** — 읽지 않고 훑어도 구조가 잡히는가(제목·표·라벨 불릿 > 산문). 3개 이상 속성은 표로
- [ ] **인지부하 최소** — 한 화면/섹션이 7±2를 넘기면 청킹·접기로 분할했는가
- [ ] **점진적 공개** — 지금 필요한 것만 보이고 상세는 깊이(접기)로 숨겼는가
- [ ] **affordance 정직** — 눌릴 것처럼 생긴 것은 실제로 눌리는가. 정적 요소는 버튼처럼 보이지 않는가(테두리·그림자 절제)
- [ ] **피드백** — 클릭·복사·토글 등 모든 동작에 즉각 반응(토스트·상태 변화)이 있는가
- [ ] **일관성** — 같은 의미는 같은 색·모양·용어로. 관습(익숙한 패턴)을 함부로 깨지 않았는가
- [ ] **접근성(POUR)** — 키보드로 조작·포커스 가능한가, 색만으로 정보 전달하지 않는가(색+텍스트 라벨), 본문 대비 4.5:1 이상인가, `prefers-reduced-motion` 존중하는가
- [ ] **단순성** — 뺄 수 있는 장식·중복·불필요 정보를 뺐는가("Less, but better")
- [ ] **정직** — 다크패턴 없음. 추정은 `추정`/`TBD`로 표시, 확정은 근거와 함께
- [ ] **Diátaxis 분리** — 이 산출물은 한 가지 유형에 집중하는가. 튜토리얼·레퍼런스·설명을 한 페이지에 섞지 않았는가
- [ ] **미적 완성도** — 정렬·여백·타이포 위계가 정돈됐는가(미학-사용성: 첫인상이 사용성 인식을 좌우)
- [ ] **대비·임팩트** — 주요 콘텐츠(표·불릿)는 near-black(`--ink`), 회색은 보조에만(전부 회색이면 흐릿·flat). 결론 헤딩은 임팩트 있는 크기(hero ≈28px). 폰트 미로드 시 시스템 폴백으로 완성도가 떨어지니 로드 확인
- [ ] **여백·밀도** — 섹션 간격 넉넉(≈52px) · 섹션 헤딩 ≥22px · 카드 패딩 ≥28px · 표 셀 패딩 ≥14px · 본문 콘텐츠 글자 ≥14.5px. 밀도가 높으면 여백·패딩으로 완화
- [ ] **장식 최소** — 스텝·노드·섹션 번호는 박스 배지 대신 inline 텍스트(겹침/박스 배지 금지, 화살표·여백으로 순서). pill·배지 텍스트는 `white-space:nowrap`(좁은 칸 세로 찌부 방지). 그림자·테두리 절제, 강조는 색만 짧게. hero eyebrow는 선택(헤드라인이 곧 결론이면 생략)
- [ ] **TOC ↔ 섹션 1:1** — 모든 TOC 항목은 본문에 같은 헤더(`sec-h`)가 존재(마지막 추천·결론 섹션도 `sec-h` 포함). 스크롤스파이는 위치 기반 + 클릭 lock(클릭한 항목 고정, 하단 폴백이 가로채지 않게)

## 1. 핵심 가치 (Core Values)

아래 8가지는 뒤따르는 모든 원칙이 결국 가리키는 곳. 세부를 잊어도 이 8개만 남으면 충분하다.

1. **사용자 중심 (User-centered)** — 시스템·작성자 편의가 아니라 사용자의 목표에서 출발한다.
2. **명료성 > 영리함 (Clarity over cleverness)** — 똑똑해 보이는 것보다 한눈에 이해되는 것이 낫다.
3. **일관성 (Consistency)** — 같은 것은 같게. 예측 가능함이 신뢰를 만든다.
4. **단순성 — 적을수록 좋다 (Simplicity, "Less, but better")** — 더할 게 없을 때가 아니라 뺄 게 없을 때 완성된다.
5. **접근성·포용 (Accessibility & inclusion)** — 가장 약한 조건의 사용자도 쓸 수 있어야 모두가 쓰기 쉽다.
6. **피드백·투명성 (Feedback & visibility)** — 지금 무슨 일이 일어나는지 늘 보이게 한다.
7. **관용·오류 예방 (Forgiveness & error prevention)** — 실수를 막고, 나더라도 쉽게 되돌리게 한다.
8. **정직 (Honesty)** — 사용자를 속이거나 몰아가지 않는다(다크패턴 배격).

### 2.1 Nielsen 사용성 휴리스틱 10

| # | 휴리스틱 (원어) | 본질 |
|---|----------------|------|

| 2 | 현실 세계와의 일치 (Match between system and the real world) | 시스템 용어가 아닌 사용자의 언어·관습을 따른다 |
| 3 | 사용자 통제와 자유 (User control and freedom) | 실수했을 때 빠져나갈 비상구(취소·되돌리기)를 둔다 |

| 6 | 회상보다 인식 (Recognition rather than recall) | 외우게 하지 않고 보면 알게 한다. 선택지를 노출한다 |
| 7 | 유연성과 효율 (Flexibility and efficiency of use) | 초보와 숙련자 모두를 위해 단축·맞춤을 허용한다 |
| 8 | 심미적·미니멀 디자인 (Aesthetic and minimalist design) | 불필요한 정보는 핵심과 경쟁한다. 덜어낸다 |
| 9 | 오류 복구 지원 (Help users recognize, diagnose, recover from errors) | 쉬운 말로 문제를 짚고 해결책을 제시한다 |
| 10 | 도움말과 문서 (Help and documentation) | 필요한 맥락에서 검색 가능한 과업 중심 도움말을 둔다 |

| # | 황금률 (원어) | 본질 |
|---|--------------|------|
| 1 | 일관성을 추구하라 (Strive for consistency) | 동일 상황엔 동일 동작·용어·배치 |
| 2 | 보편적 사용성을 추구하라 (Seek universal usability) | 초보~전문가의 차이를 수용(숙련자용 단축키 포함) |

### 2.3 Norman 인지·행동 모델

| 개념 (원어) | 본질 |
|------------|------|
| 어포던스 (Affordance) | 대상이 "무엇을 할 수 있는지"를 형태로 암시(버튼은 눌리게 생겼다) |
| 시그니파이어 (Signifier) | 어디서·어떻게 쓰는지 알려주는 단서(화살표, "여기 누르세요") |
| 매핑 (Mapping) | 조작과 결과의 자연스러운 대응(위 스위치 = 위 조명) |
| 피드백 (Feedback) | 행동의 결과를 즉시·명확히 알림 |
| 제약 (Constraints) | 가능한 행동을 물리·논리·문화적으로 제한해 오류를 줄임 |
| 개념 모델 (Conceptual model) | 사용자가 머릿속에 그리는 "이게 어떻게 동작하나" |

추가 — **실행의 간극 / 평가의 간극 (Gulf of Execution / Evaluation)**: "어떻게 하지?"와 "잘 된 건가?" 사이의 거리. 좋은 디자인은 이 두 간극을 좁힌다.

### 2.4 Dieter Rams 좋은 디자인 10계명

| # | 좋은 디자인은 (Good design is...) | 한 줄 |
|---|----------------------------------|-------|
| 1 | 혁신적이다 (innovative) | 기술 발전과 함께 새 가능성을 연다 |
| 2 | 제품을 유용하게 한다 (makes a product useful) | 미적·심리적 기능까지 포함해 쓸모를 높인다 |
| 3 | 아름답다 (aesthetic) | 매일 쓰는 물건의 미감은 삶의 질이다 |
| 4 | 이해를 돕는다 (makes a product understandable) | 구조가 스스로를 설명한다 |
| 5 | 드러나지 않는다 (unobtrusive) | 도구일 뿐, 장식·자기과시가 아니다 |
| 6 | 정직하다 (honest) | 실제보다 좋아 보이게 속이지 않는다 |
| 7 | 오래간다 (long-lasting) | 유행을 좇지 않아 낡지 않는다 |
| 8 | 마지막 디테일까지 철저하다 (thorough down to the last detail) | 우연에 맡기지 않는다 |
| 9 | 환경친화적이다 (environmentally-friendly) | 자원을 아끼고 오염을 줄인다 |
| 10 | 가능한 한 디자인을 적게 한다 (as little design as possible) | 적을수록 좋다 — 본질에 집중 |

### 2.6 Laws of UX (핵심 12)

| 법칙 (원어) | 본질 | 적용 |
|------------|------|------|
| Fitts의 법칙 (Fitts's Law) | 표적 도달 시간은 거리와 크기의 함수 | 자주 쓰는 버튼은 크고 가깝게 |
| Hick의 법칙 (Hick's Law) | 선택지가 많을수록 결정이 느려진다 | 옵션을 줄이고 단계로 나눈다 |
| Miller의 법칙 (Miller's Law) | 작업기억은 7±2개 | 정보를 청킹해 묶는다 |
| Jakob의 법칙 (Jakob's Law) | 사용자는 익숙한 방식을 기대한다 | 관습을 함부로 깨지 않는다 |
| Tesler의 법칙 (Tesler's Law) | 모든 시스템엔 줄일 수 없는 복잡성이 있다 | 그 복잡성을 사용자가 아닌 시스템이 떠안는다 |
| Doherty 임계 (Doherty Threshold) | 응답이 400ms 이내면 생산성·몰입이 급증 | 빠른 피드백, 낙관적 UI |
| 미학-사용성 효과 (Aesthetic-Usability Effect) | 예뻐 보이면 더 쓰기 쉽다고 느낀다 | 첫인상·시각 완성도에 투자 |
| 정점-종점 규칙 (Peak-End Rule) | 경험은 절정과 끝으로 기억된다 | 결정적 순간과 마무리를 설계 |
| 계열 위치 효과 (Serial Position Effect) | 목록의 처음·끝을 가장 잘 기억 | 핵심 항목을 양 끝에 배치 |
| 폰 레스토프 효과 (Von Restorff Effect) | 다른 하나가 가장 잘 기억된다 | 핵심 행동을 시각적으로 차별화 |
| 자이가르닉 효과 (Zeigarnik Effect) | 미완의 일을 더 오래 기억한다 | 진행률·체크리스트로 완료를 유도 |
| Postel의 법칙 (Postel's Law) | 받을 땐 너그럽게, 보낼 땐 엄격하게 | 입력은 관대하게 검증·정규화 |

### 2.7 시각 디자인 기본기 — C.R.A.P.

| 원리 (원어) | 본질 |
|------------|------|
| 대비 (Contrast) | 다르면 확실히 다르게 — 위계와 강조를 만든다 |
| 반복 (Repetition) | 시각 요소를 반복해 통일감을 준다 |
| 정렬 (Alignment) | 보이지 않는 선에 맞춰 질서를 만든다 |
| 근접 (Proximity) | 관련된 것은 모으고 무관한 것은 떨어뜨린다 |

곁들이는 셋 — 여백(whitespace)은 비어있는 게 아니라 위계를 만드는 도구다 / 타이포 위계(크기·무게·색)로 읽는 순서를 안내한다 / 색은 의미를 담되 수를 제한하고 대비를 확보한다.

### 2.8 실전 전술 — Refactoring UI & Krug

Refactoring UI (Adam Wathan & Steve Schoger):

- 처음엔 여백을 과하게 주고 줄여나간다.
- 위계는 글씨 크기만이 아니라 무게(weight)와 색(명도)으로 만든다.
- 회색조부터 정하고 색상 수를 제한한다.
- 그림자로 깊이(elevation)를 표현해 위계를 만든다.
- 텍스트는 검정·회색 대신 명도를 낮춘 색으로, 배경 대비를 확보한다.
- 빈 상태(empty state)와 경계 케이스부터 디자인한다.

Don't Make Me Think (Steve Krug):

- 제1법칙 "생각하게 만들지 마라" — 자명함이 최고의 사용성이다.
- 사람은 읽지 않고 훑는다(scan). 스캔 가능하게 구조화한다.
- 관습을 따르면 학습 비용이 0에 수렴한다.
- 명확함이 설득보다 우선한다.

### 2.9 접근성 — WCAG의 POUR

| 원칙 (원어) | 본질 |
|------------|------|
| 인식 가능 (Perceivable) | 모든 사용자가 정보를 지각할 수 있어야(대체텍스트·자막·대비) |
| 운용 가능 (Operable) | 키보드 등으로 조작 가능해야(시간 제한·포커스 관리) |
| 이해 가능 (Understandable) | 읽고 동작을 예측할 수 있어야(일관성·오류 안내) |
| 견고함 (Robust) | 보조기술·미래 도구와도 호환되게(표준 마크업) |

구체 기준 동반 — 색만으로 정보를 전달하지 않기, 본문 명도 대비 최소 4.5:1 등.

## 3. 문서화 원칙

### 3.1 Diátaxis — 문서의 4유형

한 문서가 네 역할을 섞으면 모두 약해진다. 사용자 상황에 따라 유형을 분리하는 것이 핵심.

| 유형 (원어) | 사용자 상황 | 두 축에서의 위치 |
|------------|------------|----------------|
| 튜토리얼 (Tutorial) | 처음 배우는 중 | 습득(Acquisition) × 실천(Action) |
| 사용법 안내 (How-to guide) | 특정 과업 수행 중 | 적용(Application) × 실천(Action) |
| 레퍼런스 (Reference) | 정보를 찾는 중 | 적용(Application) × 인지(Cognition) |
| 설명 (Explanation) | 이해를 넓히는 중 | 습득(Acquisition) × 인지(Cognition) |

두 축 — 습득↔적용(공부하는 중 vs 일하는 중), 실천↔인지(방법을 앎 vs 사실을 앎). 두 축이 영역 전체를 정의하므로 유형은 정확히 넷이다.

> 산출물은 한 가지 유형에 집중한다 — 설명·튜토리얼·레퍼런스를 한 화면에 섞지 말 것.

### 3.2 독자 우선 (Audience first)

누가 읽는가, 무엇을 이미 아는가, 무엇을 하려 하는가에서 출발한다. 작성자 머릿속 구조가 아니라 독자의 과업 흐름을 따라 배치한다.

### 3.3 명료한 글쓰기

Google·Microsoft 스타일 가이드가 공유하는 규범.

- 쉬운 말. 전문용어는 처음에 정의한다.
- 능동태·현재형으로 쓴다.
- 2인칭("사용자/당신")으로 직접 말한다.
- 한 문장에 한 생각. 짧게 끊는다.
- 같은 개념엔 같은 단어(용어 일관성).

### 3.4 구조·스캔 가능성

- 위계적 제목과 의미 있는 앵커를 둔다.
- 청킹: 긴 글을 작은 단위로 나눈다.
- 역피라미드: 결론·핵심을 먼저 둔다.
- 예시·코드·그림을 곁들인다("show, don't just tell").

### 3.5 단일 출처·DRY

같은 사실은 한 곳에만 둔다(SSoT). 중복은 곧 불일치가 된다. "Every Page is Page One"(Mark Baker) — 사람은 어느 페이지로든 진입하므로, 각 페이지가 스스로 맥락을 제공해야 한다.

### 3.6 미니멀리즘

필요한 것만 남긴다. 사용자는 매뉴얼을 끝까지 읽지 않고 바로 행동한다(능동적 사용자의 역설). 과업 중심으로 최소화한다.

### 3.7 Docs as Code

문서를 코드처럼 다룬다 — 버전관리(git), 리뷰(PR), 자동 빌드·배포, 코드와 같은 리듬으로 갱신. (Write the Docs 커뮤니티 규범)

### 3.8 살아있는 문서

문서는 자산이자 부채다. 틀린 문서는 없느니만 못하다. 코드 변경과 함께 갱신하고, 만료·오너십을 명시한다. 좋은 문서는 "미래의 나에게 보내는 편지"다.

## 4. 둘을 관통하는 5대 원리

UI/UX와 문서화는 결국 "사람의 인지를 돕는 일"이라 핵심이 겹친다.

1. **인지 부하 최소화 (Cognitive load)** — 화면이든 문서든, 머리 쓸 일을 줄인다.
2. **점진적 공개 (Progressive disclosure)** — 지금 필요한 것만 보이고 나머지는 깊이로 숨긴다.
3. **관습 따르기 (Conventions)** — 익숙한 패턴이 학습 비용을 없앤다(Jakob의 법칙 = 문서의 관례).
4. **위계 (Hierarchy)** — 무엇이 중요한지 시각·구조로 첫눈에 드러낸다.
5. **즉각 피드백 (Feedback)** — 행동·탐색의 결과를 바로 확인시킨다(UI의 상태표시 = 문서의 예시·결과).
