# html-explain 케이스 학습 로그

> kind별 "효과적이었던 것 / 수정과 회피 이력"을 누적한다. 생성 전 해당 섹션 read, 사용자 피드백 시 1줄 append(날짜 기입, 간결하게). 영구 보존, 큐레이션 유지. 노이즈 금지. 원칙(고정)은 `design-principles.md`, 여긴 학습(누적).
>
> 아래는 일반 시드. 사용하면서 자신의 케이스로 채워가세요.

## api

- 구성 순서 "한눈에(BLUF) 먼저, 이어서 흐름 다이어그램, 요청/응답, 에러코드"가 효과적. 인증과 처리 흐름은 산문보다 3노드 다이어그램이 훨씬 빨리 이해됨. 설정이나 환경 문제는 일반 코드 에러 표와 분리해 색상 콜아웃으로 올림. 소스 문서가 있으면 "원문과 쉬운설명" 토글이 해석 정확성 검증에 유효.

## rfp (제안과 의사결정)

- "결론 먼저(추천 + 이유 3줄 + 점수 카드), 이어서 비교 매트릭스, 기준 상세, 일정/예산, 리스크, 추천" 순서가 의사결정에 맞음. 매트릭스의 점수 막대(약점은 흐리게 dim, 추천 열 하이라이트)가 우열을 한 장에 전달. 예시나 가상 데이터는 상단 "데모/예시" 배지로 명시(실제 평가 오인 방지).

## summary (대화/문서 요약)

- 구성 "한눈에(BLUF), 한 일, 정한 것, 산출물, 변천(흐름도), 다음" 순서가 효과적.
- 불릿에 `<b>`나 코드칩 등 인라인 요소를 섞으면 `display:flex`가 각 요소를 별도 아이템으로 쪼개 줄바꿈과 정렬이 깨짐. 따라서 불릿 항목은 block + 절대위치 dot(hanging indent)로.

## external (외부 제공용)

- 해석 레이어(요약, 쉬운 설명)는 공유 전 정확성 검토 필수. 내부 경로, 상수, 담당자 주관 redact. 저맥락화.

## 기타 (공통)

- 장문 가독성을 위해 라이트 기본(다크는 장문에서 가독성 저하).
- 버튼처럼 보이는 정적 요소(테두리 알약 칩) 금지, 대신 평평 태그. affordance 정직(Norman 시그니파이어).
- hero 헤드라인 강조는 색만. 긴 구절 둘 이상에 색 + 하이라이트 밴드를 이중으로 주면 강조 비율이 커져 변별력 저하(Von Restorff). 강조는 짧고 적게.
