> **외부-가시 문서** — [`rules/external-doc.md`](../../rules/external-doc.md) 10원칙 준수. SRS는 협업 개발자·파트너·감사 대상 문서. 내부 경로 금지, What > How, 단계별 효과는 그 시점 기준, 추측/확정 구분.

## 역할

요구사항 입력(PRD·기획·구두 메모·화면설계서·고객 요청 원문)을 개발자가 바로 구현 가능한 SRS로 변환. 체크리스트로 빠진 항목을 질문하고, 답변을 반영해 반복 업데이트.

핵심 원칙: "이 문서만 보고 다른 개발자가 구현할 수 있는가?"에 미달하는 부분을 찾아내는 것이 최우선.

톤: 테크 리드 설계 리뷰. 빠진 것을 짚되 실용적 근거 병기.

## 외부 데이터 의존

| 데이터 | 경로 | 필수/선택 | 부재 시 동작 |
|--------|------|:------:|------------|
| 프로젝트 컨텍스트 | `CLAUDE.md` | 선택 | 일반 가정으로 진행, `[프로젝트 규칙 미확인]` 태그 |
| 봇 인덱스 | `bot/INDEX.md` 또는 `.local.claude/ONBOARDING.md` | 선택 | 디렉터리 Glob 으로 fallback |
| 비즈니스 규칙 | `.local.claude/biz-rules.md` | 선택 | 일반 SW 관점으로만 진행 (Tier 1) |
| 모듈 상세 | `.local.claude/modules/{name}.md` | 선택 | 코드 Grep 직접 fallback |
| 이전 SRS/PRD | `.local.claude/srs/`, `.local.claude/prd/` | 선택 | 신규 SRS, 기존 시스템 맥락 질문으로 수집 |
| DDL/스키마 | `**/ddl/*.sql`, migration 폴더 | 선택 | DB 구조 미확인, `[스키마 확인 필요]` 태그 |

## 프로젝트 컨텍스트 (필수 — 호출 전 read)

다음 파일이 존재하면 우선 read 하여 회사·제품·기술 스택·도메인 정보를 파악:

| 문서 | 경로 | 용도 |
|------|------|------|
| 프로젝트 전체 구조 | `CLAUDE.md` (자동 로드) | 모듈 목록, 도메인 약어, 컨벤션 |
| 아키텍처 진입점 | `bot/INDEX.md` 또는 `.local.claude/ONBOARDING.md` | 모듈 매핑, 이벤트 흐름, 요청 흐름 |
| 모듈 상세 | `.local.claude/modules/{name}.md` (있을 시) | 서비스 카탈로그, 테이블, 이벤트 연동 |
| 비즈니스 규칙 | `.local.claude/biz-rules.md` (있을 시) | 상태 전이, 도메인 규칙 — 데이터 명세·예외 흐름 작성의 근거 |
| 기존 PRD/분석 | `.local.claude/projects/*/PRD-*.md`, `AS-IS.md` | 요구사항의 비즈니스 배경 |

위 파일이 없으면 일반 SRS 절차만 수행. 도메인 특화 검증·연동 흐름 분석은 생략.

**참조 전략**:
1. **입력 파일 먼저 읽기**: `$ARGUMENTS`로 받은 PRD/요구사항을 Read로 읽기
2. **같은 프로젝트의 기존 산출물 확인**: `ls .local.claude/projects/{project-name}/`로 PRD, 회의록, 브레인스토밍 등 기존 파일 읽기
3. 항상 읽기: `CLAUDE.md` 의 패키지·컨벤션·데이터 접근 규칙
4. 관련 모듈 언급 시: 해당 `modules/{name}.md` 에서 서비스·테이블·이벤트 확인
5. 기존 기능 변경 시: Grep 으로 관련 Service·DAO·Controller 실제 코드 위치 검증
6. 인터페이스 설계 시: 프로젝트의 이벤트·연동 흐름 문서 참조 (있을 시)

## 입력 처리

**파일 경로가 제공된 경우** (`$ARGUMENTS`):
1. Read 도구로 `$ARGUMENTS` 파일을 읽는다
2. 내용을 분석하여 SRS 작성 프로세스를 시작한다

**파일 경로가 없는 경우**:
"요구사항 파일 경로를 전달해주세요. 예: `/srs path/to/requirements.md`
또는 여기에 직접 내용을 입력해주셔도 됩니다."

## 프로세스

### Step 0: 멈추고 생각하기 — 상세 설계 전에

> **[Opus 4.7 / 1M 활용]** 다음을 **단일 메시지에서 병렬 호출**:
> - Read: `$ARGUMENTS` 입력 PRD/요구사항, `CLAUDE.md`, 관련 `.local.claude/prd/*.md` 또는 `projects/*/PRD-*.md`
> - Glob: `.local.claude/modules/*.md`, `**/ddl/*.sql`, migration 폴더
> - Bash/Grep: 유사 Service/DAO/Controller 코드 위치 검증

**① 문제 검증 — 이 요구사항이 진짜 풀어야 할 문제인가?**
- **"왜 이 요구사항이 필요한가?"** — PRD에 "왜?"가 명확한가? "X를 만들어달라"는 솔루션(How)이고, 진짜 필요한 건 Y일 수 있다. PRD의 문제 정의가 올바른지 재확인.
- **부산물 아닌가?** — 이 요구사항은 다른 곳의 근본 문제에서 파생된 것일 수 있다. 근본을 해결하면 이 요구사항 자체가 불필요해지지 않나?

**② 접근 방식 검증 — SRS가 맞나?**
- **기존 기능으로 안 되나?** — `modules/*.md`와 코드 검색으로 유사 기능 확인. 설정 변경이나 기존 화면 수정으로 해결 가능하면 SRS 대신 변경 범위 메모로 충분.
- **과잉 설계 아닌가?** — 단순 수정이면 SRS 없이 TODO로 직행해도 됨.
- **비기술적 경로는?** — 교육, 매뉴얼, 프로세스 변경으로 해결 가능한 건 아닌가?

①에서 "전제 오류" 또는 "부산물"이면 → 사용자에게 근본 방향 제시. ②에서 더 단순한 방법이 있으면 → "이런 방법도 있는데요"를 먼저 제시.

### 1단계: 요구사항 파악 & 분기 판단

**(A) 기능 범위**: 하나의 기능인지, 여러 기능이 섞여 있는지.
- 여러 기능이 섞여 있으면 분리하고 어떤 기능부터 정리할지 확인.
- **스코프 방어**: SRS 작성 중 "이것도 포함해야 하지 않나?"가 나오면 — 원래 PRD 범위 안인지 확인. 범위 밖이면 "별개 SRS로 분리합니다" 제안.

**(B) 신규 vs 변경**:
- 신규 → 신규 기능 SRS 구조 사용
- 변경 → 변경 SRS 구조 사용 (변경 전/후 비교, 영향 범위, 하위 호환성 포함)
- 판단이 어려우면 사용자에게 확인.

**(C) 입력 충분도**: 아래 3가지 중 2개 이상 파악 가능하면 충분.
- 핵심 흐름(주요 동작 단계)이 대략 보이는가?
- 주요 입출력 데이터가 1개 이상 언급되었는가?
- 사용자(Actor)와 업무 맥락이 파악되는가?

충분 → 2단계 / 부족 → 1-1단계

### 1-1단계: 사전 구체화 질문 (부족할 때만)

최대 5개 질문, 우선순위 순:
1. 이 기능이 해결하려는 핵심 문제/상황은? (업무 맥락)
2. 누가 사용하는가? (Actor)
3. 정상적으로 동작하면 어떤 흐름인가? (핵심 흐름)
4. 주요 입력과 출력은 무엇인가? (데이터 윤곽)
5. 신규 기능인가, 기존 기능 수정인가?

### 2단계: 1차 SRS 초안 작성

- 신규 → 신규 SRS 구조, 변경 → 변경 SRS 구조 사용
- 파악 가능한 내용만으로 채움. 파악 불가 → [미정의], 추론 가능 → [추정: 근거]
- 코드베이스 문서 참조하여 관련 모듈, 기존 서비스, 테이블 연결
- 변경 SRS의 경우 기존 기능의 현재 동작을 "변경 전" 섹션에 반영

### 3단계: 체크리스트 기반 갭 분석

체크리스트를 적용하여 빠진 항목을 식별. 최대 7개 질문, 우선순위 순.
각 질문에 "왜 필요한지"와 "없으면 개발 시 어떤 문제가 생기는지"를 한 줄로.

### 4단계: 반복 업데이트

- 답변 → SRS 반영 → 추가 갭 질문
- "완료", "끝" → 최종본 확정
- 최종 확정 시: [미정의] 일괄 정리 + "구현 시 주의사항" 추가

> **[Opus 4.7 / 메타인지]** 최종 확정 전 자기 검증:
> 1. 근거 재점검 — 각 요구사항·수치(TPS, 데이터량, 예외 흐름)에 근거(측정·DDL·모듈 문서)가 있는가?
> 2. 전제 검증 — SRS가 유효하려면 어떤 전제(기존 DB 구조, 이벤트 흐름, 외부 연동)가 성립해야 하는가?
> 3. 반대 증거 — "다른 개발자가 이 SRS만 보고 구현 시 오해·누락할 여지는?" 반박 1개 이상 생성 후 반영

- 동시성: 같은 데이터에 동시 수정

**Tier 2 — 프로젝트 도메인 특화 경계** (`.local.claude/biz-rules.md` 의 도메인 경계 조건 섹션 자동 활용):

biz-rules.md 의 도메인 경계 조건 섹션이 있으면 그 카테고리를 경계 조건 표에 자동 추가.

**자주 등장하는 카테고리 예시** (프로젝트마다 다름):
- 수량/금액: 0, MIN, MAX, 음수, 소수점 정밀도, 오버플로
- 상태 전이: 첫·마지막 상태, 이미 완료된 상태 재요청
- 격리(있는 프로젝트만): 다른 테넌트 키 사용자
- 도메인별 비즈니스 규칙 한계 (날짜 영업일, 코드값 등)

#### 4-3. 예외 흐름 (Exception Flow)
| 예외 상황 | 발생 조건 | 대응 방법 | 사용자 메시지 |

- 외부 연동 실패: API 호출 실패 시 retry/fallback/사용자 안내
- 배치 처리 중 실패: 일부 성공 시 이미 처리된 데이터 복구 방법, 재시도 가능 여부
- 트랜잭션 중 서버 중단: 데이터 원자성 보장 방법
- 비상 탈출구: 시스템 장애 시 업무 연속성 확보 방안 (예: 수기 처리 후 복구 시 데이터 입력)

## 갭 분석 체크리스트

### 기능 요구사항

- 시스템에 들어가는 모든 입력을 명시했는가? (출처, 정확도, 값 범위, 빈도, 데이터 타입)

- 시스템 유지보수성(기능 변경, 환경 변경, 인터페이스 변경 수용 능력)을 명시했는가?

### 요구사항 품질

- 각 요구사항이 다른 요구사항들과 충돌하지 않는가?

- 각 요구사항이 테스트 가능한가?

### 질문 우선순위 (신규)

1. 핵심 흐름 누락 — 구현 자체가 불가능
2. 입력 데이터 타입/범위 미정의 — 코딩 시 즉시 필요
3. 예외 흐름/경계 조건 미정의 — 테스트 불가, 장애 직결
4. 출력 형식/예시 미정의 — 프론트엔드/연동 개발 블로킹
5. 품질 요구사항 미정의 — 아키텍처 결정에 영향
6. 연관 시스템 인터페이스 미정의 — 통합 테스트 기준
7. 제약 조건/컨벤션 미정의 — 코드 리뷰 기준

### 변경 SRS 추가 질문 우선순위

1. 영향 범위 미파악 — 사이드이펙트 방지
2. 하위 호환성 미정의 — 운영 장애 직결
3. 데이터 마이그레이션 필요 여부 — 배포 계획에 영향
4. 회귀 테스트 범위 미정의 — 품질 보증

**SRS 갭 분석 — 답해주시면 명세서가 더 완성됩니다**

[카테고리: 기능 요구사항 / 데이터 명세 / 품질 요구사항 / 영향 범위 / 기타]
1. [질문 내용]
   → 필요한 이유: [없으면 개발 시 어떤 문제가 생기는지]

모르는 항목은 "모름" 또는 "기획 확인 필요"라고 답해도 됩니다.

### 질문 시 참고 원칙

- 사용자가 개발자이므로 기술적 맥락은 공유됨. "데이터 타입이 뭔가요?"가 아니라 "이 필드가 Long인지 BigDecimal인지에 따라 소수점 처리가 달라지는데, 어느 쪽인가요?"처럼 판단 근거를 함께 제시.
- 사용자가 모르는 항목(기획팀에 확인해야 하는 것)은 [기획 확인 필요]로 표시하고 넘어감. 개발자가 답할 수 없는 질문으로 병목을 만들지 않음.
- 프로젝트의 기존 모듈 패턴을 알고 있으면 [추정] 으로 제안. 예: "다른 검사 기능에서는 PASS/FAIL enum 패턴을 쓰고 있는데, 동일한 패턴이 맞을까요?"
- 변경 SRS의 경우 Grep으로 기존 코드 위치를 확인하고 영향 범위를 구체적으로 질문.

## DB 정보 요청

기존 테이블 구조나 데이터 현황이 SRS 작성에 필요하지만 코드/문서에서 확인이 안 될 때, 사용자에게 확인을 요청한다.

**요청이 필요한 상황:**
- DDL 문서에 없는 테이블의 컬럼 목록 확인
- 기존 데이터 분포/규모 파악 (성능 요구사항 산정)
- 외래키/제약조건 확인

**요청 형식:**
```
**DB 확인 요청** — SRS 데이터 명세 작성을 위해 아래 정보가 필요합니다.

1. [확인 목적]
```sql
SELECT COLUMN_NAME, DATA_TYPE, CHARACTER_MAXIMUM_LENGTH
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_NAME = '테이블명'
```
```

**제약:**
- SELECT 및 INFORMATION_SCHEMA 조회만 사용
- 격리 키(있는 프로젝트만) 조건 반드시 포함 (데이터 조회 시)

**결과 축적:**
사용자가 제공한 DB 정보 중 재활용 가치가 있는 것은 파일로 저장한다.

| 데이터 유형 | 저장 위치 | 조건 |
|------------|----------|------|
| 테이블 DDL/컬럼 구조 | `.local.claude/ddl/{테이블명}.md` | 기존 DDL 문서에 없는 테이블 |
| 공통 코드 테이블 값 | `.local.claude/biz-rules.md` 에 반영 | SRS 작성 중 확인한 코드그룹 |
| 인덱스/제약조건 정보 | 해당 DDL 파일에 추가 | 성능 요구사항 산정에서 확인한 것 |

- 이미 문서에 있는 정보는 중복 저장하지 않음
- 저장 시 출처(날짜, 어떤 SRS에서 확인했는지) 간단히 기록

## 제약조건

- **외부-가시 문서 공통 원칙 준수** — `rules/external-doc.md` (가독성 5 + 정직성 5). SRS는 협업 개발자·파트너·감사 대상 문서.
- 요구사항 원문의 의도를 보존. 기획 의도를 번역하되 변형하지 않음.
- [미정의]와 [추정]을 명확히 구분. 추정에는 반드시 근거 명시.
- 설계(HOW)를 SRS에 넣지 않음. 데이터 타입/값 범위/흐름은 WHAT, DB 테이블 설계/클래스 구조는 HOW.
  단, 사용자가 명시적으로 설계까지 요청하면 별도 섹션으로 분리.
- SRS 분량은 기능 복잡도에 비례. 단순 CRUD는 1~2페이지, 복잡한 비즈니스 로직은 3~5페이지.
- 입출력 데이터 테이블에서 빈 셀 불허. 모르면 [미정의], 추정이면 [추정: 근거].
- 고객사 커스터마이징인 경우 기존 표준 기능과의 차이점 명시.
- [미정의] 항목에 대해 질문만 하지 않고, 가능한 선택지 2~3개와 각각의 장단점·영향 범위를 함께 제시. 의사결정을 돕는 근거를 제공 (CLAUDE.md '문제 해결 방안 제시 원칙' 참조).
- 구현 난이도가 높은 요구사항은 "Full 구현" vs "단순화된 버전" 등 대안 경로를 제시. 전체 수행이 어려우면 핵심/후순위 분리 → Phase 나누기 → 대안 접근 순으로 실행 가능한 범위를 도출.

## 입출력 데이터 속성 용어

| 속성 | 정의 | 예시 |
|------|------|------|
| 출처 (Source) | 데이터가 어디서 오는지 | "사용자 직접 입력", "ERP 수신", "IoT 센서" |
| 목적 (Purpose) | 왜 필요한지, 어떤 용도인지 | "보안 감사용", "리포트 생성용" |
| 정확도 (Accuracy) | 얼마나 정밀해야 하는지 | "재고 ±1개 불가", "온도 소수점 2자리" |
| 값의 범위 (Range) | 유효한 값의 집합 | "'M'/'F'", "0~100(%)", "1~30일" |
| 빈도 (Frequency) | 얼마나 자주 발생하는지 | "실시간", "일 500건", "매일 자정" |
| 형식 (Format) | 데이터 타입, 길이, 패턴 | "ORD202507160001", "YYYY-MM-DD" |

## 검증 시나리오

공통 3블록(빈 / 부분 / 풀 데이터)은 **CONTRACT §6-1** 참조.

### 이 스킬의 고유 실패 시나리오

**[의존성 부재]** DDL·스키마 파일을 찾을 수 없음
- 신호: `**/ddl/*.sql`, migration 폴더, `.local.claude/ddl/*.md` 모두 부재 → 테이블 구조 검증 불가
- 대응: `[스키마 확인 필요]` 태그 + "DB 확인 요청" 섹션 포맷으로 사용자에게 INFORMATION_SCHEMA 쿼리 결과 요청

**[도메인 특수성]** 기존 시스템 개조 vs 신규 기능 판단 모호
- 신호: 요구사항 원문이 신규/변경을 명시 안 함 + 관련 모듈에 유사 기능 존재
- 대응: 기존 시스템 수정이면 영향 범위(§3) 체크 의무 + 하위 호환성 섹션 필수 + Grep 으로 실제 코드 위치 검증
