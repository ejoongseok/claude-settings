## 1. 재질문 예시 모음

### 케이스 A: 목적은 있으나 형태가 모호한 경우

**원본:** "이 코드 설명해줘"

**재질문:**
> 현재 파악한 의도: 코드의 동작 원리를 이해하기 위한 설명 요청인 것 같습니다. 최적화 전에 아래를 확인할게요.
>
> 1. 설명의 깊이는 어느 정도인가요? (예: 전체 흐름 요약 / 라인별 상세 설명 / 핵심 로직만)
> 2. 설명 대상의 수준은? (예: 주니어 개발자 / 비개발자 / 시니어 동료)
> 3. 특별히 궁금한 부분이 있나요? (예: 특정 함수 / 디자인 패턴 / 성능 이슈)

---

### 케이스 B: 의도가 여러 방향으로 해석되는 경우

**원본:** "이력서 좀 봐줘"

**재질문:**
> 현재 파악한 의도가 여러 가지로 해석됩니다.
> - 해석 1: 이력서의 문법/표현을 교정해주는 것
> - 해석 2: 이력서의 내용/구성을 전략적으로 개선하는 것
> - 해석 3: 특정 포지션에 맞게 이력서를 커스터마이징하는 것
>
> 어느 쪽인가요? 또는 여러 개 해당되나요?

---

### 케이스 C: 추상적 키워드만 있는 경우

**원본:** "좋은 발표 자료 만드는 법 알려줘"

**재질문:**
> 현재 파악한 의도: 발표 자료 제작에 대한 가이드를 원하는 것 같습니다.
>
> 1. 어떤 종류의 발표인가요? (예: 사내 프로젝트 보고 / 컨퍼런스 발표 / 투자 피칭)
> 2. 어떤 부분이 가장 어려운가요? (예: 스토리라인 구성 / 슬라이드 디자인 / 데이터 시각화)
> 3. 결과물 형태는? (예: 체크리스트 / 단계별 가이드 / 실제 슬라이드 구성안)

---

### 케이스 D: 재질문 없이 바로 최적화한 경우

**원본:** "Spring Boot에서 JWT 인증 구현하는 코드 짜줘. Security 6 기준으로."

**바로 최적화 진행 이유:** 기술 스택(Spring Boot, Security 6), 행동(JWT 인증 구현), 형태(코드)가 모두 명시됨. 부족한 부분(응답 구조, 상세 요구사항)은 합리적 추론으로 보완 가능.

## 2. 최적화 예시 모음

### 예시 1: 경량 모드 - 에러 해결

**원본:** "이 에러 메시지 해결해줘"

**핵심 개선:** 맥락(C)과 행동(A) 보강, 결과물 형태를 상단에 배치

**최적화 결과:**

```
결과물: 원인 → 수정 코드 → 재발 방지 팁 순서로 정리된 짧은 해결 가이드.

당신은 [Java/Spring] 시니어 개발자입니다.
아래 에러 메시지의 원인을 분석하고 해결책을 제시하세요.

[에러 메시지 붙여넣기]

응답 구성:
1. 원인 요약 (1~2문장)
2. 해결 방법 (수정 코드 포함)
3. 재발 방지 팁 (1개, 한 줄)
```

**💡 사용 팁:**
- [Java/Spring]을 실제 기술 스택으로 교체하세요.
- 에러 메시지와 함께 관련 코드 스니펫을 붙여넣으면 정확도가 올라갑니다.
- 결과가 너무 장황하면 이제는 "간결하게" 한 줄 추가로 충분합니다. Opus 4.7은 분량 지시를 문자 그대로 따릅니다.

### 예시 2: 표준 모드 - 회의록 정리 (부족한 항목이 많은 경우)

**원본:** "회의록 정리해줘"

**🎯 의도 파악:** 회의 내용을 정리된 회의록으로 변환하는 요청.

**🔍 분석:** 맥락, 역할, 형식, 기준이 모두 빠져 있고, 행동도 모호합니다. 전면 보강이 필요합니다.

**최적화 결과:**

```
결과물: 아래 5개 섹션으로 구성된 회의록. 원문 분량의 1/3 이하.
섹션: 회의 개요 / 주요 논의사항 / 결정사항 / 액션 아이템(표) / 다음 단계

당신은 프로젝트 매니저 경험이 있는 회의록 전문가입니다.
아래 회의 내용을 위 구조로 정리하세요.

[회의 녹취/메모 붙여넣기]

---

작성 원칙:
- 발언자의 의견은 객관적으로 요약하되, 발언자 이름을 병기합니다.
- 불확실한 내용은 "[확인 필요]"로 표시합니다.
- 액션 아이템 표는 담당자 / 내용 / 기한 3열로 구성합니다.
```

**💡 사용 팁:**
- 녹취 전사본이 길면 Claude가 더 정확하게 정리합니다. 메모보다 전사본을 권장합니다.
- 특정 안건만 집중하고 싶으면 "특히 [안건명]에 대해 상세히 정리하세요"를 추가하세요.
- 4.7은 "원문 분량의 1/3 이하" 같은 수치 기준을 정확히 따릅니다. 분량을 조절하려면 이 숫자를 바꾸는 것이 가장 빠릅니다.

### 예시 3: 표준 모드 - 기술 면접 준비 (Few-shot 예시 포함)

**원본:** "기술 면접 준비 도와줘"

**🎯 의도 파악:** 기술 면접 대비를 위한 질문+답변 가이드 요청.

**🔍 분석:** 기술 분야, 난이도, 출력 형태가 모두 빠져 있지만, "면접 준비"라는 목적은 명확합니다. 역할(R)과 형식(F)을 중심으로 보강하고, Opus 4.7이 규칙보다 예시에 잘 반응하는 점을 활용해 예시를 하나 포함합니다.

**최적화 결과:**

```
결과물: [기술 주제]에 대한 면접 질문 3개(기초/심화/실무) + 각 모범 답변 포인트 + 꼬리 질문 1개.

당신은 10년차 백엔드 개발자이자 기술 면접관 경험자입니다.
[기술 주제]에 대해 기술 면접 모의 연습을 진행하세요.

---

형식 (예시를 따라주세요):

주제: Java 동시성

질문 1 (기초, 1분): synchronized 키워드의 역할과 한계를 설명하세요.
- 합격 포인트: 상호 배제 보장, 모니터 락 기반, 메서드/블록 단위 적용
- 우수 포인트: ReentrantLock과의 차이, 데드락 가능성, volatile과의 관계
- 꼬리 질문: "그렇다면 ConcurrentHashMap은 내부적으로 어떻게 동시성을 처리하나요?"

---

위 형식대로 [기술 주제]에 대한 질문 3개를 기초 → 심화 → 실무 적용 순서로 작성하세요.
답변 포인트는 "합격 포인트(이것만 말해도 합격)" / "우수 포인트(이것까지 말하면 우수)" 두 단계로 구분합니다.
각 질문마다 예상 답변 시간을 표기합니다.
```

**💡 사용 팁:**
- [기술 주제]에 구체적인 키워드를 넣을수록 좋습니다 (예: "Java 동시성" > "Java").
- 지원하는 회사의 기술 스택을 알고 있다면 "회사에서 Spring WebFlux를 사용합니다"를 추가하면 맞춤 질문이 나옵니다.
- 4.7은 예시의 포맷을 매우 정확히 복제합니다. 위 예시 구조를 바꾸면 출력 구조도 따라 바뀝니다.

### 예시 4: 경량 모드 - 이미 괜찮은 프롬프트

**원본:** "당신은 시니어 Java 개발자입니다. 아래 Spring Boot 코드를 리뷰하고 버그, 성능, 가독성 관점에서 개선 사항을 제시하세요. 각 항목에 수정 코드를 포함하세요."

**🎯 의도 파악:** 코드 리뷰 요청. 역할, 행동, 관점, 형식이 이미 잘 지정되어 있습니다.

**핵심 개선점:** 거의 충분하지만, 우선순위 기준과 출력 구조를 명시하면 더 일관된 결과를 얻을 수 있습니다.

**최적화 결과:**

```
결과물: 심각도가 높은 순서로 정렬된 코드 리뷰. 각 항목마다 위치 → 문제 → 수정 코드.

당신은 시니어 Java 개발자입니다.
아래 Spring Boot 코드를 리뷰하고 개선 사항을 제시하세요.

[코드 붙여넣기]

---

리뷰 관점 (우선순위순):
1. 버그 또는 잠재적 오류
2. 성능 개선 여지
3. 가독성 및 유지보수성

각 항목 포맷: 위치(라인 또는 메서드) → 문제 설명(1~2문장) → 수정 코드
```

**💡 사용 팁:** 원본도 이미 잘 작성되어 있습니다. 추가한 것은 결과물 미리보기(상단), 우선순위 명시, 출력 포맷 통일뿐입니다.

### 예시 5: 4.6 스캐폴딩 제거가 핵심인 경우 (Opus 4.7 특화)

**원본:**
```
당신은 시스템 설계 전문가입니다.
단계별로 신중하게 생각하세요.
먼저 계획을 세우고 실행하세요.
답변 전에 반드시 검토하세요.

[요구사항]에 맞는 시스템 아키텍처를 설계해주세요.
```

**🎯 의도 파악:** 시스템 아키텍처 설계 요청. 사고 과정을 강조하는 스캐폴딩이 다수 포함되어 있습니다.

**🔍 분석:** "단계별로 생각", "계획 후 실행", "답변 전 검토"는 모두 Opus 4.7이 네이티브로 수행하는 동작입니다. 4.6에서는 품질을 높이는 장치였지만, 4.7에서는 토큰만 소모하고 때로는 응답 구조를 어색하게 만듭니다. 제거하고, 검증이 필요하다면 구체적 체크리스트로 대체합니다.

**최적화 결과:**

```
결과물: 아래 5개 섹션으로 구성된 아키텍처 설계안.
섹션: 요구사항 해석 / 컴포넌트 구성도 / 주요 기술 선택과 근거 / 예상 병목과 대응 / 대안 1개

당신은 시스템 설계 전문가입니다.
아래 요구사항에 맞는 시스템 아키텍처를 설계하세요.

[요구사항 붙여넣기]

---

검증 체크리스트 (답변 마지막에 자가 점검):
- 모든 기능 요구사항이 컴포넌트 구성도에 반영되었는가
- 비기능 요구사항(성능, 확장성, 보안)이 명시적으로 다루어졌는가
- 기술 선택의 근거가 대안 비교를 포함하는가
```

**💡 사용 팁:**
- 스캐폴딩 4줄을 제거하고 자가 검증을 "구체적 3개 체크리스트"로 대체했습니다. 이쪽이 4.7에서 더 정확히 작동합니다.
- 검증이 여전히 약하다고 느끼면 체크리스트 항목을 추가하는 방향으로 조정하세요. "신중하게"를 다시 추가하는 것보다 효과적입니다.

## 3. 프롬프팅 레퍼런스

Instructions의 writing_principles에서 참조하는 상세 레퍼런스입니다.
기본 원칙은 Instructions에 있으므로, 여기서는 구체적 변환 예시와 테크닉만 다룹니다.

### 3-1. 부정문 → 긍정문 변환 사전

Opus 4.7은 부정문을 문자 그대로 따릅니다. 3개 이상 누적되면 의도 전달이 오히려 흐려지므로, 긍정문 또는 예시로 변환합니다.

| 원본 (부정문) | 변환 (긍정문) |
|---|---|
| 추측하지 마세요 | 확실한 정보만 사용하세요 |
| 장황하게 쓰지 마세요 | 핵심만 간결하게 작성하세요 (또는 "3문장 이내") |
| 코드를 생략하지 마세요 | 전체 코드를 포함하세요 |
| 전문 용어를 남발하지 마세요 | 비전공자가 이해할 수 있는 용어를 사용하세요 |
| 개인 의견을 넣지 마세요 | 객관적 사실만 기술하세요 |
| 번역투를 쓰지 마세요 | 자연스러운 한국어로 작성하세요 |

### 3-2. 모호한 표현 → 구체적 기준 변환 사전

Opus 4.7은 빈칸을 추측으로 채우지 않습니다. 모호한 형용사는 측정 가능한 기준으로 바꿔야 기대한 결과가 나옵니다.

| 원본 (모호) | 변환 (구체적) |
|---|---|
| 자세하게 설명해줘 | 각 단계를 코드 예시와 함께 설명하세요 |
| 짧게 요약해줘 | 3문장 이내로 요약하세요 |
| 전문적으로 써줘 | [분야] 실무자가 바로 사용할 수 있는 수준으로 작성하세요 |
| 쉽게 설명해줘 | 프로그래밍 경험이 없는 사람도 이해할 수 있도록 비유를 사용하세요 |
| 좋은 코드로 짜줘 | SOLID 원칙을 따르고, 메서드당 15줄 이내로 작성하세요 |
| 최신 방식으로 해줘 | [프레임워크] [버전] 기준의 권장 패턴을 사용하세요 |
| 친절한 톤으로 | 격려 문장 1~2개를 포함하고, 명령형 대신 제안형 어미("~하시면 좋습니다")를 사용하세요 |

### 3-3. 스캐폴딩 제거 사전 (Opus 4.7 특화)

Opus 4.7은 사고 과정(thinking), 자가 검증, 계획 수립을 네이티브로 수행합니다. 4.6 시절의 스캐폴딩은 대부분 제거하는 것이 낫습니다.

| 제거 대상 (4.7에서 불필요) | 이유 | 필요 시 대체안 |
|---|---|---|
| "단계별로 생각해" / "신중하게 추론해" | Adaptive thinking이 네이티브로 동작 | 과제가 복잡하면 문제 분해 기준을 직접 제시 (예: "먼저 요구사항 해석, 그 다음 아키텍처, 마지막으로 위험 분석 순서") |
| "답변 전에 검토해" / "꼼꼼히 확인해" | Self-verification이 네이티브로 동작 | 구체적 체크리스트로 대체 (예: "답변 후 다음을 확인: (1) 숫자 정확성, (2) 누락된 요구사항") |
| "계획을 먼저 세우고 실행해" | 복잡 과제에서 자동 수행 | 계획이 별도 산출물로 필요한 경우에만 명시 ("먼저 설계 개요를 제시한 후 상세 구현으로 진행") |
| "Think carefully" / "Take your time" | 효과 미미, 토큰만 소모 | 제거 |
| "중간 진행 상황을 알려줘" | 긴 응답에서 네이티브로 구분 | 제거 (에이전트 용도에서만 선택적) |

대체의 원칙: **모호한 지시를 제거하고, 진짜로 필요하면 구체적 체크리스트/기준으로 바꿉니다.**

### 3-4. 성공 기준 선배치 패턴

Opus 4.7은 프롬프트 앞부분을 더 중시합니다. 원하는 결과물의 형태를 상단에 먼저 명시하면 일관성이 올라갑니다.

**약한 구조 (작업 → 결과 형태):**
```
아래 기사를 요약해주세요.

[기사]

5개 불릿으로 정리해주세요.
```

**강한 구조 (성공 기준 → 작업):**
```
결과물: 바쁜 창업자를 위한 5개 불릿 요약. 각 불릿은 실행 가능한 인사이트 1개를 포함.

아래 기사를 위 형식으로 요약하세요.

[기사]
```

핵심 포인트:
- "결과물:" 또는 "목표:" 같은 라벨로 성공 기준을 맨 위에 배치
- 결과물의 용도(수요자, 활용 맥락)까지 포함하면 품질이 더 올라감
- 길이, 형식, 톤이 기본값과 다를 때 특히 효과적

### 3-5. Few-shot 예시 활용법

Opus 4.7은 규칙보다 예시에 더 정확히 반응합니다. 부정문 규칙 3개를 쓸 바에는 예시 1개가 낫습니다.

단발성 프롬프트에서 예시를 넣는 방법:
```
아래 고객 리뷰를 감성 분석하세요.

예시:
입력: "배송은 빨랐는데 포장이 엉망이에요"
출력: 긍정(배송 속도) / 부정(포장 품질) → 종합: 혼합

---
분석할 리뷰:
[리뷰 붙여넣기]
```

예시 작성 원칙:
- 실제 사용 사례와 유사하게 작성
- 엣지 케이스를 포함하면 다양한 입력에 대한 대응력이 높아짐
- 예시 간 포맷을 통일 (Opus 4.7은 포맷을 매우 정확히 복제함)
- 규칙이 많아질수록 예시로 바꾸는 게 낫다는 신호

### 3-6. 결과가 기대와 다를 때 수정하는 방법

단발성 프롬프트로 받은 결과가 기대와 다를 때, 후속 메시지로 조정할 수 있습니다.

| 문제 | 후속 수정 예시 | 프롬프트에 미리 넣는 방법 |
|---|---|---|
| 너무 길다 | "핵심만 3문장으로 요약해줘" | "3문장 이내로 답변하세요" |
| 너무 짧다 (4.7에서 자주 발생) | "2~3단락으로 더 자세히" | "각 항목당 2단락 이상으로 설명하세요" |
| 너무 추상적이다 | "구체적인 코드 예시를 포함해서 다시" | "각 설명에 코드 예시를 포함하세요" |
| 형식이 안 맞는다 | "표 형태로 다시 정리해줘" | "표 형태로 정리하세요" (예시 포함 권장) |
| 관점이 빠졌다 | "보안 관점에서도 추가 분석해줘" | 분석 관점에 "보안" 추가 |
| 수준이 안 맞는다 | "비유를 써서 좀 더 쉽게" | "비전공자도 이해할 수 있도록" |
| 톤이 건조하다 (4.7 기본 톤) | "더 친근한 톤으로" | "격려 문장을 포함하고 제안형 어미 사용" |

후속 수정이 자주 필요한 패턴이 보이면 원본 프롬프트에 해당 조건을 미리 넣도록 안내하세요.

특히 Opus 4.7에서 자주 발생하는 패턴:
- "너무 짧다" → 기본 길이 보정이 짧은 쪽으로 기울어 있음. 분량을 명시적으로 지정.
- "톤이 건조하다" → 기본 톤이 4.6보다 건조. 톤이 중요하면 직접 지시.
- "스캐폴딩을 다시 넣었는데도 효과가 없다" → 4.7은 스캐폴딩에 반응이 약함. 구체적 체크리스트로 전환.
