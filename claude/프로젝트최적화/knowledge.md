## 1. 재질문 예시 모음
 
### 케이스 A: 목적은 있으나 대상과 형태가 모호한 경우
 
**원본:** "영업 미팅 요약해주는 AI 만들어줘"
 
**재질문:**
> 현재 파악한 의도: 영업 미팅 내용을 자동 정리해주는 AI 지시문 작성인 것 같습니다. 최적화 전에 아래를 확인할게요.
>
> 1. 입력 방식은 어떻게 되나요? (예: 회의 녹취 텍스트 붙여넣기 / 메모 형식 / 음성 전사본)
> 2. 요약 결과물은 어디에 쓰이나요? (예: CRM 입력용 / 팀 공유 슬랙 메시지 / 고객사 보고서)
> 3. 반드시 포함해야 할 항목이 있나요? (예: 다음 액션 아이템 / 고객 Pain Point / 예산 언급 여부)
 
---
 
### 케이스 B: 의도 자체가 여러 방향으로 해석되는 경우
 
**원본:** "코드 리뷰 도와줘"
 
**재질문:**
> 현재 파악한 의도가 두 가지로 해석됩니다.
> - 해석 1: 사용자가 코드를 붙여넣으면 문제점을 찾아주는 AI
> - 해석 2: 코드 리뷰 코멘트를 잘 작성하도록 도와주는 AI (리뷰어 지원)
>
> 어느 쪽인가요? 또는 둘 다인가요?
 
---
 
### 케이스 C: 추상적 키워드만 있는 경우
 
**원본:** "친절하고 전문적으로 답변해줘"
 
**재질문:**
> 현재 파악한 의도: 특정 도메인에서 응대하는 AI의 톤을 설정하려는 것 같습니다.
>
> 1. 어떤 역할/도메인의 AI인가요? (예: 법률 상담 / 쇼핑몰 CS / 사내 IT 헬프데스크)
> 2. "친절"과 "전문적"이 충돌할 때 어느 쪽을 우선하나요? (예: 어려운 법률 용어 vs 쉬운 말로 풀어 설명)
> 3. 답변 불가 질문이 들어왔을 때 어떻게 처리해야 하나요?
 
---
 
### 케이스 D: 리터럴 해석에 취약한 불완전 지시
 
**원본:** "코드 변경 전에 프로젝트 구조를 확인해줘"
 
**재질문:**
> 현재 파악한 의도: 코드 변경 작업을 수행하는 AI에 사전 확인 단계를 넣으려는 것 같습니다. 4.8은 이런 지시를 리터럴하게 해석하므로 명시적으로 해야 합니다.
>
> 1. "확인"은 구체적으로 어떤 행동인가요? (예: 파일 목록 읽기 / 기존 패턴 파악 / 의존성 그래프 확인)
> 2. 확인 결과를 사용자에게 공유해야 하나요, 아니면 내부적으로만 활용하나요?
> 3. 확인 없이는 절대 변경하면 안 되는 구속력 있는 규칙인가요, 아니면 권장 사항인가요?

## 2. 최적화 예시 모음
 
### 예시 1: 경량 모드
 
**원본:** "블로그 글 써줘"
 
**의도:** 블로그 포스트 작성 요청이나, 주제·독자·톤·길이가 모두 미지정.
 
**핵심 개선:** 맥락(C)과 형식(F) 보강
 
**최적화 결과:**
 
```xml
<role>
당신은 IT 분야 전문 블로그 작가입니다. 복잡한 기술 개념을 비전공자도 이해할 수 있도록 쉽고 친근하게 설명합니다.
</role>
 
<instructions>
[주제]에 대해 블로그 포스트를 작성하세요.
 
대상 독자: [예: 개발 입문자, 비전공 직장인 등]
</instructions>
 
<output_format>
- 분량: 1,500~2,000자
- 구성: 도입(호기심 유발) → 본문(핵심 포인트 3가지) → 마무리(실천 팁 1개)
- H2, H3 헤더 사용, 각 섹션에 구체적 예시 1개 이상 포함
- 톤: 친근한 설명체 ("~입니다" 체)
</output_format>
```
 
**배포 정보:**
- Project 이름: "IT 블로그 작성기"
- Project 설명: IT 주제를 비전공자 대상 블로그 포스트로 변환하는 전문 작성 도구
---
 
### 예시 2: 표준 모드
 
**원본:** "고객 이메일에 답장 써줘. 친절하게."
 
**C.R.A.F.T. 분석:**
 
| 항목 | 평가 | 상세 |
|------|------|------|
| C (맥락) | 누락 | 업종, 고객 문의 내용, 불만/문의 유형 불명 |
| R (역할) | 보완 | "친절하게"라는 톤 힌트만 있고 CS 전문가 역할 미명시 |
| A (행동) | 보완 | "답장" 행동은 있으나, 단순 응대인지 해결책 제시인지 불명 |
| F (형식) | 누락 | 이메일 길이, 서명 포함 여부, 인사말 스타일 미지정 |
| T (기준) | 누락 | 반드시 포함할 요소(사과/해결책/후속 조치) 없음 |
 
**최적화 결과:**
 
```xml
<role>
당신은 SaaS 기업의 고객 성공(CS) 전문가입니다. 5년간 B2B 고객 응대 경험이 있으며, 전문적이면서도 따뜻한 톤으로 소통합니다.
 
4.8은 기본적으로 직설적인 톤을 사용하므로, 여기서는 따뜻한 협력적 톤을 명시적으로 유지하세요.
</role>
 
<instructions>
아래 고객 이메일에 대한 답장을 작성하세요.
 
[고객 이메일 내용 붙여넣기]
</instructions>
 
<constraints>
포함할 요소: (1) 문의에 대한 감사/공감 (2) 구체적 해결책 또는 다음 단계 (3) 추가 도움 제안
전개 순서: 공감 → 해결 → 신뢰
표현 원칙: 구체적이고 실행 가능한 해결책을 제시하고, 고객의 상황을 인정하는 문구로 시작합니다.
</constraints>
 
<output_format>
- 길이: 150~250자
- 구조: 인사말 → 본문 → 서명 ([이름/직책] placeholder 사용)
</output_format>
 
<examples>
<example index="1">
입력: "결제가 두 번 되었습니다. 환불해주세요."
 
출력:
[고객명]님, 안녕하세요.
 
이중 결제로 불편을 드려 죄송합니다. 확인 결과 [날짜]에 중복 결제가 발생한 것을 확인했으며, 즉시 환불 처리를 진행했습니다. 영업일 기준 3~5일 내 원래 결제 수단으로 환불됩니다.
 
추가로 궁금하신 점이 있으시면 언제든 말씀해 주세요.
 
감사합니다.
[이름] | [직책]
</example>
</examples>
```
 
**배포 정보:**
- Project 이름: "CS 이메일 답장 도우미"
- Project 설명: 고객 문의/불만 이메일에 공감적이고 해결 중심적인 답장을 작성하는 CS 전문 도구
**4.8 적용 포인트:** `constraints`에서 "제외할 요소: 책임 회피 표현, 과도한 사과"를 쓰는 대신, "표현 원칙"에 긍정적으로 재구성했습니다. 부정 지시보다 긍정 지시가 4.8에서 더 효과적입니다.

## 3. Claude Opus 4.8 프롬프팅 공식 가이드 (Anthropic 공식 문서 기반, 2026.06 기준)
 
이 섹션은 Anthropic의 공식 "Prompting best practices" 문서(2026.05.28 Opus 4.8 출시에 맞춰 업데이트)에서 Claude Project Instructions 작성에 직접 관련된 원칙을 정리한 것입니다.
 
### 3-1. 현재 모델 라인업 (2026.06 기준)
 
| 모델 | 특징 | 컨텍스트 | 적합한 용도 |
|------|------|---------|------------|
| Claude Opus 4.8 | 현 플래그십, 판단력·정직성 향상, adaptive thinking, fast mode | 최대 1M 토큰 (Microsoft Foundry는 200K) | 장기 자율 에이전트, 복잡 코딩, 지식 업무, 장문 분석 |
| Claude Opus 4.7 | 이전 플래그십 | 최대 1M 토큰 | 4.8 이전 최적화 프롬프트 호환/롤백용 |
| Claude Opus 4.6 | 구 플래그십 | 최대 1M 토큰 | 레거시 호환 |
| Claude Sonnet 4.6 | 속도·성능 균형, 기본 effort=high | 200K 토큰 | 일상 작업, 코드 작성, 문서 생성 |
| Claude Haiku 4.5 | 빠른 응답 | 200K 토큰 | 간단 질의, 분류, 요약 |
| Claude Mythos Preview | 더 상위·최고 정렬 모델 (미공개, 제한 릴리즈) | - | 일반 릴리즈 아님 (Project Glasswing) |
 
Opus 4.8 model string: `claude-opus-4-8`
가격: 입력 $5 / 출력 $25 per million tokens (4.5/4.6/4.7과 동일, 인상 없음). 프롬프트 캐싱 최대 90% 절감, 배치 50% 절감. 미국 전용 추론은 1.1배.
최대 출력: 128K 토큰. 기본 effort: high (API·Claude Code·claude.ai 공통).
 
### 3-2. 핵심 원칙: 계약서(Contract) 스타일 (4.7 이후 중요)
 
Claude는 프롬프트를 "계약서"처럼 해석하도록 설계되어 있습니다. 4.7 이후 이 계약 해석이 더 엄격해졌고, 4.8도 동일합니다.
 
효과적인 프롬프트는 대화가 아닌 명세서입니다:
- 역할(누구인가), 규칙(무엇을 해야/하지 말아야 하는가), 출력 형식(어떤 모양인가)을 명확히 정의
- "~해주실 수 있나요?" 대신 "당신은 [역할]입니다. [행동]하세요. 조건: [제약]"
- 검증 가능한 성공 기준을 포함 (예: "3문장 이내로 요약" / "비전공자가 이해할 수 있는 수준")
검증 팁: 작성한 프롬프트를 맥락 없는 동료에게 보여주고 따라할 수 있는지 확인하세요.
동료가 혼란스러워하면 Claude도 마찬가지입니다. 4.8에서는 특히 이 원칙이 중요합니다 — 동료가 "아마 이런 뜻일 것"이라고 채워 넣는 빈틈을 4.8은 채워 넣지 않습니다.
 
### 3-3. 리터럴 해석 원칙 (4.7 도입 · 4.8 유효)
 
4.7에서 도입된 큰 변화는 "암묵적 일반화"를 하지 않는 것이며, 4.8도 이를 유지합니다.
 
공식 문서 인용 요지: 4.7/4.8은 하나의 지시를 다른 항목으로 암묵적으로 일반화하지 않으며, 사용자가 요청하지 않은 것을 추론하지 않습니다.
 
실무 적용:
- 범위를 명시하세요: "이 포맷을 모든 섹션에 적용하세요. 첫 섹션에만 적용하지 마세요."
- 전제 조건을 구속력 있게 쓰세요: "확인한 후 제안해" (약함) → "제안하기 전에 반드시 먼저 다음을 수행하세요: [구체적 행동]" (강함)
- 암묵적 기대를 명시화하세요: "잘 요약해줘" → "다음 3가지 요소를 포함하세요: 핵심 주장 / 근거 / 결론"
이 변화의 업사이드: API 파이프라인, 구조화된 추출, 예측 가능한 동작이 필요한 use case에서 성능이 향상됩니다.
다운사이드: 모호한 프롬프트는 4.6 같은 이전 세대보다 결과가 나빠질 수 있습니다.
 
### 3-4. XML 태그 활용
 
- Claude는 XML 태그로 구조화된 프롬프트를 정확하게 파싱합니다
- 태그 이름에 특별한 정답은 없음. 내용을 설명하는 일관된 이름이면 충분
- 태그를 지시문 안에서 참조할 때 동일한 이름 사용 (예: "`<contract>` 태그의 계약서를 기반으로...")
- 태그 중첩으로 계층 구조 표현 가능 (예: `<documents>` 안에 `<document index="1">`)
권장 구조:
```
<role> → <context> → <instructions> → <constraints> → <examples> → <output_format>
```
모든 태그가 항상 필요한 것은 아닙니다. 프롬프트의 복잡도에 따라 필요한 태그만 사용하세요.
 
### 3-5. Few-shot 예시 활용법 (4.7 이후 강력)
 
- 3~5개의 예시가 최적. 긴 규칙 설명보다 잘 만든 예시가 더 강력함
- 예시는 `<example>` 태그로 감싸서 지시문과 명확히 구분
- 여러 예시는 `<examples>` 태그로 묶기
- 예시 작성 원칙:
  - 실제 사용 사례와 유사하게 작성
  - 엣지 케이스를 포함하여 다양성 확보
  - 예시 간 포맷을 반드시 통일
4.8에서도 특히 중요: 긍정 예시(desired output sample)는 "하지 마세요" 지시보다 훨씬 강력합니다. 원하는 톤·길이·구조를 보여주는 예시 하나가 다섯 줄의 부정 규칙보다 효과적입니다.
 
### 3-6. Claude Opus 4.8 특화 주의사항
 
#### (1) 응답 길이 적응형 (4.7 이후)
- 4.8은 고정 verbosity가 없고, 작업 복잡도에 맞춰 길이를 스스로 판단합니다.
- 단순 조회는 더 짧게, 열린 분석은 더 길게 응답합니다.
- 특정 길이/스타일이 필요하면 명시하세요:
  - 간결함: "간결하고 집중된 응답을 제공하세요. 비본질적인 맥락은 생략하고 예시는 최소화하세요."
  - 상세함: 원하는 상세도의 긍정 예시를 `<example>`로 제공
- "장황하게 쓰지 마세요"(부정) 보다 짧은 예시 하나(긍정)가 훨씬 효과적입니다.
#### (2) Effort 레벨 (4.8에서 가장 중요)
- effort 레벨(공통): low / medium / high / xhigh / max — 5단계. 4.8 기본값은 high (API·Claude Code·claude.ai 공통).
- 핵심 변화: claude.ai 사이드바에 effort 다이얼이 노출됨. claude.ai의 "Extra" = API/Claude Code의 xhigh. 이제 Project 대화에서도 사용자가 직접 effort를 조절할 수 있음 (4.7까지는 불가능했음).
- Anthropic 권장: 코딩·에이전트 작업은 xhigh로 시작, 지능이 중요한 작업은 high를 최소선으로. 비용·지연이 중요하면 medium/low로 내림. max는 xhigh에서 더 올릴 여지가 평가로 확인될 때만.
- 4.8은 effort를 엄격히 respect, 특히 low/medium에서 요청 범위만 수행. 복잡한 작업을 low로 돌리면 under-thinking 위험. 얕은 추론이 보이면 프롬프트로 우회하지 말고 effort를 올릴 것.
- xhigh/max로 돌릴 때는 max_tokens를 넉넉히(예: 64K부터) 둬서 추론·툴콜 공간 확보.
- "effort는 이전 어떤 Opus보다 4.8에서 더 중요" — 업그레이드 시 적극적으로 실험.
- Project 배포 시: 이 사실 덕분에 "Effort 레벨 권장" 출력이 실효성을 가짐. 사용자에게 "이 Project는 claude.ai에서 effort를 high/xhigh로 설정해 사용하세요"처럼 구체적으로 안내할 수 있음. 다단계 판단이 필요한 Project일수록 high(기본)~xhigh가 스윗스폿이며, 모든 작업을 max로 올리는 것은 대개 토큰만 소모함.
#### (3) 과도한 강조 표현 완화 (4.6 이후, 4.8도 동일)
- 4.6 이후 모델은 시스템 프롬프트에 강하게 반응합니다 (4.8 포함).
- 강한 어조("CRITICAL: You MUST use this tool when...", "ALWAYS do this without exception")는 overtrigger를 유발합니다.
- 자연스러운 톤으로 완화하세요: "Use this tool when..." 정도면 충분합니다.
- 최적화 시 원본에 과도한 강조 표현이 있다면 완화를 권장하세요.
#### (4) Scaffolding 제거 기회 (4.7 이후)
- 4.8은 긴 에이전트 트레이에서 사용자 대상 진행 상황 업데이트를 native하게 잘 수행합니다.
- 다음과 같은 scaffolding은 이제 제거해도 됩니다:
  - "3번의 tool call마다 진행 상황을 요약하세요"
  - "다음 단계로 넘어가기 전에 무엇을 했는지 설명하세요"
- 만약 업데이트의 길이/내용이 만족스럽지 않다면, 원하는 모습을 긍정 예시로 보여주세요.
#### (5) 툴/서브에이전트 사용 감소 (4.8 기본 동작)
- 4.8은 4.6보다 툴을 덜 쓰고 reasoning을 더 사용합니다 — 대부분의 경우 결과가 좋아집니다.
- 4.8은 4.6에서 보이던 툴콜 이슈를 개선해 툴 트리거가 더 정확합니다.
- 툴 사용이 중요한 use case에서는 "언제/왜/어떻게" 명시하세요.
- 서브에이전트도 기본적으로 덜 spawn됨 → 병렬 탐색이 필요하면 명시적으로 요청.
#### (6) 톤 변화: 직설적이고 덜 validating (4.7 이후)
- 4.8은 4.6의 따뜻한 스타일보다 더 직설적이고 opinionated하며, 이모지가 적습니다.
- 제품 음성이 따뜻하거나 대화적이어야 한다면 명시:
  - "따뜻하고 협력적인 톤을 사용하세요. 답변 전에 사용자의 맥락을 먼저 인정하세요."
#### (7) 과도한 탐색과 오버엔지니어링 (4.6 이후, 4.8도 동일)
- Opus 계열은 요청하지 않은 추가 파일 생성, 불필요한 추상화, 과도한 방어적 코딩 경향이 있습니다.
- 범위를 명확히 제한하세요:
  - "요청된 변경만 수행하세요. 관련 없는 개선을 추가하지 마세요."
  - "현재 작업에 필요한 최소한의 복잡성만 사용하세요."
#### (8) 부정문보다 긍정문 지시 (4.7 이후 강화)
- Anthropic 공식 가이드: "긍정 예시가 부정 지시보다 효과적이다"
- "마크다운을 사용하지 마세요" 대신 "매끄럽게 흐르는 산문 단락으로 작성하세요"
- "외부 정보를 참조하지 마세요" 대신 "제공된 문서의 정보만 사용하여 답변하세요"
#### (9) 프롬프트 스타일이 출력에 영향
- 프롬프트에서 마크다운을 많이 사용하면 응답에도 마크다운이 많아짐
- 원하는 출력 형태와 유사한 스타일로 프롬프트 자체를 작성하면 효과적
#### (10) Prefilled 응답 Deprecated (4.6부터)
- 마지막 assistant turn에 프리필 응답을 넣는 기능은 4.6부터 deprecated, Mythos에서는 400 error
- 모델의 지시 준수 능력이 향상되어 대부분의 prefill 사례는 더 이상 필요 없습니다
- 출력 포맷 제어는 prompt로 수행하세요
#### (11) 새 토크나이저 (토큰 사용량 변화)
- 4.7부터 새 토크나이저를 사용해 동일 입력이 1.0~1.35배 더 많은 토큰으로 인코딩될 수 있습니다 (4.8도 동일 토크나이저).
- 이는 4.7 마이그레이션 때 도입된 변경이며, 4.8 업그레이드만으로 새로 생기는 것은 아닙니다.
- 가격 단가는 동일하지만 실효 비용이 상승 가능
- `count_tokens` API로 실제 워크로드 토큰 수를 사전 측정 권장
- max_tokens 파라미터에 여유를 더 두는 것이 좋음
#### (12) 프론트엔드 디자인 기본값 (4.7 기준 관찰, 4.8 재확인 권장)
- 4.7 기준으로 Opus는 강한 default 디자인 스타일을 가짐: 따뜻한 크림색/오프화이트 배경, serif 서체(Georgia/Fraunces/Playfair), 테라코타/앰버 엑센트
- 에디토리얼/호스피탤리티/포트폴리오에는 잘 맞지만, 대시보드/개발자 툴/핀테크/헬스케어/엔터프라이즈 앱에는 어울리지 않음
- 일반 지시("크림색 쓰지 마")는 효과가 약함 — 다른 고정 팔레트로 전환될 뿐
- 효과적인 두 가지 방법:
  1. 구체적 대안을 명시 (정확한 컬러 팔레트, 서체, 여백 규칙)
  2. 빌드 전에 옵션 제안 요청: "빌드하기 전에 이 브리프에 맞춘 4가지 시각 방향을 제안하세요(배경 hex / 엑센트 hex / 서체 / 1줄 근거). 사용자가 선택한 것만 구현하세요."
- 주의: 이 항목은 4.7 기준 관찰입니다. 4.8에서 기본 팔레트가 동일한지는 별도 확인을 권장합니다.
#### (13) 정직성과 보정된 불확실성 (4.8 신규)
- 4.8의 가장 두드러진 변화는 지능이 아니라 판단력·정직성입니다. 근거가 약한데 진척을 확신하는 경향이 줄었고, 불확실한 질문에는 답을 늘리기보다 기권하는 쪽으로 환각률이 낮아졌습니다.
- 자신이 작성한 코드의 결함을 그냥 넘기는 비율이 4.7 대비 약 4배 낮습니다.
- 프롬프팅 함의:
  - 거짓 확신을 강요하지 마세요. 불확실해도 시도하길 원하면 명시: "불확실하면 최선의 추정치를 제시하고 불확실함을 라벨링하세요."
  - 자기 검증 능력이 향상되었으므로 무거운 이중확인 scaffolding을 덜어낼 수 있습니다.
  - 최적화 시, 모델이 "모른다/확신 못 한다"고 답하는 것을 결함이 아니라 의도된 동작으로 다루세요.
#### (14) Adaptive thinking 기본 off (4.8)
- thinking은 `thinking: {type: "adaptive"}`를 명시할 때만 켜집니다(기본 off). 켜면 4.8이 턴별로 추론 필요 여부를 판단해, 같은 effort에서 4.7보다 낭비되는 thinking 토큰이 줄어듭니다.
- 트리거 동작은 조정 가능(steerable)합니다. 단순 조회에는 바로 응답하고, 복잡한 다단계 문제에서만 깊게 추론합니다.
- 주로 API 사항이지만, effort와 함께 4.8의 추론 깊이를 결정하는 핵심 축입니다.
#### (15) 기타 4.8 변경 (주로 API)
- 프롬프트 캐시 최소 길이: 1,024 토큰으로 낮아짐(4.7보다 낮음). 4.7에서 너무 짧아 캐시되지 않던 프롬프트도 코드 변경 없이 캐시됩니다.
- 대화 중간 system 메시지: user 턴 직후 `role: "system"` 허용(4.7은 400 에러). 지시 갱신 시 전체 히스토리 재구성 없이 캐시 히트를 보존할 수 있습니다. 처음부터 적용할 지시는 top-level system 필드를 사용하세요.
- 거부 stop 상세: 거부 응답의 `stop_details`가 공식 문서화되어 거부 카테고리를 식별합니다(별도 베타 헤더·옵트아웃 없음).
- Fast mode: `speed: "fast"`로 출력 토큰 처리량 최대 2.5배(프리미엄 가격, 리서치 프리뷰). 이전 모델 대비 3배 저렴.
- Claude Code "dynamic workflows": 대규모 문제를 위한 병렬 서브에이전트 기능(Enterprise·Team·Max 플랜).
### 3-7. 장문 컨텍스트 처리
 
- 긴 문서/데이터는 프롬프트 상단에 배치, 쿼리/지시는 하단에 배치
- 이 배치만으로 응답 품질이 최대 30% 향상됨 (Anthropic 내부 테스트 기준)
- 여러 문서를 사용할 때는 각 문서를 `<document>` 태그로 감싸고 메타데이터 서브태그 사용
- 긴 문서에서 정보를 추출할 때는 Claude에게 먼저 관련 부분을 인용하게 한 뒤 작업을 수행하게 하면 정확도 향상
- 4.8은 1M 토큰까지 long-context premium 없음 (Microsoft Foundry는 출시 시점 200K)
### 3-8. Knowledge Files 활용
 
- 파일 수: 무제한 (파일당 최대 30MB)
- 기본 컨텍스트: 200K 토큰 (약 500페이지)
- 유료 플랜(Pro, Max, Team, Enterprise)에서는 RAG 모드 자동 활성화 → 용량 최대 10배 확장
- 지원 포맷: PDF, DOCX, CSV, TXT, HTML, ODT, RTF, EPUB 등
- Google Docs, Google Sheets 연결 가능 (Pro 이상) → 원본 수정 시 자동 반영
분리 원칙:
- Instructions에 남길 것: 핵심 역할, 행동 원칙, 워크플로우, 제약 조건
- Knowledge Files로 옮길 것: 참고 예시, 도메인 지식, 긴 템플릿, 용어집
### 3-9. 출력 포맷 제어
 
마크다운 최소화가 필요한 경우 (긍정 지시 버전):
```xml

보고서, 문서, 기술 설명, 분석 등 장문 콘텐츠를 작성할 때는 완전한 문장과 단락으로 구성된 자연스러운 산문으로 작성하세요. 인라인 코드(`code`)와 코드 블록, 간단한 제목(###)만 사용하세요.

(a) 정말로 개별적인 항목이라 목록 형태가 최선인 경우
(b) 사용자가 명시적으로 목록이나 순위를 요청한 경우
</formatting_guide>
```

## 4. 마이그레이션 체크리스트
 
### 4-A. 4.7 → 4.8 (현행 업그레이드)
 
4.8은 기존 4.7 프롬프트에서 별도 튜닝 없이도 잘 동작하며 breaking API 변경이 없습니다. 다만 effort 기본값 전환으로 동일 프롬프트도 출력이 달라질 수 있으니 아래를 점검하세요.
 
| # | 점검 항목 | 조치 |
|---|----------|------|
| 1 | model string이 `claude-opus-4-7`인가? | `claude-opus-4-8`로 교체 (또는 alias 갱신) |
| 2 | effort를 명시적으로 설정하는가? | 기본값이 high(전 표면 공통)로 바뀜 — 동일 프롬프트도 출력이 달라질 수 있어 재검증 |
| 3 | 코딩·에이전트 Project인가? | claude.ai에서 effort를 xhigh로 올려 사용 권장 |
| 4 | 출력 포맷·구조화 추출에 의존하는가? | high 기본값 전환으로 출력이 달라질 수 있어 eval 재실행 |
| 5 | thinking을 쓰는가? | adaptive thinking은 `thinking:{type:"adaptive"}` 명시 시에만 켜짐 (기본 off) |
| 6 | 프롬프트 캐싱을 쓰는가? | 최소 캐시 길이가 1,024 토큰으로 낮아짐 — 짧은 프롬프트도 캐시되니 재측정 |
| 7 | 대화 중간 지시 갱신 로직이 있는가? | 4.8은 user 턴 직후 `role:"system"` 허용 — 히스토리 재구성 없이 캐시 보존 가능 |
| 8 | 무거운 자체 검증/이중확인 scaffolding이 있는가? | 4.8 정직성·자기검증 향상 — 완화 가능 |
 
### 4-B. 레거시(4.6 이하 → 4.7/4.8) 체크리스트
 
4.6 이하용으로 튜닝된 기존 프롬프트를 올릴 때 다음 체크리스트를 적용하세요. (4.7에서 도입된 원칙은 4.8에도 그대로 적용됩니다.)
 
| # | 점검 항목 | 조치 |
|---|----------|------|
| 1 | "하지 마세요"가 3개 이상 있는가? | 긍정 지시 또는 긍정 예시로 전환 |
| 2 | "After every N tool calls, summarize..." 같은 진행 보고 scaffolding이 있는가? | 제거 (4.7/4.8은 native하게 잘 보고함) |
| 3 | "CRITICAL:", "MUST", "ALWAYS" 같은 강압적 표현이 있는가? | 자연스러운 어조로 완화 |
| 4 | 암묵적 기대(예: "잘 요약해줘", "알아서 확인해줘")가 있는가? | 구체적 요소 / 구속력 있는 단계로 명시화 |
| 5 | 범위 불명확한 지시(예: "이 포맷을 적용하세요")가 있는가? | "모든 섹션에" 또는 "첫 섹션에만"처럼 범위 명시 |
| 6 | 따뜻한 톤이 제품 음성에 중요한가? | Instructions에 명시 (4.7/4.8 기본은 더 직설적) |
| 7 | 툴 사용이 핵심인 use case인가? | 툴을 언제/왜/어떻게 써야 하는지 명시 |
| 8 | 응답 길이가 중요한가? | 원하는 길이의 긍정 예시 제공 |
| 9 | prefilled assistant message를 쓰고 있는가? | 제거하고 프롬프트로 포맷 제어 |
| 10 | 서브에이전트 병렬 실행이 필요한가? | 명시적으로 지시 (4.7/4.8 기본은 덜 spawn) |

## 5. 활용 가이드 요약
 
**재질문이 필요한 신호:**
- "잘", "친절하게", "전문적으로" 같은 추상적 형용사만 있을 때
- 사용 대상(사람)이 전혀 언급되지 않을 때
- 결과물의 형태(길이, 구조, 포맷)가 완전히 불명확할 때
- 하나의 문장이 2가지 이상의 완전히 다른 의미로 읽힐 때
- 4.8 리터럴 해석에 취약한 모호한 전제 조건이 있을 때
**바로 최적화해도 되는 신호:**
- 누가 사용하고, 무엇을 원하고, 어떤 형태로 원하는지 파악 가능할 때
- 부족한 부분이 1~2개이고 합리적 추론으로 채울 수 있을 때
  (이 경우 추론 내용을 "의도 파악" 섹션에 명시하고 진행)
**Instructions 길이 관리 팁:**
- Instructions이 너무 길면 중간 규칙이 무시될 수 있음
- 핵심 규칙만 남기고, 예시·참고 자료는 Knowledge Files로 이동
- 규칙이 많을 때는 번호를 매기고 우선순위를 명시
- 4.8(4.7 이후)의 새 토크나이저는 동일 텍스트가 최대 1.35배까지 많은 토큰을 소모할 수 있으므로, 간결함이 더욱 중요
**최적화 결과물 자기 점검 체크리스트 (4.8 기준):**
1. 원본의 핵심 의도가 보존되었는가?
2. 불필요한 복잡성이 추가되지 않았는가?
3. 계약서 스타일(명세서 형태)로 작성되었는가?
4. XML 태그가 일관되고 적절히 사용되었는가?
5. 예시가 포함되어야 하는 경우 `<example>` 태그로 감싸져 있는가?
6. 4.8 리터럴 해석: 명시되지 않은 행동을 기대하는 지시가 없는가?
7. 과도한 강조 표현("MUST", "CRITICAL", "ALWAYS")이 없는가?
8. 부정 지시("~하지 마세요")가 과도하지 않고, 긍정 예시 또는 긍정 지시로 표현되어 있는가?
9. 성공 기준이 구체적이고 검증 가능한가?
10. 불필요한 진행 상황 scaffolding("N번마다 요약")이 제거되어 있는가?
11. 톤이 제품 음성에 중요하다면 명시되어 있는가? (4.8 기본은 직설적)
12. 범위가 불명확한 지시가 없는가? (4.8은 자동 일반화하지 않음)
13. 거짓 확신을 강요하는 지시가 없는가? (불확실성 기권을 존중하되, 시도가 필요하면 "불확실하면 추정치를 라벨링해 제시"로 명시)
14. effort 권장이 포함되어 있는가? (claude.ai에서 사용자가 effort 조절 가능 — Project 복잡도에 맞는 레벨 안내)
